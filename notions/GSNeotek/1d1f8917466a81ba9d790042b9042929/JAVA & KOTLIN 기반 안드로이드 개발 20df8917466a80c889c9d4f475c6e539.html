항목Java 기반 앱 개발Kotlin 기반 앱 개발
개발 안정성전통적인 방식으로 안정적이며 예측 가능Android 공식 지원 언어로, 최신 API에 가장 유리

기능 구현 적합도(전화 수신, 웹소켓 등)모두 구현 가능하며 시스템 API 지원 확실동일하게 가능하며, 코루틴 기반의 비동기 처리 등 현대적 방식 사용
유지보수 용이성기존 코드 스타일 유지에 용이코드가 훨씬 간결해지고, 유지보수/리팩토링 쉬움
코드 품질 및 생산성복잡하고 길어질 수 있음 (보일러플레이트 多)코드가 짧고 명확함 (생산성, 가독성 우수)
향후 기술 확장성최신 Jetpack Compose 등 적용 어려움최신 UI/기술 적용에 유리 (Compose, Hilt 등)
기술 트렌드 대응력점점 축소되는 방향 (Legacy화 우려)구글 공식 주력 언어, 장기적 투자 가치 높음
비용 및 속도 측면기존 조직/환경에 따라 빠르게 대응 가능초기 전환 시 학습 필요하지만 장기적으로 효율적

Java → Kotlin 리팩토링 전략

전략 원칙

항목설명
점진적 전환전체 코드를 한 번에 바꾸지 말고, 모듈 단위로 Kotlin 전환
테스트 우선리팩토링 시 반드시 기존 기능이 유지되는지 단위 테스트로 확인
Kotlin-Java 혼용 가능성 활용Android 프로젝트는 Kotlin과 Java를 동시에 쓸 수 있음 (.kt와 .java 혼재 가능)

리팩토링 단계별 전략

단계목표작업 예시
1단계Kotlin 지원 설정build.gradle에 Kotlin 플러그인 적용 및 설정
2단계신규 기능부터 Kotlin으로 작성예: 새로 만드는 API 통신 클래스, 유틸 클래스는 .kt로 작성
3단계ViewModel, Repository 계층부터 전환비즈니스 로직 단위의 모듈부터 Kotlin으로 이전
4단계반복 사용되는 Java 클래스를 Kotlin으로 변환DTO, Model, 헬퍼 클래스 등
5단계UI 관련 코드 전환 (선택)Activity/Fragment를 점진적으로 kt 파일로 전환
6단계코틀린 DSL 적용Gradle, Navigation 등 코틀린 스타일로 구조 개선
7단계Jetpack Compose 도입 (Optional)UI 개편 시 Jetpack Compose 고려

업무 복잡도현재 수준에선 충분단순해도 장기적으로 깔끔하고 생산성 높음
기술 투자 가치투자 대비 리턴 낮음 (점점 줄어듬)기술자산으로 지속 사용 가능

최종 판단

항목선택
현재 기능 수준에서는Java도 "충분히 가능" 또한 NCC팀 내에 Java 개발자가 많기 때문에 유지보수에 더 적합함.
향후 유지보수, 고도화, 기술 일관성 고려하면Kotlin이 "더 좋은 선택" 이지만, 완전한 Kotlin 개발로 진행할 경우, 유지보수를 위해 추후 Kotlin 개발자의 추가 투입이 필요함.

최종 제안

기존 Java 기반 팀이므로 Java로 MVP(최소 기능 앱)을 빠르게 출시하고, 유지보수 및 고도화 단계에서 Kotlin으로 점진 전환하는 하이브리드 전략이 가장 사업적으로 합리적입니다. 즉, Java와 Kotlin 모두 사용하는 WiseN TM 앱 개발이어야 합니다.
