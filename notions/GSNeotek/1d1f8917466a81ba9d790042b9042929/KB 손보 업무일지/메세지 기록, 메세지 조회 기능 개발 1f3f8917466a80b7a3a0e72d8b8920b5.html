업무 내역[개발] SmsLogMapper 쿼리 작성 및 최적화
• INNER JOIN과 GROUP BY를 활용해 고객별 최신 메세지 1건씩만 조회하는 쿼리 작성
• 성능 및 정합성을 고려하여 중복 데이터 없이 대표 메세지를 도출
• WHERE 1=1 사용 지양:

→ SQL 조건절 관리가 느슨해지고 유지보수 시 조건 누락에 따른 오류 발생 가능성이 있기 때문
• Mapper.xml에서 <if> 문 사용 시, Java(Spring)에서 전달된 파라미터를 활용하여 동적 SQL 구현[개발] 서비스 및 컨트롤러 구현
• SmsLogController에서 클라이언트 요청(사번, 시작일, 종료일, 고객번호, 보상번호)을 수신하여 SmsLogService로 전달
• SmsLogService에서 다음 기능 처리:
◦ 고객별 최신 메세지 조회
◦ 미리보기용 메세지: 메세지 본문 중 최대 40자까지만 추출하여 요약 표시
◦ 요청 조건에 따라 유동적으로 조회 조건 적용
• 응답 포맷:
◦ 성공 여부 (success: true/false)
◦ 요청된 사번 기준 전체 메세지 수
◦ 각 고객의 최신 메세지 요약 정보 목록
요약SQL과 서비스 로직을 유기적으로 연계하는 작업을 통해 **"쿼리 최적화와 API 설계는 별개가 아니라 하나의 흐름이다"**라는 점을 실감할 수 있었다.

특히 고객별 최신 메세지를 도출하기 위한 JOIN + GROUP BY 구성과 40자 미리보기 구현은 단순 출력이 아니라 데이터 요약 처리 및 응답 최적화라는 실질적 가치를 제공하는 작업이었다.
또한 Mapper의 <if>문 활용과 Spring 파라미터 연동을 통해 동적 SQL 처리의 유연성을 익혔으며, 컨트롤러-서비스-쿼리 계층 간 명확한 역할 분리를 지키면서 안정적인 코드 구조를 유지하는 것의 중요성도 다시 한 번 체감했다.

의제

특정 상담사(사번)에 따른 메세지 기록 조회하기

메모

  • DB는 tb_sms_log 테이블을 사용한다
  • 조회 기준은 sms_date를 기준으로 가져온다
  • 사번과 기간(start_date, end_date)이 꼭 있어야 조회가 가능하다
  • 사번은 SSO Token으로 부터 추출한다.

Request Parameters

  • start_date : String - 시작 기간
  • end_date : String - 종료 기간
  • customer_number : String - 고객 번호
  • rcv_no : String - 접수 번호
  • emp_no : String - 상담사 사번 ( 토큰으로 부터 가져온다)
