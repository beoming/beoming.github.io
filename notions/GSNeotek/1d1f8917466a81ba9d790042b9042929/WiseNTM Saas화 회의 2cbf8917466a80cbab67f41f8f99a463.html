WiseNTM 고도화 상세 기획안

1. 인프라 (Infrastructure)

1-1. 서버 및 컨테이너 통신 (Networking)

Goal: MSA 환경에서 서비스 간의 안전하고 효율적인 통신 구조 설계
  • 서비스 디스커버리(Service Discovery): 동적으로 생성/삭제되는 컨테이너의 IP 관리 방안 (예: K8s Service, AWS Cloud Map)
  • 통신 프로토콜:
    • Internal: gRPC (고성능, 서비스 간 통신) vs REST API (범용성)
    • External: REST API, WebSocket (실시간 통신), Pub/Sub
  • API Gateway 도입: 인증/인가, 라우팅, Rate Limiting의 중앙 관리 여부 (Kong, AWS API Gateway 등)

1-2. 오케스트레이션 (Orchestration)

Goal: 컨테이너 관리 및 배포 자동화
  • 플랫폼 선정:
    • AWS ECS: 러닝 커브가 낮고 AWS 친화적 (문서 내 언급)
    • K8s (EKS/GKE): 표준화된 오케스트레이션, 운영 복잡도는 높으나 확장성 우수
  • 전략: "컨테이너가 서버 1개"라는 개념을 넘어 Pod 단위의 스케일링 전략 수립

1-3. 보안 (Security)

  • 네트워크 보안:
    • VPC 설계 (Public/Private Subnet 분리)
    • DB 및 내부 서버의 외부 접근 차단 (Bastion Host 또는 VPN 활용)
  • 애플리케이션 보안:
    • 개인정보 마스킹 처리 - 개인정보 확립 필요 (ex. 전화번호+이름+생년월일)
    • JWT를 통한 세션 관리 및 인증 강화(보안성 심의 통과)

1-4. CI/CD (지속적 통합/배포)

  • 파이프라인 구축: Git Push to Build (Docker Image) to Test to Deploy 자동화
  • 도구 선정: Jenkins, GitHub Actions, GitLab CI, ArgoCD (GitOps 방식)
  • 환경 분리: Dev / Staging / Production 환경 구성 및 배포 전략

1-5. 이중화(HA) 및 DR(재해 복구)

  • 고가용성 (HA): Multi-AZ(가용 영역) 구성을 통한 서버/DB 이중화
    • ALB 자체가 AWS에 의해 자동으로 이중화되어 관리됩니다. 들어오는 트래픽을 가동 중인 여러 AZ의 서버로 분산시킵니다.
    • 최소 2개 이상의 AZ에 컨테이너를 분산 배치합니다.
    • Auto Scailing Group 을 설정하여, 특정 서버가 죽으면 자동으로 새 서버를 띄우고 컨테이너를 다시 실행합니다.
  • DR 전략: 데이터 백업 주기 및 리전 간 복제(Cross-Region Replication) 필요성 논의

2. 개발 및 아키텍처 (Development)

2-1. 메시징 시스템 전환 (MQTT → Kafka/Redis)

Goal: 추적 및 관리가 용이한 구조로 변경
  • 기술 검토:
    • Kafka: 대용량 데이터 처리, 메시지 영속성(저장), 재생(Replay) 가능 to 로그, 이력 추적용
    • Redis (Pub/Sub): 빠른 속도, 인메모리, 상태 관리(Presence) to 실시간 알림, 소켓 상태 핸들링
  • 논의 사항: 현재 MQTT의 브로커 기능을 Kafka와 Redis로 어떻게 역할 분담할 것인가? (예: 채팅/알림은 Redis, 로그/통계는 Kafka)

2-2. MSA 구조 도입 기준

Goal: 기능별 분리를 통한 유지보수성 향상 (단, 과도한 분리 지양)
  • 서비스 분리(안) 검토:
    1. 단말기 API Server: 단말기 관련 비즈니스 로직
    1. 클라이언트 API Server: 사용자단 인증, 화면 관련 비즈니스 로직
    1. Socket Server: 실시간 통신 (웹소켓), 상태 전달
    1. Batch Server: 통계 집계, 데이터 후처리, SMS/알림 발송
  • 데이터 분리: 서비스별 DB(Schema) 분리 여부 및 트랜잭션 관리(Saga 패턴 등) 필요성

2-3. 애플리케이션 기능 정의 (UI/UX 기반)

화면 설계서 기반 기능 확정
  • My Phone: 통화/메시지 이력 조회, 녹취 청취, 상세 정보, 메모
  • 모바일 Box: 파일(이미지/PDF) 업로드, 조회, 다운로드, 삭제
  • 지식창고: 사내 지식 검색, 카테고리 관리
  • 분석관리: 통화 통계(대시보드), 시각화 통계, 데이터 분석
  • 설정: 제외번호 관리, 업무시간/휴일 설정, 사용자/부서/단말기 관리, 메뉴 권한 관리
  • 고도화 신규 기능: 명함 발송, MDM 연계, 개인정보 마스킹, 메시지 추적 기능

2-4. 현 WiseNTM 문제점 및 개선

  • 통합 이슈: 고객사별 커스터마이징으로 파편화된 소스코드 통합 방안 (Core 모듈과 Custom 모듈 분리 전략)
  • 유지보수: 기능 개선 시 사이드 이펙트 최소화를 위한 테스트 코드 작성 의무화

2-5. 문서화 및 개발 공통화

  • API 문서: Swagger (OpenAPI Spec) 자동화 적용
  • 코드 컨벤션: 언어별(Java/Node.js 등) 린트(Lint) 규칙 및 네이밍 표준 확정
  • 에러 핸들링: 표준 에러 코드 및 응답 포맷 정의

4. 개발계 구축 및 활용 전략

MSA 구조는 서비스 간 통신이 복잡하기 때문에, 로컬 개발 환경과 실제 운영 환경의 차이를 줄이는 전략이 필수적입니다.

1. 환경 분리 전략 (3-Tier Environment)

SaaS 안정성을 위해 Local(개인) - Dev(통합 개발) - Prod(운영)의 3단계 구성을 제안합니다. 필요시 Staging(검증) 단계를 추가할 수 있습니다.

환경용도 (Purpose)구성 방식 (Infrastructure)배포 주체
Local개인별 기능 개발 및 단위 테스트Docker Compose 활용 (로컬 PC에 Redis, Kafka, DB 컨테이너 띄움)개발자 개인
Dev (개발계)MSA 서비스 간 통합 테스트, API 연동 확인, 클라이언트(FE) 개발용 서버AWS ECS (스펙 낮춤), RDS (Single AZ), ElastiCache (t3.micro 등 저사양)CI/CD (Develop 브랜치 푸시 시 자동 배포)
Prod (운영계)실제 고객 서비스 제공AWS ECS (Auto Scaling 적용), RDS (Multi-AZ), 고성능 인스턴스관리자 승인 후 배포 (Master/Main 브랜치)
전략 포인트:
  • Local 환경 표준화: docker-compose.yml 파일을 프로젝트 루트에 포함시켜, 신규 입사자도 명령어 한 번(docker-compose up)으로 로컬에 WiseNTM 구동 환경(Redis, Kafka, DB 포함)을 완벽하게 세팅할 수 있게 합니다.

2. 개발계(Dev) 데이터 활용 전략

SaaS 환경에서 가장 민감한 '고객 데이터 보호'와 '테스트 편의성' 사이의 균형을 맞춰야 합니다.

  • 원칙: 개발계 DB에는 실제 고객의 개인정보(전화번호, 녹취파일, 주민번호 등)를 절대 적재하지 않습니다.
  • 데이터 생성 방안:
    1. Mock Data Seeding: 개발 초기에 미리 정의된 가짜 데이터(Dummy Data)를 DB 초기화 시점에 자동으로 넣는 스크립트 실행 (User 100명, 통화 이력 100건 등).
    1. 데이터 마스킹(Masking): 운영계에서 발생한 버그를 잡기 위해 데이터가 꼭 필요할 경우, 별도의 ETL 툴을 통해 개인정보를 변환(홍길동 to 000, 전화번호 뒷자리 마스킹) 후 개발계로 이관합니다.
  • 파일 스토리지: 개발계는 S3의 dev-bucket을 별도로 사용하며, 일정 기간(예: 30일)이 지나면 자동 삭제되도록 수명주기(Lifecycle) 정책을 겁니다.

3. Git 브랜치 전략 (Git Flow for MSA)

개발계 활용을 위한 소스 코드 관리 전략입니다.

브랜치 종류는 다음과 같습니다.

  1. master: 제품 출시 버전을 관리하는 메인 브랜치
  1. develop: 다음 출시 버전을 위해 개발하는 브랜치
  1. feature: 새로운 기능을 개발하는 브랜치
  1. release: 다음 출시 버전을 준비하는 브랜치
  1. hotfix: 출시된 제품의 버그를 고치기 위한 브랜치

이러한 Git Flow 를 사용할 때, 어떻게 개발을 진행하는지 그 흐름에 대해서 살펴보도록 하겠습니다.

  1. 신규 기능 개발을 위해 개발자는 develop 브랜치를 기준으로 한 feature 브랜치를 따서 작업을 진행합니다.
  1. 작업이 완료된 feature 브랜치는 develop 브랜치로 병합 (Merge) 됩니다. (일반적으로 프로젝트 진행 시에는 Pull Request 를 통해 작업 내용을 Review 받은 후 해당 PR 을 Merge 하는 방식으로 진행합니다.)
  1. 다음 출시 버전을 위해 개발 중인 develop 브랜치에서 release 브랜치를 따서 배포를 위한 준비를 합니다. (이 때 발견되는 버그들은 release 브랜치에서 바로 반영합니다.)
  1. 충분한 테스트 후에, 일정한 주기로 (일반적으로는 배포하고자 하는 버전 단위) master 브랜치로 Merge 하여 제품을 출시합니다.
  1. 상용 배포 이후, release 브랜치에서 미처 발견되지 못한 새로운 버그들은 hotfix 브랜치에 바로 반영합니다.

개발계 시나리오

  • Feature: 개발자가 feature/login 브랜치에서 작업 to Local 테스트 완료.
  • Develop (PR): 작업분을 develop 브랜치로 Pull Request → Merge.
    • Action: Github Actions가 감지하고 Dev(개발계) ECS에 자동 배포.
    • QA: 기획자/테스터가 Dev 서버 접속하여 기능 확인.
  • Main/Master: 출시 준비 완료 시 develop을 main으로 Merge.
    • Action: Prod(운영계) 배포 (승인 절차 포함).

[예상 질문 사항]

1. 도커(Container) 기반이 왜 좋은가요?

도커는 애플리케이션을 실행하는 데 필요한 모든 것(코드, 런타임, 시스템 도구, 라이브러리 등)을 하나의 패키지로 묶어버리는 기술입니다. WiseNTM 고도화 관점에서 4가지 핵심 이점이 있습니다.

  1. 환경 일관성 ("내 컴퓨터에선 되는데 서버에선 안 돼요" 해결)
    • 기존에는 OS 버전이나 설치된 라이브러리 버전에 따라 에러가 잦았습니다. 도커는 어디서 실행하든 동일한 환경을 보장하므로, 유지보수가 간편해집니다.
  1. 가볍고 빠른 배포 (Efficiency)
    • 가상머신(VM)처럼 OS 전체를 설치하는 게 아니라, 호스트 OS의 커널을 공유하기 때문에 훨씬 가볍습니다. 이는 SaaS 환경에서 비용 절감으로 직결됩니다.
  1. MSA(Microservices Architecture) 최적화
    • WiseNTM 계획에 있는 API 서버, 소켓 서버, 배치 서버를 각각 독립된 컨테이너로 쪼개기 좋습니다. 특정 기능(예: 배치 서버)에 부하가 걸려도 다른 서버(API 서버)는 죽지 않습니다.
  1. 확장성 (Auto-Scaling)
    • 사용자가 몰릴 때 VM은 부팅에 몇 분이 걸리지만, 컨테이너는 몇 초 만에 실행됩니다. 트래픽에 따라 서버를 늘리고 줄이는 오토스케일링에 최적화되어 있습니다.

2. 다른 서버에 있는 컨테이너끼리는 어떻게 통신하나요?

"서버 A의 API 컨테이너"가 "서버 B의 Redis 컨테이너"와 통신해야 하는 상황을 가정해 보겠습니다. MSA 구조에서는 이 통신 설계가 핵심입니다

가장 많이 사용되는 3가지 방식이 있습니다.

방법 1: 호스트 IP와 포트 포워딩 (가장 원시적인 방법)

  • 방식: 서버 B가 자신의 8080 포트를 컨테이너의 80 포트와 연결(매핑)해 둡니다.
  • 통신: 서버 A의 컨테이너는 "서버 B의 공인 IP : 8080"으로 요청을 보냅니다.
  • 단점: 서버 IP가 바뀌거나, 포트가 충돌하면 일일이 수정해야 해서 관리가 매우 힘듭니다. 클라우드 환경에서는 비추천합니다.

방법 2: 오버레이 네트워크 (Overlay Network)

  • 방식: 도커 스웜(Swarm)이나 쿠버네티스(K8s) 같은 도구가 여러 서버를 덮는 거대한 가상 네트워크(Overlay)를 만듭니다.
  • 통신: 물리적으로는 다른 서버에 있어도, 컨테이너끼리는 마치 같은 공유기를 쓰는 옆자리 PC처럼 사설 IP로 통신할 수 있습니다.

방법 3: 서비스 디스커버리 (Service Discovery) - 추천 (AWS ECS/K8s 환경)

  • 방식: WiseNTM 계획에 언급된 AWS ECS나 GCP 환경에서는 "이름"으로 통신합니다.
  • 통신:
    1. 서버 B의 컨테이너가 켜질 때 "나 Redis-Service야, 내 IP는 10.0.1.5야"라고 등록소(Service Registry)에 신고합니다.
    1. 서버 A의 컨테이너는 " Redis-Service랑 통신하고 싶어"라고 요청하면, 시스템이 알아서 현재 IP로 연결해 줍니다.
  • 장점: 컨테이너가 죽었다가 다시 살아나서 IP가 바뀌어도, 이름은 그대로이기 때문에 개발자는 IP를 신경 쓸 필요가 없습니다.

3. 프로토콜 변경 관련 (MQTT to Kafka/Redis)

Q1. 잘 쓰고 있던 MQTT를 굳이 Kafka와 Redis 조합으로 바꾸려는 이유가 무엇입니까? 오버 엔지니어링 아닌가요?

  • 답변: 현재 MQTT 구조에서는 메시지 전송 후 추적이나 이력 관리가 어렵다는 단점이 있습니다.
    • Kafka를 도입하면 메시지를 디스크에 저장하므로 유실 방지와 이력 추적(Trace)이 가능해져 안정성이 높아집니다.
    • Redis의 Pub/Sub은 현재 접속 중인 사용자 상태(Presence)를 실시간으로 핸들링하는 데 MQTT보다 훨씬 빠르고 효율적입니다.
    • 즉, 단순 전송은 Redis가, 중요 데이터의 보관 및 파이프라인은 Kafka가 맡아 시스템의 신뢰성을 높이기 위함입니다.

4. MSA 분리 기준 관련

Q2. MSA로 가면 관리가 복잡해질 텐데, 서비스를 얼마나 잘게 쪼갤 계획인가요?

  • 답변: 너무 잘게 쪼개면 관리가 어려워지므로, 초기에는 '기능의 성격'에 따라 크게 4가지 그룹으로만 나눌 계획입니다
    1. API 서버: 비즈니스 로직 및 사용자 요청 처리
    1. 상태 관리(Status) 서버: 사용자 접속 상태 및 세션 관리
    1. 소켓(Socket) 서버: 실시간 알림 및 메시지 전송
    1. 배치(Batch) 서버: 통계 집계 및 대량 작업 처리

      이후 트래픽이 집중되는 곳만 부분적으로 분리하는 전략을 취해야 합니다.’

5. 클라우드 및 컨테이너 전략

Q3. AWS와 GCP 중 어디를 고려 중이며, 서버리스(Cloud Run) 대신 컨테이너(ECS/EC2)를 선택한 이유는 무엇입니까?

  • 답변: 현재 계획상 AWS EC2 기반의 ECS(Elastic Container Service) 구성을 우선 고려하고 있습니다
    • GCP의 Cloud Run 같은 서버리스 방식은 초기 설정은 쉽지만, WiseNTM처럼 소켓 연결(Stateful)을 오래 유지해야 하는 서비스에서는 비용 예측이 어렵고 연결 끊김 이슈가 발생할 수 있어 부적합하다고 판단했습니다
    • 대신 ECS를 사용하여 도커 컨테이너를 관리하면 기존 서버 환경과 유사하면서도 오토스케일링의 이점을 가져갈 수 있습니다

6. 커스터마이징 대응 (SaaS화)

Q4. 고객사마다 요구사항(Customizing)이 다 다른데, 통합된 SaaS 형태로 가면 개별 요구사항은 어떻게 반영하나요?

  • 답변: 이번 고도화의 핵심 목표 중 하나가 '다양하게 커스터마이징 된 솔루션의 통합'입니다
    • 기존처럼 소스 코드를 분기하는(Fork) 방식이 아니라, '설정(Configuration) 기반'으로 대응할 예정입니다.
    • 화면 설계서의 '설정' 메뉴에 있는 메뉴 관리, 부서 관리, 화면 권한 기능을 강화하여, 코드를 수정하지 않고도 고객사가 원하는 기능의 ON/OFF가 가능하도록 설계하는 것을 제안합니다.

7. 모니터링 및 장애 대응

Q6. 서비스가 여러 개로 쪼개지면(MSA), 장애가 났을 때 원인을 찾기 힘들지 않을까요?

  • 답변: 맞습니다. 그래서 '행위 이력 로그 기능'과 '메시지 추적 기능(브로커)'이 필수적입니다
    • 모든 요청에 고유 ID(Trace ID)를 부여하여, API 서버에서 시작된 요청이 소켓 서버를 거쳐 DB까지 가는 경로를 한눈에 볼 수 있는 분산 추적(Distributed Tracing) 시스템을 도입할 예정입니다.
    • 또한, 1차적으로 '사용자 모니터링 기능'을 통해 관리자가 실시간 접속 현황을 파악할 수 있도록 설계되어 있습니다

8. 보안 (개인정보 보호)

Q7. SaaS는 여러 고객사가 같은 시스템을 쓰는데, 데이터가 섞이거나 유출될 위험은 없나요?

  • 답변: 논리적인 데이터 분리(Multi-tenancy)를 철저히 적용할 것입니다.
    • DB 레벨에서 Company_ID 컬럼으로 엄격하게 데이터를 격리하거나, 필요시 고객사별로 DB 스키마(Schema)를 분리하는 전략을 사용합니다.
    • 추가로 화면 설계에 포함된 '개인정보 마스킹' 기능과 '전화기록/문자내역 삭제' 기능을 통해, 민감 정보의 생명주기를 시스템적으로 관리하여 보안 우려를 해소하겠습니다

9. 대용량 파일 처리 (녹취/이미지)

Q8. 모바일 Box 기능이나 녹취 파일은 용량이 클 텐데, 이를 처리 서버가 직접 감당하면 부하가 심하지 않을까요?

  • 답변: 맞습니다. 그래서 파일 업로드/다운로드는 API 서버를 거치지 않고, 클라우드 스토리지(예: AWS S3)와 클라이언트가 직접 통신(Pre-signed URL)하도록 설계하여 서버 부하를 줄일 것입니다.
    • 화면 설계서의 '모바일 Box'와 '녹취 청취' 기능은 메타데이터(파일 정보)만 DB에서 가져오고, 실제 파일 스트리밍은 스토리지 서버가 전담하는 구조로 개발됩니다 [WiseNTM_화면설계본.pptx] 참고
