WiseN TM 쿠팡

  • Public Subnet
    • ALB (HTTP/L7)
    • 외부용 NLB (MQTT)
    • Internet Gateway 연결
  • Private Subnet
    • EC2 기반 WAS
    • Docker/Containerized WAS
    • Message Broker
    • 필요 시 내부용 NLB
  • 멀티 AZ
    • AZ-a, AZ-c 각각 동일하게 구성
    • 한쪽 AZ 장애 시 다른 AZ에서 계속 서비스

아래처럼 단계별로 보면 됩니다.

1. 먼저 전체 구조를 이렇게 이해하면 됩니다

HTTP 계열

  1. 휴대폰/PC가 외부 도메인으로 접속
  1. Public Subnet의 ALB 가 요청 수신
  1. ALB가 Private Subnet 안의 WAS EC2 또는 컨테이너 WAS 로 전달

MQTT 계열

  1. 단말/앱이 MQTT 접속
  1. Public Subnet의 NLB 가 TCP/TLS 레벨로 수신
  1. Private Subnet 안의 Broker 또는 MQTT 처리 서버 로 전달

내부 서비스 연동

  • Private Subnet 내 WAS가 같은 서브넷 또는 다른 AZ의 브로커와 통신
  • 컨테이너, 브로커, WAS 간 내부 트래픽은 VPC 내부망으로만 통신
  • 외부 인터넷은 직접 노출하지 않음

즉, 질문하신 “private subnet을 이렇게 구성” 한다는 것은 결국 아래 의미입니다.

  • 애플리케이션 서버는 private subnet
  • public IP 없음
  • 외부 트래픽은 반드시 ALB/NLB를 통해서만 들어옴
  • 운영/배포/패치용 외부 통신은 NAT Gateway 또는 SSM으로 처리
  • AZ별로 동일 구성하여 이중화

2. VPC 설계를 먼저 잡아야 합니다

예시로 VPC를 하나 만든다고 하면:

  • VPC: 10.0.0.0/16

서브넷은 최소 4개 이상 권장입니다.

Public Subnet

  • 10.0.1.0/24 → AZ-a
  • 10.0.2.0/24 → AZ-c

Private App Subnet

  • 10.0.11.0/24 → AZ-a
  • 10.0.12.0/24 → AZ-c

필요하면 더 나눌 수도 있습니다.

  • Private App Subnet: WAS/Container
  • Private Data Subnet: Broker/DB/Cache
  • Private Management Subnet: bastion/ops

하지만 그림 수준에서는 보통 아래 2계층으로도 충분합니다.

  • Public
  • Private

3. Public / Private Subnet의 차이는 라우팅입니다

많이 헷갈리는 부분인데, 서브넷이 public인지 private인지는 이름이 아니라 Route Table이 결정합니다.

Public Subnet의 Route Table

예:

  • 10.0.0.0/16 → local
  • 0.0.0.0/0 → Internet Gateway

이렇게 되어 있으면 인터넷과 직접 통신 가능하므로 Public Subnet입니다.

Private Subnet의 Route Table

예:

  • 10.0.0.0/16 → local
  • 0.0.0.0/0 → NAT Gateway

또는 아예 인터넷 outbound도 막고 싶으면:

  • 10.0.0.0/16 → local

이 경우 외부에서 직접 접근 불가, public IP도 안 붙이므로 Private Subnet입니다.

4. Internet Gateway와 NAT Gateway를 어떻게 둘지

이 구조에서 보통 이렇게 둡니다.

Internet Gateway

  • VPC에 1개 연결
  • Public Subnet의 ALB, Public NLB가 인터넷 수신 가능

NAT Gateway

Private 서버가 아래 작업을 해야 하면 필요합니다.

  • OS 패키지 다운로드
  • 외부 API 호출
  • Docker image pull
  • CloudWatch/SSM/외부 인증서 갱신 등

권장 구성은:

  • NAT Gateway를 AZ별 Public Subnet에 1개씩
    • AZ-a Public Subnet에 NAT 1개
    • AZ-c Public Subnet에 NAT 1개

그리고

  • AZ-a Private Route Table → NAT-a
  • AZ-c Private Route Table → NAT-c

이렇게 해야 AZ 독립성이 좋아집니다.

비용 아끼려고 NAT 하나만 둘 수도 있지만,

  • 크로스 AZ 트래픽 발생 가능
  • 특정 AZ 장애 시 outbound 영향 가능

그래서 운영서비스면 AZ별 NAT가 더 안전합니다.

5. ALB / NLB는 어디에 배치해야 하나

그림 기준으로는 두 종류입니다.

5-1. ALB (HTTP/L7)

ALB는 보통 Public Subnet 2개 이상에 배치합니다.

예:

  • Public Subnet AZ-a
  • Public Subnet AZ-c

ALB Listener:

  • 80 → 443 redirect
  • 443 → Target Group(Private EC2 or ECS)

Target Group 대상:

  • Private Subnet의 WAS EC2
  • 또는 ECS 서비스/컨테이너
  • 또는 IP target 방식

왜 ALB를 Public에 두나

외부 클라이언트 HTTP/HTTPS 요청을 받아야 하기 때문입니다.

5-2. NLB (MQTT)

MQTT는 보통 TCP/TLS 레벨이라 NLB가 맞습니다.

외부 단말이 MQTT로 들어오면:

  • Public NLB
  • Listener: 1883 또는 8883(TLS)
  • Target Group: Private Subnet의 Broker/WAS

이렇게 갑니다.

포트 예시

  • MQTT 평문: 1883
  • MQTT over TLS: 8883
  • WebSocket MQTT: 경우에 따라 443 + ALB/NLB 조합도 가능

실서비스면 평문보다 TLS(8883) 권장입니다.

6. Private Subnet 안에는 무엇을 어떻게 두나

그림대로라면 Private Subnet 안에 다음이 들어갑니다.

AZ-a

  • WAS #1 EC2
  • Message Broker #1 EC2
  • Containerized WAS Layer

AZ-c

  • WAS #2 EC2
  • Message Broker #2 EC2
  • Containerized WAS Layer

이때 실무적으로는 2가지 패턴이 있습니다.

패턴 A. EC2 안에 Docker 직접 운영

각 AZ의 EC2 안에 Docker를 띄워서:

  • Nginx / App container
  • Worker
  • MQTT handling app

이런 식으로 운영

장점

  • 단순
  • 익숙함
  • 빠르게 구축 가능

단점

  • 오토스케일/배포/헬스체크/운영 자동화가 ECS보다 불편

패턴 B. ECS on EC2 또는 ECS Fargate

더 AWS스럽게 가려면:

  • ECS Cluster
  • Service
  • Task
  • Target Group 연동

이 구조가 더 깔끔합니다.

다만 그림은 EC2 + Docker 느낌이 강해서, 현재 설계 의도는 아마 EC2 위 Docker 로 보입니다.

7. “Private Subnet으로 이렇게 구성”하는 실제 생성 순서

아래 순서대로 만들면 됩니다.

1단계. VPC 생성

예:

  • 10.0.0.0/16

DNS 옵션도 켜두는 게 좋습니다.

  • DNS hostnames: enabled
  • DNS resolution: enabled

2단계. Subnet 생성

Public

  • subnet-public-a / AZ-a / 10.0.1.0/24
  • subnet-public-c / AZ-c / 10.0.2.0/24

Private

  • subnet-private-app-a / AZ-a / 10.0.11.0/24
  • subnet-private-app-c / AZ-c / 10.0.12.0/24

여기서 Private Subnet은 auto-assign public IP 비활성화 상태여야 합니다.

3단계. IGW 연결

  • Internet Gateway 생성
  • VPC에 Attach

4단계. Route Table 생성

Public Route Table

  • 0.0.0.0/0 -> IGW
  • Public Subnet 2개 연결

Private Route Table A

  • 0.0.0.0/0 -> NAT Gateway A
  • Private Subnet A 연결

Private Route Table C

  • 0.0.0.0/0 -> NAT Gateway C
  • Private Subnet C 연결

인터넷 outbound가 아예 필요 없으면 NAT 대신 local만 둬도 됩니다.

5단계. NAT Gateway 생성

  • Public Subnet A에 NAT-A
  • Public Subnet C에 NAT-C
  • 각각 Elastic IP 연결

6단계. 보안그룹 설계

이 부분이 제일 중요합니다.

ALB Security Group

인바운드

  • 80 from 0.0.0.0/0
  • 443 from 0.0.0.0/0

아웃바운드

  • WAS SG의 8080/80/443 등 앱 포트

WAS EC2 Security Group

인바운드

  • ALB SG에서만 8080 또는 80/443 허용
  • 운영용이면 SSM 권장
  • SSH는 가능하면 열지 않음

아웃바운드

  • Broker 포트
  • 내부 API
  • CloudWatch/SSM/NAT outbound

MQTT Broker SG

인바운드

  • Public NLB 타겟 포트 허용
  • 예: 1883, 8883
  • WAS SG에서 broker 관리 포트 허용

아웃바운드

  • 필요 최소
참고로 NLB는 Security Group 동작 방식이 ALB와 다르게 느껴질 수 있으니, 실제 타겟 포트 기준으로 인스턴스 SG/NACL을 같이 봐야 합니다.

7단계. ALB 생성

  • Internet-facing ALB
  • Public Subnet A, C 선택
  • HTTPS 리스너 구성
  • ACM 인증서 연결
  • Target Group 생성
    • 타입: instance 또는 ip
    • 대상: Private Subnet의 WAS EC2 또는 컨테이너 IP

헬스체크 경로도 설정합니다.
예:

  • /health
  • /actuator/health

8단계. NLB 생성

MQTT용으로 생성

  • Scheme: internet-facing
  • Public Subnet A, C
  • Listener:
    • TCP 1883
    • 또는 TLS 8883
  • Target Group:
    • Broker EC2
    • 또는 MQTT 처리 WAS

9단계. Private Subnet에 EC2 배치

각 AZ에 최소 1대씩

예:

  • ec2-was-a
  • ec2-broker-a
  • ec2-was-c
  • ec2-broker-c

설정 시 주의:

  • Public IP 할당 안 함
  • Private IP만 사용
  • IAM Role 연결
    • CloudWatch Agent
    • SSM
    • ECR pull 필요 시 ECR 권한
  • 보안그룹 최소 허용

10단계. Docker/애플리케이션 배포

Private EC2에 Docker 또는 container runtime 설치 후:

  • App container 실행
  • MQTT broker 실행
  • reverse proxy 필요 시 Nginx 추가
  • systemd 또는 docker compose로 부팅 자동화

예시:

  • EC2 WAS #1 → App container 2개
  • EC2 Broker #1 → EMQX / Mosquitto / RabbitMQ MQTT plugin 등

8. Route 53 도메인 연결은 어떻게 하나

보통 이렇게 연결합니다.

HTTP 도메인

  • api.example.com → ALB Alias

MQTT 도메인

  • mqtt.example.com → NLB Alias

이렇게 하면 외부에서는 도메인만 보고 접속하고,
실제 서버는 Private Subnet에 숨겨집니다.

9. 운영 접속은 어떻게 해야 하나

Private Subnet 서버는 외부에서 직접 SSH 접속하지 않는 게 좋습니다.

권장 순서:

가장 권장

  • AWS Systems Manager Session Manager
  • 인스턴스에 SSM Agent
  • IAM Role 부여
  • VPC Endpoint까지 구성하면 NAT 없이도 가능

이 방식이면:

  • bastion host 불필요
  • 22번 포트 불필요
  • 감사 추적도 쉬움

차선책

  • Bastion Host를 Public Subnet에 둠
  • Bastion에서 Private EC2로 SSH

하지만 요즘은 SSM이 더 깔끔합니다.

10. Private Subnet에서 외부로 나가야 하는 경우

서버가 외부와 통신해야 할 수도 있습니다.

예:

  • OS 업데이트
  • 패키지 설치
  • 외부 API 호출
  • Docker image pull
  • NTP, 인증서 관련

이때 선택지는 2개입니다.

방식 1. NAT Gateway 사용

가장 일반적

방식 2. VPC Endpoint 적극 활용

AWS 서비스만 쓴다면 NAT를 줄일 수 있습니다.

예:

  • S3 Gateway Endpoint
  • ECR API Endpoint
  • ECR DKR Endpoint
  • CloudWatch Logs Endpoint
  • SSM Endpoint
  • EC2 Messages Endpoint
  • SSM Messages Endpoint

이렇게 하면 Private Subnet 서버가 인터넷 없이도 AWS 서비스와 내부적으로 통신 가능합니다.

실무적으로는:

  • AWS 서비스 위주면 VPC Endpoint 적극 권장
  • 외부 인터넷 호출 있으면 NAT 필요

11. 그림처럼 AZ별 이중화하려면 꼭 필요한 것

이 부분이 핵심입니다.

꼭 맞춰야 하는 것

  • Private Subnet을 AZ별로 분리
  • 각 AZ에 동일한 역할의 서버 배치
  • ALB/NLB를 2개 AZ 이상에 걸쳐 배치
  • Target Group에 각 AZ의 서버 등록
  • 헬스체크 활성화
  • 세션 저장소가 있다면 외부화
    • Redis
    • DB
    • ElastiCache
    • DynamoDB 등

주의

WAS를 2대 두었는데 세션을 로컬 메모리에만 저장하면 장애 시 로그인/상태 유지 문제가 날 수 있습니다.

그래서

  • Stateless하게 설계하거나
  • 세션 저장소 분리해야 합니다.

12. 이 구조에서 자주 하는 실수

1. Private Subnet인데 public IP 붙임

그러면 사실상 private 설계 의미가 약해집니다.

2. ALB 없이 Private EC2를 직접 노출

보안/운영성 모두 안 좋아집니다.

3. AZ는 2개인데 NAT는 1개만 두고 장애 고려 안 함

운영환경이면 신중해야 합니다.

4. 보안그룹을 0.0.0.0/0으로 과하게 오픈

특히 Private EC2의 8080, 22, 1883 같은 포트

5. 헬스체크 경로 없음

ALB가 정상/비정상을 판단 못합니다.

6. NLB와 ALB 역할을 혼동

  • HTTP/HTTPS, URL/path 기반 분기 → ALB
  • TCP/TLS, MQTT → NLB

13. 구조를 AWS 리소스 기준으로 풀면

아래 식으로 매핑하면 됩니다.

네트워크

  • 1 VPC
  • 2 Public Subnet
  • 2 Private Subnet
  • 1 IGW
  • 2 NAT Gateway

로드밸런서

  • 1 Internet-facing ALB
  • 1 Internet-facing NLB(MQTT)
  • 필요 시 1 Internal NLB

컴퓨팅

  • AZ-a Private:
    • WAS EC2
    • Broker EC2
  • AZ-c Private:
    • WAS EC2
    • Broker EC2

컨테이너

  • 각 WAS EC2 내부 Docker container 2개 이상
    또는
  • ECS Service 2세트

운영

  • IAM Role
  • CloudWatch Logs
  • SSM Session Manager
  • Route 53
  • ACM

14. 가장 추천하는 실제 구현안

그림대로 최대한 유지하면서 운영성 좋게 가려면 이렇게 추천합니다.

추천안

  • Public Subnet
    • ALB
    • MQTT용 NLB
    • NAT Gateway
  • Private Subnet
    • WAS는 ECS 또는 EC2+Docker
    • Broker는 전용 EC2 또는 매니지드 서비스 검토
  • 운영 접속
    • Bastion 대신 SSM
  • AWS 서비스 통신
    • VPC Endpoint 적극 사용
  • 도메인
    • Route 53 + ACM
  • 고가용성
    • AZ-a, AZ-c 동일 배치

15. 아주 간단한 배치 예시

Public

  • subnet-public-a
    • ALB ENI
    • NLB ENI
    • NAT Gateway A
  • subnet-public-c
    • ALB ENI
    • NLB ENI
    • NAT Gateway C

Private

  • subnet-private-a
    • EC2-WAS-A
    • EC2-BROKER-A
  • subnet-private-c
    • EC2-WAS-C
    • EC2-BROKER-C

트래픽

  • 인터넷 → ALB → WAS-A/C
  • 인터넷 → NLB → BROKER-A/C
  • WAS-A ↔ BROKER-A/C
  • WAS-C ↔ BROKER-A/C
