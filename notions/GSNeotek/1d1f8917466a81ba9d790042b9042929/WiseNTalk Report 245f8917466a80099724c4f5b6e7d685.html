Zendesk-Connect CTI 기능 개선 보고: 전화번호 식별 옵션 추가

작성일: 2025-08-05

담당자: 권신범

검토자:

1. 개요 (Overview)

본 프로젝트는 Zendesk 내 Amazon Connect 소프트폰의 개인정보보호 강화를 목표로 진행되었습니다. Zendesk 관리자 설정에 "전화번호 식별 활성화" 옵션을 추가하여, 해당 설정에 따라 소프트폰 UI와 Zendesk 티켓에 기록되는 데이터(요청자, 제목, 본문 등)에서 전화번호를 선택적으로 마스킹하는 기능을 구현했습니다.

이를 통해 고객의 민감 정보를 보호하고, 기업의 컴플라이언스 요구사항을 충족시키는 안정적인 CTI 솔루션 기반을 마련했습니다.

2. 주요 기능 및 최종 로직 설명

이번 기능의 핵심은 설정(Configuration), 데이터 흐름(Data Flow), UI/데이터 마스킹(Masking), 그리고 안정성 강화(Stability) 네 가지로 요약할 수 있습니다.

2-1. 설정 (Configuration)

  • manifest.json: Zendesk 관리자 페이지에 phoneNumberIdentification이라는 checkbox 파라미터를 추가했습니다. 이 파일은 실제 Zendesk 환경에 앱이 설치되었을 때 사용자가 보게 될 설정 UI를 정의합니다.
  • zcli.apps.config.json: 로컬 개발 환경에서만 사용하는 설정 파일입니다. 여기에 동일한 phoneNumberIdentification 키를 추가하여, 개발자가 Zendesk UI에 접속하지 않고도 true/false 값을 바꿔가며 기능을 테스트할 수 있도록 했습니다.

2-2. 데이터 흐름 (Data Flow Architecture)

데이터 동기화 및 타이밍 문제를 해결하기 위해 아래와 같은 안정적인 단방향 데이터 흐름을 확립했습니다.

  1. 데이터 로딩 (Zendesk.js): 앱이 시작되면 Zendesk.js가 ZAF(Zendesk App Framework)를 통해 manifest.json의 모든 설정값을 가져옵니다.
  1. 중앙 저장소 저장 (localStorage): 가져온 설정값 전체를 localStorage.setItem("zdConfig", ...)를 통해 브라우저의 localStorage에 즉시 저장합니다. 이로써 모든 컴포넌트가 언제든 최신 설정값을 조회할 수 있는 단일 진실 공급원(Single Source of Truth)이 마련됩니다.
  1. UI 상태 전파 (Content.js → Voice.js): Content.js는 Redux 스토어의 zafInfo 또는 config를 구독하여 전체적인 앱의 상태를 관리합니다. 동시에, Voice.js와 같은 자식 컴포넌트에는 isPhoneNumberVisible이라는 prop을 전달하여 UI 렌더링을 제어합니다. (최종적으로는 각 컴포넌트가 localStorage를 직접 조회하는 방식으로 안정성을 더욱 높였습니다.)

2-3. UI 및 데이터 마스킹 (Masking Logic)

phoneNumberIdentification 설정이 false일 때, 아래 두 영역에서 마스킹이 적용됩니다.

  • 소프트폰 UI (Voice.js):
    • render() 함수가 실행될 때 localStorage에서 직접 phoneNumberIdentification 값을 읽어옵니다.
    • JSX 코드 내에서 전화번호를 표시하는 모든 부분(displayNum 등)을 삼항 연산자로 처리하여, 값이 false일 경우 '**********'를 표시합니다.JavaScript
      <strong>{isPhoneNumberVisible ? item.displayNum : '**********'}</strong>
  • Zendesk 티켓 데이터 (Zendesk.js):
    • 티켓 제목: createTicketByVoice 함수가 티켓 제목을 생성할 때, 설정값에 따라 전화번호 대신 '**********'를 포함하여 생성합니다.
    • 티켓 댓글 (통화 기록): updateTicketWithContactDetails 함수가 티켓에 댓글을 추가할 때, '발신자/수신자 전화번호' 필드 값을 마스킹 처리하여 저장합니다.
    • 신규 사용자 이름: createUserByPhoneNumber 함수가 신규 사용자를 생성할 때, 설정값이 false이면 "전화고객: 010-1234-5678"이 아닌 **"전화고객: ********" 와 같이 마스킹된 이름으로 사용자를 생성하여 요청자 정보 자체에 민감 정보가 남지 않도록 처리했습니다.

2-4. 안정성 강화 (Stability Enhancement)

  • 중복 사용자 생성 방지: 동일한 번호로 거의 동시에 여러 통의 전화가 올 경우, 해당 번호에 대한 유저가 중복으로 생성되는 경합 조건(Race Condition) 문제가 있었습니다.
  • 해결: isCreatingUser라는 Set 객체를 '잠금(Lock)' 장치로 활용했습니다. searchUserByPhoneNumber 함수가 시작될 때 전화번호를 Set에 추가하고, 작업이 끝나면 finally 구문을 통해 성공/실패 여부와 관계없이 반드시 Set에서 제거하도록 하여, 특정 번호에 대한 유저 생성 작업은 한 번에 하나만 실행되도록 보장했습니다.

3. 문제점 및 해결 방안

배포 후 발생할 수 있는 잠재적 이슈와 그에 대한 해결 방안입니다.

예상 문제점원인해결 방안
1. 설정 변경이 즉시 반영되지 않는 현상브라우저 캐시(Cache) 문제로, 사용자가 설정을 변경해도 브라우저가 이전 버전의 JavaScript 파일을 계속 사용함.사용자 가이드: 설정 변경 후에는 반드시 강력 새로고침(Ctrl+Shift+R)을 하도록 안내합니다.
2. 기존에 생성된 사용자의 전화번호 노출이번 코드는 새로 생성되는 사용자의 이름만 마스킹 처리함. 이미 "전화고객: 010..." 형태로 생성된 사용자는 그대로 유지됨.수동 조치 필요: Zendesk 관리 센터의 사람 > 구성원 메뉴에서 '전화고객:'으로 검색하여, 이름에 전화번호가 노출된 사용자를 수동으로 수정해야 합니다.
3. 특정 상황에서 설정값이 늦게 반영될 가능성localStorage를 사용해 타이밍 문제를 대부분 해결했지만, 앱 초기 로딩 시 아주 드물게 이전 설정값으로 UI가 잠시 보일 수 있음.현상 유지: 현재 로직으로 대부분의 문제를 해결했으며, 찰나의 UI 깜빡임은 사용자 경험에 큰 영향을 주지 않는 수준으로 판단됩니다. 추후 더 정교한 상태 관리가 필요할 시 Redux-Saga 등 비동기 미들웨어 도입을 고려할 수 있습니다.
4. 중복 로그인 로직으로 불특정하게 한 콜에 여러개 티켓이 생성됨브라우저 캐시 및 localStorage의 유저 아이디가 각 브라우저마다 저장됨세션 아이디를 저장할 수 있는 서버를 구축

문제 개요

현상: 여러 브라우저/탭에서 동시 접속 시 하나의 인바운드 콜에 대해 여러 개의 티켓이 중복 생성됨

영향도:

  • 고객 한 통화 → 티켓 N개 생성 (N = 열린 탭 개수)

문제 원인 분석

1️⃣ 중복 로그인 감지 로직 부재

현재 구조

// src/views/layout/Content.js - 중복 로그인 UI는 존재
{this.props.loginStatus.duplicated ?
    <div className="duplicate-login-warning">
        <h1>중복 로그인 상태입니다.</h1>
        <p>다른 브라우저에서 이미 로그인되어 있습니다.</p>
        <p>소프트폰 기능이 제한됩니다.</p>
    </div>
    :
    // 정상 소프트폰 UI
}

문제점

❌ 타이밍 문제: 중복 체크가 너무 이른 시점(subscribeToAgentEvents)에서 실행

❌ Redux 상태 변경 로직 없음

  • this.props.loginStatus.duplicated를 true로 설정하는 코드가 존재하지 않음
  • CCP.js, Zendesk.js 어디에서도 중복 감지 로직을 찾을 수 없음

❌ 감지 메커니즘 없음

  • BroadcastChannel, LocalStorage, Server API 등 탭 간 통신 수단 없음
  • 각 탭이 독립적으로 동작하여 서로의 존재를 모름
  • 초기화 시에만 체크하고 이후 지속적으로 감시하지 않음

중복 로그인으로 인해 한 통화로 인해 여러개의 티켓이 생성됨

중복 로그인 현상

(3개의 서로 다른 브라우저, 탭에서 동시에 로그인됨)

근본적 문제

🔥 "모든 탭이 메인이라고 생각"

  • 각 탭이 독립적으로 Amazon Connect Streams API에 연결
  • 동일한 인바운드 이벤트를 N개 탭이 모두 수신
  • 각 탭이 "내가 이 콜을 처리해야 한다"고 판단

🔥 세션 관리 부재

  • 리더 탭 선출 로직 없음
  • 탭 간 조정 메커니즘 없음
  • 이벤트 중복 처리 방지 로직 없음

  1. 서버 기반 세션 관리
    • Session API Server 구축 (Node.js + Express)
    • 사용자당 하나의 활성 세션만 허용
    • 리더-팔로워 구조로 역할 분담
  1. 클라이언트 통합
    • serverSessionManager.js 추가
    • Zendesk.js, CCP.js에 세션 체크 로직 추가
    • Redux 상태 연동
  1. 이벤트 처리 개선
    • 리더 탭에서만 Amazon Connect 이벤트 처리
    • 팔로워 탭은 UI만 표시하고 실제 로직 비활성화

결론

현재 문제의 핵심:

중복 로그인 감지 로직의 완전한 부재로 인해 모든 탭이 독립적으로 동작하여 중복 티켓이 생성됨

해결 방안:

서버 기반 세션 관리를 통한 리더-팔로워 구조 도입으로 단일 책임 원칙 적용

기대 효과:

완벽한 중복 방지 + 데이터 정합성 보장 + 사용자 경험 개선

이 문제는 아키텍처 수준의 개선이 필요하며, 단순한 버그 수정이 아닌 시스템 설계 변경을 통해서만 근본적으로 해결할 수 있습니다.
