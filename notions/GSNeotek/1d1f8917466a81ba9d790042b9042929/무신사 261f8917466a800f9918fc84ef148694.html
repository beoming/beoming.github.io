무신사

무신사 PoC 요청사항 관련 기술적 제약 및 고려사항

무신사에서 요청하신 Conversational Agents(주) + Agent Assist(부) + Insights AI(부) PoC는 Google Cloud의 핵심 컨택센터 AI 기술 스택을 검증하는 이상적인 조합입니다.

다만, 각 솔루션의 가치를 PoC 기간 내에 명확히 검증하기 위해서는 AI 모델의 핵심 성능 검증과 한국어 처리 및 지원 능력이 핵심 전제 조건입니다.

AI 모델(Gemini) 성능 및 한국어 지원 능력 검증

Google CES 기능의 근간이 되는 AI 모델의 한국어 처리 능력과 한국어 자료 학습에 대한 신뢰도를 검증하는 단계입니다.

1.1 답변 품질 및 신뢰성 ( Hallucination ) 검증

  • 검증 목표 : 챗봇 또는 음성봇이 학습 데이터에 없는 내용에 대해 답변을 지어내거나 사실과 다른 정보를 제공하는 현상이 없는지 확인해야 합니다.
  • 실행 방안 : PoC 시나리오의 핵심 질문들에 대한 정답(Ground Truth) 을 사전에 공동으로 정의하고, AI가 생성한 답변과 비교하여 정답률과 신뢰도를 측정합니다.

1.2 한국어 자연어 처리(NLP) 검증

  • 검증 목표 : 한국어의 복잡한 문장 구조, 중의적 표현, 구어체, 신조어, 높임말, 사투리 등을 Gemini가 얼마나 정확하게 이해하고, 대화의 맥락에 맞는 자연스러운 답변을 생성하는지 검증합니다.
  • 실행 방안 : 시나리오의 ‘고객 문의 예시’외, 실제 고객들이 사용할 법한 다양한 형태의 복잡한 문장으로 테스트를 진행하여 Gemini의 한국어 이해력을 검증합니다.

1.3 음성 인식 및 사투리 대응 능력 검증

  • 검증 목표 : 음성봇(Voice Bot)이 한국어(표준어)는 물론, 국내 주요 사투리와 다양한 억양, 발음을 얼마나 잘 인식하고 답변할 수 있는지 검증합니다.
  • 실행 방안 : 주요 지역별(경상, 전라, 충청 등) 사투리 및 억양이 포함된 테스트 스크립트를 공동으로 작성하고, 실제 음성으로 PoC를 진행하여 인식률을 측정합니다.

1.4 학습 데이터 요구사항 확인

  • 검증 목표 : 최적의 성능을 내기 위한 학습 데이터의 조건과 제약을 명확히 해야합니다.
  • 주요 확인 사항 :
    • 데이터 분량 : DataStore의 답변 품질을 높이기 위한 최소/권장 데이터(FAQ, 정책문서 등)의 분량은 어느 정도인지 논의합니다.
    • 파일 제약 : 업로드하는 개별 파일(Pdf, Txt, Html)의 최대 용량 또는 글자 수 제한이 있는지 확인합니다.

Conversational Agents (Dialogflow CX) 검증 관련

요청사항: PoC 우선 시나리오로 Conversational Agents를 기본으로 검증 희망

Conversational Agents로 의미 있는 시나리오(예: 주문/배송 조회, 반품/교환 접수)를 구현하기 위해서는 무신사의 백엔드 시스템과의 실시간 데이터 연동이 필수적입니다.

요청 시나리오는 기술적 구현 난이도에 따라 크게 3단계로 구분하였습니다.

</br>

Level 1. 정적 데이터 조회(Data Store 활용)

  • 해당 시나리오 :
    • 단순 변심, 주문 실수 시 반품/교환’정책’ 및 ‘절차’, ‘배송비’ 안내
  • 기술 구현 :
    • 무신사 스토어의 FAQ, 정책 안내 페이지 URL을 DataStore에 연결하여 Gemini가 학습 후 관련 질문에 자동으로 답변하도록 구성합니다.
  • 제약 사항 :
    • 데이터 정제 관련 : Gemini가 학습할 웹페이지의 내용이 텍스트 기반으로 명확하고 체계적으로 정리되어 있어야 합니다. 핵심 정보가 ‘이미지’ 안에 있거나, 복잡한 UI(탭, 아코디언 등) 속에 숨겨져 있다면 인식률이 저하될 수 있습니다.
    • 정책 안내 페이지 URL 목록 및 해당 페이지의 크롤링 허용 여부 확인 필요

</br>

Level 2. 실시간 동적 데이터 조회

  • 해당 시나리오 :
    • 배송 조회 : “배송이 시작되지 않아요”, “송장 번호 조회가 안돼요”
    • 누락 문의 : “주문한 상품 중 일부가 오지 않아요”
  • 기술 구현 :
    • 고객의 주문번호를 입력받고, 무신사 주문/물류 시스템의 API를 호출하여 실시간 배송 정보 확인, 송장 번호, 출고 상품 목록 등을 조회 후 답변해야 합니다.
  • 제약 사항 :
    • Read-Only API 필수 : 주문번호를 기반으로 배송 상태, 출고 내역을 조회할 수 있는 API가 Poc 기간 내에 제공되어야 합니다. 이 API가 없다면 해당 시나리오 검증은 불가능합니다.
  • 확인 필요 :
    • 주문 정보 조회 API 존재 여부 및 명세서
    • Poc를 위한 테스트용 API 엔드포인트 및 인증 방식

</br>

Level 3. 복합 트랜잭션 방법에 대한 논의 확정 필요

  • 해당 시나리오 :
    • 검수 불만 으로 인한 반품/교환 ‘접수’
    • 교환 요청 시 다른 사이즈/색상 ‘재고 확인’ 후 ‘교환 접수’
  • 기술 구현 1 :
    • 시스템에 실제 데이터를 생성하거나 수정(write)하는 작업입니다.
    • Read & Write API 활용 : 재고 확인 API, 반품/교환 접수 처리 API, 등 시스템을 직접 제어하는 API를 개발하여 실행합니다.
  • 기술 구현 2 :
    • Cloud Run을 활용하여 Target DB를 지정 후 데이터를 입력 및 수정합니다.
    • Publie IP 또는 Cloud VPN을 활용하여 GCP 내부 또는 외부 데이터베이스와 연결하여 통신할 수 있습니다.
  • 제약 사항 :
    • 트랜잭션 데이터를 기록할 Target DB의 확정 필요성
      • 고객의 요청이 실시간으로 기록되어야 하며, 어떤 데이터베이스에 어떤 형태로 저장할지에 대한 정의가 선행되어야 합니다.
    • BigQuery는 분석용 DB이므로 실시간 트랜잭션 처리에는 적합하지 않습니다. 단일 요청을 즉시 처리하고 기록하는 용도로는 Cloud SQL(MySQL, PostgreSQL)과 같은 정형 RDBMS나 Firebase, Mongo와 같은 비정형 NoSQL DB가 적합합니다.
    • Target DB가 확정되더라도 챗봇 또는 음성봇이 해당 DB에 안전하게 접근하여 데이터를 생성/수정할 수 있어야 합니다.

Level 4. 백엔드 시스템 연동 및 API 가용성

  • 제약사항
    • 구체적인 설명 및 확인 필요사항
      • API 존재 여부: PoC에서 구현할 시나리오(예: 주문번호 기반 배송 상태 조회)를 위한 API가 현재 존재하는가?
      • API 문서: API 명세서(Request/Response 형식, 인증 방식 등)가 제공될 수 있는가?
      • API 환경: PoC를 위한 개발/테스트용 API 엔드포인트(Endpoint)가 제공되는가? 실제 운영 DB에 영향을 주지 않는 환경이 필요합니다.
      • 인증/권한: API 호출을 위한 인증 토큰 발급 등 보안 정책은 어떻게 되는가? GS네오텍/Google의 개발 환경 IP에 대한 방화벽 허용이 필요한가?

<br>

Level 5. 지식 데이터베이스(Knowledge Base) 활용을 위한 데이터 형태

  • 제약사항

    상품 정보, 정책 안내 등 방대한 데이터를 효과적으로 활용하는 Data Store 기능을 검증하려면, Google AI가 학습할 수 있는 형태의 데이터 소스가 필요합니다. 데이터가 분산되어 있거나 비정형적인 포맷(예: 이미지 속 텍스트, PDF) 위주일 경우 데이터 전처리 과정이 추가로 필요합니다.

  • 구체적인 설명 및 확인 필요사항
    • 데이터 소스: FAQ, 상품 정보, 반품 정책 등의 정보가 웹사이트(URL) 형태로 존재하는가, 아니면 내부 문서(PDF, DOCX) 형태로 관리되는가?
    • 데이터 구조: 웹사이트의 경우, 정보가 체계적인 구조(HTML 태그)로 정리되어 있는가? 크롤링이 용이한 구조인지 확인이 필요합니다.
    • 데이터 범위: PoC 기간 동안 테스트에 사용할 특정 카테고리나 브랜드의 상품 데이터를 미리 선별하여 제공받을 수 있는가?

Agent Assist 검증 관련

요청사항: 일부 Agent Assist 기능 검증 희망

Agent Assist는 독립적인 상담사 애플리케이션이 아닌, 기존에 사용 중인 상담 솔루션(CTI, CRM)에 '통합'되는 방식으로 동작합니다. 따라서 기존 시스템과의 연동 가능성 확인이 가장 중요합니다.

<br>

1.1. 기존 상담 솔루션(CTI/CRM)과의 통합(Integration) 제약

  • 제약사항

    Agent Assist의 핵심 기능인 '실시간 답변 추천(Smart Reply)', '관련 문서 추천(Article Suggestion)'은 상담사가 고객과 대화하는 화면에 실시간으로 표시되어야 합니다. 현재 무신사에서 사용하는 상담 솔루션이 Google의 Agent Assist API 연동을 지원하지 않거나, 커스터마이징이 불가능한 솔루션일 경우 PoC 검증이 어려울 수 있습니다.

  • 구체적인 설명 및 확인 필요사항
    • 현재 솔루션 확인: 현재 상담사들이 사용하는 주된 상담용 애플리케이션은 무엇인가? (예: 자체 개발 CRM, Genesys, Zendesk 등)
    • API 연동 지원 여부: 해당 솔루션이 외부 API를 호출하고 화면에 정보를 표시하는 기능(UI Customization)을 제공하는가?
    • 대안: 만약 직접 연동이 어렵다면, PoC 기간 동안 Google에서 제공하는 '데모용 Agent Assist 화면'을 통해 기능 자체를 검증하는 방식으로 진행 범위를 협의해야 합니다. 이 경우 실제 상담 환경과는 차이가 있을 수 있습니다.

Insights AI 검증 관련 (대화 데이터 분석)

요청사항: 일부 AI Insight 기능 검증 희망

Insights AI는 수많은 고객과의 '대화 데이터(녹취, 채팅로그)'를 분석하여 의미 있는 인사이트를 도출하는 솔루션입니다. 따라서 충분한 양의 고품질 데이터 확보가 관건입니다.

<br>

2.1. 분석을 위한 대화 데이터(Call/Chat Log) 확보

  • 제약사항

    PoC 기간(보통 1~2개월) 동안 수집되는 데이터만으로는 통계적으로 의미 있는 트렌드나 이상 감지를 발견하기 어렵습니다. 솔루션의 가치를 제대로 검증하려면, 과거에 축적된 대화 데이터를 활용하는 것이 필수적입니다.

  • 구체적인 설명 및 확인 필요사항
    • 데이터 제공 가능 여부: 분석을 위해 과거 3~6개월 치의 고객 상담 데이터(음성 녹취 파일, 채팅 로그 텍스트)를 제공받을 수 있는가?
    • 데이터 포맷:
      • 음성: 고객과 상담사의 음성이 분리된 스테레오(Stereo) 녹취 파일(wav, mp3 등)인가? GCS(Google Cloud Storage)에 업로드 가능한 형태인가?
      • 채팅: 발화자, 시간 등이 구분된 정형화된 포맷(JSON, CSV 등)으로 추출이 가능한가?
    • 개인정보보호(PII): 데이터 제공 시, 주민번호, 주소, 카드번호 등 민감한 개인정보를 비식별화(Masking) 처리하여 제공할 수 있는가? 이는 PoC의 가장 중요한 보안 요건입니다.

이번 PoC에서 검증할 Google CES 기능 상세 소개

단순히 챗봇, 상담사 지원, 음성봇 등을 개별적으로 보는 것이 아니라, 세부 기술들이 어떻게 유기적으로 결합하여 하나의 완성된 AI 컨택센터를 구성하는지 검증하는 과정입니다.

Conversational Agents (Dialogflow CX): 대화형 AI 플랫폼

고객과의 모든 상호작용을 처리하는 핵심 엔진입니다. 단순히 질문에 답하는 것을 넘어, 정교한 업무 처리와 지능적인 대화 생성까지 가능한 다층적 구조를 가지고 있습니다.

1. 대화 흐름 제어 계층 (Orchestration Layer)

  • Flows & Pages: 대화의 전체적인 시나리오를 시각적으로 설계하는 '설계도'입니다. 고객의 '반품 요청' 시나리오를 예로 들면, 주문번호 확인 페이지 -> 반품 사유 선택 페이지 -> 최종 접수 확인 페이지 등으로 상태를 나누어, 어떤 경우에도 대화가 길을 잃지 않도록 정교하게 제어합니다.
  • Intents & Entities: 사용자의 '의도(Intent)'와 '핵심 정보(Entity)'를 파악하는 언어 이해의 핵심입니다. "검정색 맨투맨 L사이즈로 교환해주세요"라는 문장에서 '교환'이라는 의도와 함께 '색상: 검정', '상품: 맨투맨', '사이즈: L'이라는 정보를 정확하게 추출해냅니다.

2. 생성형 AI 계층 (Generative Layer)

  • Data Store (Vertex AI Search): AI 상담원의 '지식 두뇌' 역할을 합니다.
    • 기능: 무신사의 방대한 FAQ 웹페이지, 상품 상세페이지, 정책 문서(PDF, TXT 등)를 그대로 학습하여, 별도의 대화 흐름을 만들지 않아도 관련 질문에 대해 Gemini 모델이 문서 내용을 근거로 요약형/답변형 대화를 실시간으로 생성해냅니다.
    • PoC 검증 포인트 (Phase 1): 복잡한 반품/교환 정책 문의에 대해 얼마나 정확하고 할루시네이션 없이 답변을 생성하는지를 검증합니다.
  • Generators: AI 상담원의 '창의적 문장 생성' 능력입니다.
    • 기능: "배송 지연에 대해 고객에게 정중하게 사과하고, 현재 배송 상태와 예상 도착일을 포함하여 안내 이메일 초안을 작성해줘" 와 같이, 정해진 답변이 아닌 특정 목적을 가진 새로운 문장을 동적으로 생성할 수 있습니다.
    • PoC 검증 포인트: PoC의 핵심은 아니지만, 특정 상황(예: 불만 고객 응대)이나 특정 인텐트에서 얼마나 인간적인 뉘앙스의 답변을 만드는지 추가 검증이 가능합니다.

3. 시스템 연동 계층 (Integration Layer)

  • Tools (OpenAPI): AI 상담원이 외부 시스템과 통신하는 '표준화된 연결 통로'입니다.
    • 기능: 무신사의 주문 시스템 API 명세(OpenAPI Spec)를 등록하면, AI가 스스로 "어떤 정보를 보내야 주문 조회가 가능한지" 등을 학습합니다. 이를 통해 재고 조회, 배송 상태 확인 등 실시간 정보 조회가 가능해집니다.
    • PoC 검증 포인트 (Phase 2): 주문 조회 API와 연동하여 실시간 배송 상태를 정확히 가져오는지 검증합니다.
  • Cloud Run: AI 에이전트의 서버리스 실행 환경입니다.
    • 기능: API가 없는 경우에도, Cloud Run에 작성된 코드(Python 등)를 통해 DB에 직접 데이터를 읽거나 쓸 수 있습니다. 예를 들어 '반품 접수' 요청이 오면, Cloud Run 코드가 직접 무신사의 Target DB에 반품 정보를 Insert/Update 합니다.
    • PoC 검증 포인트 (Phase 3): 재고 확인 후 교환 접수 등, 실제 DB의 데이터를 변경하는 복합 트랜잭션을 안정적으로 처리하는지 검증합니다.
  • Agentspace (Optional): 기업용 'AI 비서 포털'입니다.
    • 기능: 단순 챗봇을 넘어, 여러 AI 에이전트와 Tool들을 하나의 통합된 인터페이스에서 사용할 수 있는 기업용 어시스턴트 앱입니다.
    • PoC 검증 포인트: 이번 PoC의 직접적인 범위는 아니지만, AgentSpace는 여러 전문봇(예약봇, 환불/교환 봇, 고객불만문의 봇 등)들을 한 곳에 모아주는 허브 역할을 합니다.
      • Conversational Agent의 각 실무에 맞는 봇을 콜하는가?
        • 사용자가 ‘A 부터 B까지 C를 목적으로 D부터 E까지 기간동안의 제품을 교환하고싶어’ 와 같이 복잡한 문장으로 요청했을 때, 이름, 구매일시, 재고확인, 불만사항 등 모든 정보를 정확하게 추출해 내는지 검증합니다.
      • Cloud Run 의 실행 능력이 있는가?
        • Conversational Agent가 호출했을 때, Cloud Run 함수가 JSON 형태의 데이터를 정상적으로 수신하는지 검증합니다.
        • 수신한 데이터를 가지고 Target DB 테이블에 새로운 행을 정확하게 insert 하는지 검증합니다.
      • 사용자가 AgentSpace에 요청했을 때, 요청의 의도를 파악하고 정확한 전문 에이전트에게 작업을 위임하는가?

Agent Assist: 상담사 AI 코파일럿

  • 기술 상세: 사람 상담사와 고객의 대화(음성/채팅)를 실시간으로 텍스트로 변환하고, 이 텍스트의 맥락을 분석하여 동작합니다. AI는 이 분석 결과를 바탕으로 상담사의 CRM 화면에 연관성이 가장 높은 지식(Data Store 연동)이나 추천 답변(Smart Reply)을 제시합니다. 이는 Conversational Agents와 동일한 NLU 기술 및 지식 기반을 공유하기에 일관된 정보 제공이 가능합니다.

Insights AI: AI 데이터 분석가

  • 기술 상세: 녹취 파일이나 채팅 로그가 저장소(Google Cloud Storage)에 업로드되면, AI가 자동으로 화자를 분리(Diarization)하고 전체 대화를 텍스트로 변환합니다. 이 변환된 텍스트 전체에 대해 Google의 강력한 NLP 모델이 적용되어, 대화의 핵심 주제, 고객의 감정, 특정 키워드의 출현 빈도 등을 분석하고 시각적인 대시보드로 보여줍니다. PoC에서는 이 과정에서 한국어 분석의 정확도를 집중적으로 검증하게 됩니다.

PoC 킥오프 미팅 사전 논의 사항

1. PoC 목표 및 운영 방안

  • 1-1. PoC 성공 측정 지표(KPI) 정의
    • [논의] PoC 완료 후 성공 여부를 판단할 핵심 비즈니스/기술 지표를 사전에 정의합니다.
      • (예: 단순 문의 자동화 처리율, 답변 정확도, 평균 상담 처리 시간(AHT) 감소율 등)
  • 1-2. PoC 기술 지원 담당자 지정
    • [제안] PoC 기간 동안 API, 데이터, 내부 시스템 관련 문의에 대응해주실 무신사 측 기술 담당자(Point of Contact)를 지정 요청 드립니다.

2. AI 모델 성능 및 한국어 지원 검증

  • 2-1. 답변 품질(Hallucination) 및 자연어 처리(NLP) 검증 방안
    • [논의] AI 답변의 신뢰도(Hallucination) 및 한국어 이해력(복잡한 문장, 신조어 등)을 검증할 테스트 케이스와 평가 기준을 확정합니다.
  • 2-2. 음성(사투리) 인식률 검증 방안
    • [논의] 음성봇의 사투리, 억양 등 인식률을 검증할 테스트 스크립트의 범위와 내용을 논의합니다.
  • 2-3. 학습 데이터 요구사항 확인
    • [확인] Data Store의 최적 성능을 위한 데이터 분량, 파일 형식, 용량/글자 수 제한 등 제약사항을 확인합니다.

3. Conversational Agents 시나리오 범위 확정

  • 3-1. [Phase 1] 정적 정보 안내
    • [확인] AI가 학습할 FAQ/정책 안내 페이지 URL 목록 또는 문서 파일을 제공받습니다.
    • [논의] 해당 데이터의 정제 수준 및 크롤링 제약 여부를 논의합니다.
  • 3-2. [Phase 2] 실시간 정보 조회
    • [논의] '배송 조회', '누락 문의' 등 실시간 조회 시나리오를 PoC 범위에 포함할지 결정합니다.
    • [확인] 포함 시, 필수적인 Read-Only API의 제공 가능 여부와 테스트 환경을 확인합니다.
  • 3-3. [Phase 3] 복합 트랜잭션 처리
    • [논의] '반품/교환 접수' 등 실제 데이터 쓰기(Write) 시나리오를 PoC에 포함할지 결정합니다.
    • [논의] 포함 시, 트랜잭션 데이터를 기록할 Target DB를 선정하고 Write API 제공 방안을 논의합니다.

4. Agent Assist 연동 방안

  • 4-1. 기존 상담 시스템 연동 가능성 확인
    • [확인] 현재 사용 중인 상담 솔루션(CRM/CTI)을 식별하고, API/UI 연동 지원 여부를 확인합니다.
    • [논의] 직접 연동이 어려울 경우, Google 제공 데모 환경을 통한 기능 검증으로 범위를 축소하는 방안을 논의합니다.

5. Insights AI 데이터 준비

  • 5-1. 분석용 과거 대화 데이터 확보 방안
    • [논의] 분석을 위해 제공 가능한 과거 상담 데이터의 기간과 양을 확정합니다. (3~6개월 분량 권장)
    • [확인] 데이터 포맷(스테레오 음성, 정형화된 채팅 로그) 및 개인정보 비식별화(PII Masking) 처리 방안을 최종 확정합니다.
