Audiohook과 외부 서버 연동

🧩 전체 흐름 요약

1. AudioHook 구성

Genesys Cloud에서 AudioHook을 설정할 때 입력하는 **"서버 주소 (Domain 주소)"**는 Genesys가 오디오 데이터를 WebSocket으로 실시간 스트리밍할 외부 서버의 엔드포인트입니다.

예시:

https://xxxx-yyy.ngrok-free.app

2. WebSocket 연결 흐름

  • Genesys는 AudioHook 트리거 이벤트가 발생하면 (예: 인바운드 콜 시작), 설정된 도메인에 대해 WebSocket 연결을 시도합니다.
  • 이 도메인 주소는 AudioHook 설정 시 입력한 URL입니다.
  • 예: wss://xxxx.ngrok-free.app → 내부적으로는 Genesys Cloud가 WebSocket handshake 요청을 보내는 것입니다.

3. Signature 검증 및 Session 인증

  • AudioHook이 서버와 WebSocket 연결을 맺을 때 인증 및 서명 검증이 필요한 경우, signature, signature-input, x-api-key 등의 헤더를 포함하여 요청을 보냅니다.
  • 서버는 이 요청을 받아서 HMAC으로 서명 검증을 수행하고, 인증이 성공해야 실제 오디오 세션이 열립니다.

4. 연동 성공 시

  • 서버는 "open" 메시지를 수신하고 "opened" 메시지를 응답함으로써 세션이 시작됩니다.
  • 이후 "audio" 바이너리 메시지, "ping", "close" 등의 메시지가 오가며 세션을 유지하게 됩니다.

🔐 도메인이 중요한 이유

  • WebSocket은 정확한 Hostname (도메인) 기준으로 서명 및 인증을 처리합니다.
  • Genesys는 AudioHook에서 요청할 때 @authority = 도메인 을 포함하여 서명하기 때문에, 서버에서도 정확히 일치하는 값으로 HMAC을 계산해야 인증에 성공합니다.

🔎 예시

sig1=("@request-target" "audiohook-session-id" ... "@authority");
@authority = 9737-211-44-198-26.ngrok-free.app

따라서:

  • ngrok을 쓸 경우 주소가 바뀔 때마다 서명도 바뀌므로 매번 정확한 도메인으로 HMAC 검증이 필요합니다.
