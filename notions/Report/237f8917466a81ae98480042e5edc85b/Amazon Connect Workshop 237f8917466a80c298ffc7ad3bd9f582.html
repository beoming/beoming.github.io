1. 핵심 개념 및 아키텍처

Q in Connect 도메인

  • 하나의 Connect 인스턴스 = 하나의 도메인 (1:1 관계)
  • 하나의 도메인은 AI Agent 구성을 통해 여러 지식베이스 지원 가능
  • 도메인 간 데이터 공유 불가 (독립적 운영)
  • 외부 애플리케이션 통합은 도메인 수준에서 수행
  • 도메인 전환 가능 (Connect 인스턴스를 다른 도메인으로 변경 가능)

암호화 및 보안

  • 기본: AWS 소유키로 암호화
  • 사용자 정의: 고객 관리형 KMS 키 사용 가능
    • Q in Connect 도메인용 키 (권장사항 발췌문 암호화)
    • 외부 소스 콘텐츠용 키 (S3, Salesforce, ServiceNow 등)
  • 검색 인덱스: 항상 AWS 소유키로 암호화

Amazon Q in Connect: 콘텐츠 세분화

1. Amazon Q in Connect 콘텐츠 세분화 개요

Amazon Q in Connect의 콘텐츠 세분화는 통화 중 수집된 데이터를 활용하여 특정 통화와 가장 관련성이 높은 지식 베이스 문서를 정확하게 식별하고 제공하는 기능이다.

핵심 원리:

  • 태깅(Tagging): 지식 베이스 콘텐츠에 특정 기준(예: 조직 구조, 제품, 서비스 유형)에 따라 태그를 지정한다.
  • 필터링(Filtering): Amazon Connect 연락처 흐름에서 Lambda 함수를 통해 통화 컨텍스트에 맞는 태그를 사용하여 Amazon Q in Connect 세션의 콘텐츠를 동적으로 필터링한다.

2. 콘텐츠 태그 구성 방법

콘텐츠에 태그를 지정하는 과정은 지식 베이스의 소스에 따라 달라진다. 본 워크샵에서는 유틸리티를 통한 S3, Salesforce, ServiceNow 태깅 방법을 설명하며, 다른 지식 베이스는 TagResource API를 사용해야 한다.

2.1. 사전 준비

  • 로컬 환경 설정:
    • Node.js 설치 (유틸리티 실행에 필요).
    • AWS CLI 설치 및 S3 버킷이 위치한 올바른 AWS 계정으로 프로필 구성 (세션 토큰, 액세스 키, 키 ID 필수).

2.2. 데이터 소스별 구성 파일 (config/*.json)

각 데이터 소스에 맞는 JSON 파일을 편집하여 태깅 규칙을 정의한다.

  • S3 (config/s3.json):
    • 규칙: S3 버킷 이름(s3BucketName)과 폴더 이름(folderName)을 기반으로 tags (키-값 쌍)를 정의한다.
    • 예시: 특정 S3 폴더에 있는 문서를 특정 부서 또는 프로젝트에 매핑한다.JSON
      {
	"taggingRules": [
		{
			"s3BucketName": "sb-qicbucket",       // S3에 있는 내 버킷 이름
			"folderName": "QiC Content/QiC Content/auto",  // 태깅 적용할 폴더 경로
			"tags": {
				"bu": "auto",   // 폴더명
				"org": "finance"  // 태그명
			}
		}
	]
}
  • Salesforce (config/salesforce.json):
    • 규칙: keyName (Salesforce 필드 이름)과 keyValue (필드 값)를 기반으로 tags를 정의한다.
    • 예시: Salesforce 레코드의 특정 필드 값(ProgramId__c: Payment)에 따라 문서를 태그한다.JSON
      {
	"taggingRules": [
		{
			"s3BucketName": "sb-qicbucket",       // S3에 있는 내 버킷 이름
			"folderName": "QiC Content/QiC Content/payment",  // 태깅 적용할 폴더 경로
			"tags": {
				"bu": "payment",   // 폴더명
				"org": "finance"  // 태그명
			}
		}
	]
}
  • ServiceNow (config/servicenow.json):
    • 규칙: keyName (ServiceNow 필드 이름)과 keyValue (필드 값)를 기반으로 tags를 정의한다. (주로 article_id 사용).
    • 예시: ServiceNow 문서 ID(article_id)에 따라 특정 부서나 프로그램에 문서를 태그한다.JSON
      {
  "taggingRules": [
    {
      "keyName": "article_id",
      "keyValue": "d7dc75b5474321009db4b5b08b9a7150",
      "tags": {
        "departmentId": "IT OS",
        "department": "OS"
      }
    }
  ]
}

2.3. 유틸리티 실행

  1. 1-content-tagging 디렉토리에서 npm install 및 npm run start를 실행한다.
  1. 프롬프트에 따라 AWS 리전, AWS 프로필, 지식 베이스 선택, 데이터 소스 선택 (S3, Salesforce, ServiceNow) 정보를 입력한다.
  1. 유틸리티는 config.json에 정의된 규칙에 따라 콘텐츠에 태그를 지정하고 KnowledgeBaseID를 출력한다.

2.4. 태그 지정 확인

  • aws qconnect list-contents --knowledge-base-id YOURKnowledgeBaseID --profile YOURProfile 명령어로 태그가 올바르게 적용되었는지 확인한다.

3. Amazon Connect에서 콘텐츠 태그 적용 (실시간 지원)

태그가 지정된 콘텐츠를 Amazon Connect 인스턴스에 연동하여 실시간으로 콘텐츠를 필터링한다.

3.1. IAM 역할 생성

  • 역할 이름: QiCContentRetrievalRole
  • 신뢰 엔터티 유형: AWS 서비스 (Lambda)
  • 인라인 정책:JSON
    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "connect:DescribeContact",
        "wisdom:UpdateSession"
      ],
      "Resource": "*"
    }
  ]
}
    • 이 정책은 Lambda 함수가 Connect의 연락처 정보를 가져오고 Amazon Q in Connect 세션을 업데이트할 수 있도록 허용한다.
  • 신뢰 정책: Lambda가 역할을 맡을 수 있도록 Service: lambda.amazonaws.com에 sts:AssumeRole 권한을 부여한다.

3.2. AWS Lambda 함수 생성

  • 함수 이름: QiCContentRetrievalFunction
  • 런타임: Python 3.12
  • 실행 역할: QiCContentRetrievalRole (단계 3.1에서 생성)
  • 코드: 제공된 Python 코드를 lambda_function.py에 붙여넣는다.
    • 이 코드는 Amazon Connect 이벤트에서 ContactData와 tagFilter를 추출한다.
    • connect_client.describe_contact를 사용하여 현재 연락처의 Amazon Q in Connect 세션 정보 (Session ARN)를 가져온다.
    • qconnect_client.update_session (또는 Boto3 버전이 낮은 경우 HTTP 요청)을 사용하여 tagFilter 값으로 Amazon Q in Connect 세션을 업데이트한다.
    • 핵심 로직: update_session API를 호출하여 세션에 tagFilter를 적용함으로써, 해당 세션에서 제공되는 지식 콘텐츠를 특정 태그로 필터링한다.
  • 타임아웃: 8초로 설정한다.

3.3. Amazon Connect 인스턴스에 Lambda 함수 추가

  • Amazon Connect 콘솔에서 해당 인스턴스를 선택한다 → "Flow" → "AWS Lambda" 섹션으로 이동한다.
  • 드롭다운에서 QiCContentRetrievalFunction을 선택 후 "Lambda 함수 추가"를 선택한다.

3.4. Amazon Connect 연락처 흐름 정의

  • 샘플 흐름 가져오기: QiC Content Segmentation Flow.json을 다운로드 후 Amazon Connect 인스턴스에 가져온다.
  • 흐름 블록 편집:
    • Amazon Q Connect 블록: Connect 인스턴스와 연결된 도메인 ARN을 업데이트한다.
    • Invoke AWS Lambda Function 블록:
      • 단계 3.2에서 생성한 QiCContentRetrievalFunction을 선택한다.
      • tagFilter 매개변수를 설정하여 콘텐츠 태깅 섹션에서 정의한 태그를 기반으로 필터링 규칙을 지정한다.JSON
        {
  "tagCondition": {
    "key":"bu",
    "value":"credit"
  }
}

        (UpdateSession API Boto3 문서를 참조하여 더 복잡한 구문 활용이 가능하다.)

    • Set working queue 블록: 연락처를 라우팅할 대기열을 선택한다.
  • 저장 및 게시: 모든 변경 사항을 저장하고 흐름을 게시한다.

3.5. Amazon Q in Connect 콘텐츠 세분화 테스트

  • 테스트 준비: 음성 또는 채팅 테스트 환경을 설정한다 (전화번호 매핑, QConnect 도메인 설정 등).
  • 테스트 실행:
    1. Amazon Connect "채팅 테스트" 페이지에서 새 흐름을 선택한다.
    1. 새 탭에서 Amazon Connect 상담원 워크스페이스를 연다.
    1. 샘플 채팅을 시작 후 상담원 워크스페이스에서 수락한다.
    1. 고객 측에서 질문을 보내고, 상담원 워크스페이스의 Amazon Q에서 제공되는 지식 콘텐츠가 tagFilter에 따라 올바르게 필터링되는지 확인한다.

4. 문제 해결 및 FAQ (콘텐츠 세분화 관련)

4.1. 문제 해결 (콘텐츠 필터링 오류 시)

  • 지식 문서 태그 확인: ListContents API로 문서에 적절한 태그가 있는지 확인한다.
  • Lambda 함수 호출 확인: 연락처 흐름 로그를 통해 Lambda 함수가 정상적으로 호출되는지, 함수 로깅을 통해 오류 발생 여부를 확인한다.
  • Lambda 함수 update_session 호출 확인: Lambda 로그를 통해 update_session에 올바른 태그가 전달되었는지, Lambda가 QConnect에 접근 권한이 있는지 확인한다.

4.2. 자주 묻는 질문 (FAQ)

  • 흐름 블록 태그가 추천/검색 결과에 미치는 영향: 기본적으로 모든 콘텐츠는 모든 상담원에게 제공되나, 태그를 통해 특정 상담원 그룹에 대해 콘텐츠를 타겟팅(필터링)할 수 있다. 연락처 흐름 내 Amazon Q 블록의 태그는 현재 문의에 대한 콘텐츠를 필터링한다 (예: 특정 브랜드 문의 시 해당 브랜드 콘텐츠만 참조).
  • 연락처 전환 시 동작: 연락처가 새로운 Amazon Q 흐름 블록을 통과하면, 새로운 흐름 블록의 태그가 이전 흐름 블록에서 적용된 태그를 덮어쓴다.
  • 기존 Amazon Q 고객 영향: 기존 고객은 별도의 변경이나 업데이트가 필요 없다. Amazon Q 블록에 태그가 적용된 경우에만 콘텐츠가 세분화된다. 태그가 없으면 기존처럼 모든 콘텐츠가 반환된다.
  • 상담원에게 미치는 영향: 콘텐츠가 연락 유형에 더 부합하게 필터링되어, 상담원은 더 유용한 추천사항과 검색 결과를 얻을 수 있다.
  • 흐름 변경 사항: Amazon Q 흐름 블록 설정 시, 어시스턴트 외에 태그를 선택할 수 있는 옵션이 추가되었다.

5. 새롭게 바뀐 AWS Connect 기능 요약 (Amazon Q in Connect 관점 - 콘텐츠 세분화)

동적 콘텐츠 세분화 기능 (wisdom:UpdateSession API 활용):

  • 핵심: Amazon Q in Connect 세션에 tagFilter를 동적으로 적용하여 콘텐츠를 필터링할 수 있게 된 점이다. 이는 단순히 지식 베이스의 모든 콘텐츠를 제공하는 것을 넘어, 현재 통화 컨텍스트(예: 고객의 문의 유형, 상담원의 소속 부서)에 따라 가장 관련성이 높은 정보만을 선별적으로 제공한다.
  • 이 기능은 wisdom:UpdateSession API를 통해 구현되며, 이전에는 없었거나 강조되지 않았던 세션 단위의 콘텐츠 필터링 기능이 명시적으로 추가된 것으로 보인다.

Amazon Connect 연락처 흐름 내 Lambda 연동 및 tagFilter 매개변수:

  • Amazon Connect 연락처 흐름에서 Lambda 함수를 호출하여 DescribeContact로 현재 연락처의 Q Connect 세션 정보를 가져오고, 이를 기반으로 UpdateSession API를 호출하는 구조가 명확히 제시되었다.
  • 특히, Invoke AWS Lambda Function 블록에서 tagFilter라는 새로운 매개변수를 직접 설정할 수 있게 된 것은 관리자가 코드 수정 없이도 특정 태그 기반의 콘텐츠 필터링 로직을 쉽게 적용할 수 있도록 지원하는 중요한 개선 사항이다.

다양한 데이터 소스에 대한 통합 태깅 유틸리티 제공:

  • Amazon S3뿐만 아니라 Salesforce, ServiceNow와 같은 외부 지식 베이스 소스에 대해서도 공통된 유틸리티를 통해 콘텐츠 태깅을 지원함으로써, 엔터프라이즈 환경에서 다양한 시스템에 분산된 지식 콘텐츠를 통합적으로 관리하고 세분화할 수 있는 기반을 제공한다. 이는 Amazon Q in Connect가 더 넓은 범위의 지식 소스를 활용할 수 있음을 의미한다.

상담원 경험 및 효율성 향상:

  • 가장 큰 이점은 상담원이 "더욱 유용한 추천사항과 검색 결과"를 얻을 수 있다는 점이다. 불필요하거나 관련 없는 정보가 제거되면서 상담원의 정보 탐색 시간이 줄어들고, 고객 문의에 대한 정확한 답변을 빠르게 찾을 수 있게 되어 상담 효율성과 고객 만족도가 크게 향상될 것으로 예상된다. 이는 Amazon Connect와 Amazon Q의 통합이 단순히 정보 제공을 넘어 실질적인 상담 업무 효율화에 초점을 맞추고 있음을 보여준다.

상담 어시스턴스를 위한 Amazon Connect Q in Connect

Amazon Q in Connect 종합 분석 및 정리: 커스터마이징 중심

1. 상담 어시스턴스를 위한 Amazon Q in Connect 커스터마이징 개요

Amazon Connect의 Amazon Q는 상담원 어시스트 및 고객 셀프 서비스 시나리오에서 고도로 맞춤화된 기능을 제공하여 고객 서비스의 질을 향상시킨다. 커스터마이징은 단순히 지식 베이스 검색을 넘어, 특정 비즈니스 요구사항과 고객 컨텍스트에 따라 AI의 동작과 응답을 정교하게 제어하는 것을 목표로 한다.

주요 커스터마이징 기회:

  • 맥락 정보 포함: 고객 충성도 상태, 구매 이력 등 실시간 고객 데이터를 응답에 반영하여 개인화된 경험을 제공한다.
  • 브랜드 톤 및 형식 조정: 기업의 브랜드 가이드라인에 맞춰 AI 응답의 톤과 형식을 일관되게 유지한다.
  • 지식 베이스 검색 동작 최적화: 특정 비즈니스 로직에 따라 지식 베이스 검색 방식과 결과의 관련성을 커스터마이징한다.
  • 다국어 지원: 글로벌 운영을 위한 언어별 맞춤 설정을 지원한다.
  • 책임감 있는 AI 사용: AI 가드레일을 구현하여 부적절하거나 편향된 응답을 방지하고 기업 표준을 준수한다.

이러한 커스터마이징은 주로 AI 프롬프트(AI Prompts)와 AI 에이전트(AI Agents)의 생성을 통해 이루어진다.

사용 사례 예시: 개인화된 뱅킹 지원 (AnyCompany)

AnyCompany의 뱅킹 부서는 다음과 같은 문제에 직면해 있었다:

  • 상담원이 여러 시스템에서 고객 정보를 수동으로 조회해야 한다.
  • 고객 프로필 해석 및 적절한 상품 추천에 많은 시간이 소요된다.
  • 정보 과부하로 인해 관련 상품 제안 기회를 놓치는 경우가 많다.
  • 다양한 고객 세그먼트 간 일관된 응답 품질 유지에 어려움이 있다.

Amazon Q의 기본 지식 베이스 검색 기능은 유용하지만, 개인화된 응답을 위해서는 고객 데이터(신용 점수, 신용카드 상품, 지출 프로필 등) 통합이 필수적이다. AI 프롬프트와 AI 에이전트를 통해 AnyCompany는 다음을 달성할 수 있었다:

  • 맥락 인식 응답: 고객 데이터를 통합하여 맞춤형 응답을 제공하는 프롬프트를 생성한다.
  • 스마트 상품 추천: 고객 데이터를 분석하여 관련 뱅킹 상품을 제안하도록 Amazon Q in Connect를 활성화한다.

    이러한 솔루션은 상담원의 정보 검색 및 해석 시간을 줄여 더 개인화된 서비스를 제공하는 데 기여한다.

2. AI 프롬프트 (AI Prompts) 상세 이해

AI 프롬프트는 생성형 AI(예: Amazon Q in Connect)에게 특정 작업을 수행하도록 요청하는 자연어 텍스트 지침이다. 이는 AI가 원하는 고품질의 관련성 있는 출력을 생성하도록 안내하는 프롬프트 엔지니어링의 핵심 요소이다.

AI 프롬프트의 역할:

  • Q in Connect가 고객 상호작용을 처리하고 응답하는 방식을 안내한다.
  • 특정 응답 형식 정의, 맥락 정보 포함, 커스텀 도구 및 작업 구성, 비즈니스 로직 구현을 돕는다.

AI 프롬프트의 유형: Amazon Q in Connect에서 수행할 활동에 따라 다양한 유형이 있다.

  • 답변 생성 (Answer Generation): 지식 베이스 발췌문을 활용하여 쿼리에 대한 해결책을 생성한다. (예: MyFirstAnswerGenerationPrompt)
  • 의도 레이블 생성 (Intent Labeling Generation): 고객 서비스 상호작용에 대한 의도를 식별하고 레이블을 생성한다.
  • 쿼리 재구성 (Query Reformulation): 대화 컨텍스트와 고객 데이터를 기반으로 관련 지식 베이스 발췌문을 검색하기 위한 최적화된 쿼리를 구성한다. (예: MyRealtimeQueryReformulationPrompt)
  • 셀프 서비스 전처리 (Self Service PreProcessing): 셀프 서비스 시나리오에서 대화를 평가하고 응답 생성을 위한 도구를 선택한다.
  • 셀프 서비스 답변 생성 (Self Service Answer Generation): 셀프 서비스 환경에서 지식 베이스 발췌문을 활용하여 답변을 생성한다.

커스텀 AI 프롬프트 생성 예시 (UI 기반):

  1. 환경 설정: Amazon Connect 관리자 콘솔에 접근하고, 보안 프로파일에서 AI 에이전트와 AI 프롬프트에 대한 view, edit, create 권한을 확보한다. Amazon Q 섹션의 Overview 페이지에서 시작한다.
  1. 프롬프트 생성: AI prompts 화면으로 이동하여 Create AI prompt를 선택한다.
  1. 세부 정보 입력: 프롬프트 이름과 AI 프롬프트 유형을 선택하고 Description을 입력한 후 Create를 클릭한다.
  1. 프롬프트 내용 작성/대체: 프롬프트 빌더 화면에서 시스템 프롬프트를 사용자 정의 내용으로 대체한다. (예: 쿼리 재구성 프롬프트에 {{$.transcript}}, {{$.Custom.productId}}, {{$.Custom.creditScore}}, {{$.Custom.spendingProfile}}과 같은 사용자 정의 변수를 포함하여 고객 컨텍스트를 활용하도록 지시).

3. AI 에이전트 (AI Agents) 상세 이해

AI 에이전트는 AI 프롬프트 버전, 지식 베이스 설정, 언어 기본 설정, 콘텐츠 필터 등을 결합한 포괄적인 구성이다. 인간이 목표를 설정하면 AI 에이전트가 이를 달성하기 위한 최적의 작업을 독립적으로 선택한다.

AI 에이전트의 역할:

  • 실시간 답변 추천: 고객 대화를 모니터링하고 관련 응답을 자동으로 제안한다.
  • 수동 검색: 상담원이 직접 질문을 입력할 때 개인화를 유지하며 특정 정보 검색을 가능하게 한다.
  • 일관된 경험 보장: 두 유형의 에이전트 모두 동일한 고객 컨텍스트(신용 점수, 제품 이력, 지출 프로필)를 사용하여 일관된 추천을 제공한다.

커스텀 AI 에이전트 생성 예시 (UI 기반):

  1. 생성 시작: AI agents 화면으로 이동하여 Create AI agent를 선택한다.
  1. 세부 정보 입력: AI 에이전트 이름, AI 에이전트 유형 (예: Answer recommendation, Manual search), Description을 입력하고 Create를 클릭한다.
  1. 프롬프트 연결: 생성된 AI 에이전트의 구성 화면에서 Add prompt를 선택하여 이전에 생성한 관련 AI 프롬프트(예: 답변 생성 프롬프트, 쿼리 재구성 프롬프트)를 연결한다.
  1. 게시 (Publish): 구성된 AI 에이전트를 게시하여 사용 준비를 완료한다.

4. Amazon Q in Connect 어시스턴트 업데이트 및 활성화

생성된 커스텀 AI 에이전트를 Amazon Q in Connect 어시스턴트에 적용하여 실시간 지원 및 수동 검색 기능에 맞춤화된 동작을 부여한다.

배포 방법:

  1. Amazon Connect 콘솔(UI) 배포:
    • AI agents 화면의 Defaults 섹션에서 Edit을 선택한다.
    • answer recommendation 및 manual search 유형에 대해 생성한 커스텀 AI 에이전트를 선택하고 Save한다.
    • 장점: 가장 간단하고 시각적이며, 코딩 지식이 필요 없다. 빠른 수정에 적합하다.
    • 제한: 자동화 기능 및 대량 작업에 제약이 있다.
  1. AWS CLI 배포:
    • update-assistant-ai-agent 명령어를 사용하여 특정 AI 에이전트 유형(ANSWER_RECOMMENDATION 또는 MANUAL_SEARCH)에 대한 AI 에이전트 ID와 버전 번호를 어시스턴트에 연결한다.
    • 장점: 스크립트화 및 자동화가 가능하며, 세밀한 제어가 가능하다.
    • 예시 명령어:Bash

      aws qconnect update-assistant-ai-agent \
      --assistant-id <your-assistant-id> \
      --ai-agent-type ANSWER_RECOMMENDATION \
      --configuration '{"aiAgentId": "<your-answer-recommendation-agent-id>:<your-answer-recommendation-agent-version-no>"}'

  1. CloudFormation 배포:
    • 장점: Infrastructure as Code(IaC)를 사용하여 모든 구성 요소(AI 프롬프트, AI 에이전트, Lambda 함수, IAM 역할 등)를 단일 템플릿으로 자동 배포한다. 빠르고 재현 가능한 결과를 제공하며, 일관된 환경 구축 및 DevOps 실무에 적합하다.
    • 단점: CloudFormation 템플릿에 대한 이해가 필요하다.

5. 신기능 및 참신한 기능에 대한 기술 정리 (커스터마이징 관점)

제공된 정보를 바탕으로 Amazon Q in Connect의 커스터마이징 관련하여 새롭고 참신하다고 평가할 수 있는 기술 기능들은 다음과 같다.

  1. 생성형 AI의 '프롬프트 엔지니어링' UI/API 통합:
    • 기술적 의미: Amazon Q in Connect는 단순한 키워드 검색을 넘어, 생성형 AI 모델에게 직접적인 '지침'을 제공하는 프롬프트 엔지니어링 개념을 서비스 내부에 깊이 통합한다. 특히, {{$.transcript}}, {{$.Custom.productId}}와 같은 변수를 프롬프트 내에서 직접 사용하여 실시간 대화 내용 및 외부 고객 데이터를 AI에게 컨텍스트로 제공할 수 있게 한 점이 매우 참신하다.
    • 영향: 이는 개발자나 데이터 과학자가 아닌 비즈니스 사용자(컨택 센터 관리자)도 AI의 동작을 매우 정교하게 제어하고 개인화할 수 있도록 지원하며, AI가 단순히 정보를 '찾아주는' 것을 넘어 '해석하고 재구성하여 추천'하는 수준으로 진화할 수 있는 기반을 마련한다.
  1. AI 에이전트의 '모듈화된 AI 행동 단위' 개념:
    • 기술적 의미: AI 프롬프트들을 단순히 묶는 것을 넘어, Answer Recommendation이나 Manual Search와 같이 특정 목적과 시나리오에 최적화된 'AI 행동 단위'인 AI 에이전트를 정의할 수 있게 한 점이다. 각 에이전트는 고유한 프롬프트 세트와 구성을 가질 수 있다.
    • 영향: 이는 컨택 센터 내에서 다양한 상담 시나리오(예: 기술 지원, 판매, 청구)에 맞춰 AI 어시스턴트의 행동을 맞춤화하고 쉽게 전환할 수 있도록 한다. 비즈니스 변화에 따라 AI의 동작을 유연하게 업데이트하고 관리할 수 있는 아키텍처적 유연성을 제공한다.
  1. Customer Context Variable 활용을 통한 심층 개인화:
    • 기술적 의미: {{$.Custom.변수명}} 형태로 고객의 신용 점수, 현재 상품, 지출 프로필 등 다양한 CRM 또는 외부 시스템의 데이터를 AI 프롬프트에 직접 주입하여 AI가 이를 기반으로 답변을 생성하거나 쿼리를 재구성하게 하는 기능이 강조되었다.
    • 영향: 이는 Amazon Q in Connect가 일반적인 정보 제공을 넘어선 '초개인화된' 고객 서비스 경험을 제공할 수 있게 한다. 상담원은 고객의 고유한 상황에 맞는 추천과 해결책을 즉시 받을 수 있어, 고객 만족도 향상과 함께 업셀링/크로스셀링 기회까지 포착할 수 있게 된다.
  1. CLI 및 IaC(CloudFormation)를 통한 AI 커스터마이징의 자동화:
    • 기술적 의미: UI를 통한 간편한 설정 외에도, AWS CLI 명령어나 CloudFormation 템플릿을 통해 AI 프롬프트와 에이전트 생성을 포함한 전체 커스터마이징 과정을 코드로 관리하고 자동화할 수 있게 한 점이다.
    • 영향: 이는 대규모 조직이나 DevOps 문화를 가진 기업에서 AI 솔루션의 배포, 버전 관리, 환경 일관성을 효율적으로 유지할 수 있도록 돕는다. 테스트 환경에서 검증된 AI 설정을 프로덕션 환경으로 쉽게 확장하고 반복적으로 배포할 수 있는 기반을 제공하여, AI 도입의 속도와 안정성을 높인다.
