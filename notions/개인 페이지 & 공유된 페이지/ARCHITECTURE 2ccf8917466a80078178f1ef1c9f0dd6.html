전체 아키텍처 문서

1. 시스템 개요

이 애플리케이션은 Spring Boot 기반의 대화형 에이전트 시스템으로, 고객과 상담사 간의 실시간 채팅을 지원하며 Dialogflow CX를 통한 AI 챗봇 기능을 제공합니다.

주요 구성 요소

  • Chat Server (chat profile): 고객용 채팅 서버
  • Monitor Server (monitor profile): 상담사용 모니터링 서버
  • Dialogflow CX: Google의 대화형 AI 플랫폼
  • Google Cloud Pub/Sub: 서버 간 비동기 메시지 전달
  • H2 In-Memory Database: 각 서버별 독립적인 인메모리 데이터베이스

2. 서버 구조

2.1 Chat Server (chat profile)

역할: 고객이 채팅을 하는 메인 서버

주요 기능:
- 고객 메시지 수신 및 처리
- Dialogflow CX API 호출
- 고객 챗 UI 제공 (/static/index.html)
- 상담사 메시지 수신 및 처리

배포 정보:
- 서비스명: customer-chat-server
- 프로파일: chat
- 포트: 8080
- 데이터베이스: H2 In-Memory (독립적)

주요 엔드포인트:
- POST /api/send-message: 고객 메시지 처리
- POST /api/agent-message: 상담사 메시지 처리 (Monitor Server에서 호출)
- GET /api/sessions/{sessionId}/messages: 세션 메시지 조회 (고객 UI 폴링용)

2.2 Monitor Server (monitor profile)

역할: 상담사가 고객 대화를 모니터링하고 개입하는 서버

주요 기능:
- 상담사 모니터링 대시보드 제공 (/static/agent-monitor.html)
- 상담사 메시지 입력 및 Chat Server로 전달
- Pub/Sub Push Subscription을 통한 실시간 이벤트 수신
- WebSocket을 통한 실시간 UI 업데이트

배포 정보:
- 서비스명: agent-monitor-server
- 프로파일: monitor
- 포트: 8080
- 데이터베이스: H2 In-Memory (독립적, Chat Server와 분리)

주요 엔드포인트:
- GET /api/agent/sessions: 모든 세션 목록 조회
- GET /api/agent/sessions/{sessionId}: 특정 세션 정보 조회
- GET /api/agent/sessions/{sessionId}/messages: 세션 메시지 조회
- POST /api/agent/sessions/{sessionId}/respond: 상담사 응답 전송
- POST /pubsub/push: Pub/Sub Push Subscription 엔드포인트
- WebSocket /ws/agent: 실시간 이벤트 브로드캐스트

3. 데이터베이스 구조

3.1 데이터베이스 분리

Chat Server DB와 Monitor Server DB는 완전히 독립적인 H2 In-Memory 데이터베이스입니다.

  • Chat Server DB: 고객 메시지, 상담사 메시지, AI 응답 저장
  • Monitor Server DB: Pub/Sub을 통해 동기화된 메시지 저장

3.2 엔티티 구조

ChatSession

- sessionId (PK): 세션 고유 ID
- language: 언어 설정
- firstRequestAt: 첫 요청 시간
- createdAt: 생성 시간
- updatedAt: 업데이트 시간
- responses: ChatResponse 리스트 (OneToMany)

ChatResponse

- id (PK): 자동 증가 ID
- responseId: 응답 고유 ID (중복 방지용)
- session: ChatSession (ManyToOne)
- userMessage: 고객 발화 (상담사 메시지인 경우 null)
- reply: AI 응답 또는 상담사 메시지
- status: "SUCCESS", "ERROR", "AGENT_RESPONSE" 등
- intentDisplayName: 인텐트 이름
- intentConfidence: 인텐트 신뢰도
- intentMatchType: 인텐트 매칭 타입
- intentsJson: 인텐트 리스트 (JSON, session_summary에서 추출)
- entitiesJson: 엔티티 정보 (JSON)
- createdAt: 생성 시간
- images: ProductImage 리스트
- shownProducts: ShownProduct 리스트

4. Pub/Sub 아키텍처

4.1 Pub/Sub 구성

Topic: chat-events
- 프로젝트: gsneotek-ncc-demo (또는 환경 변수로 설정)
- 생성 위치: infra/cloudrun-deploy-monitor.sh 또는 infra/pubsub-create.sh

Subscription: agent-monitor-sub
- 타입: Push Subscription
- Push Endpoint: https://agent-monitor-server-{region}.run.app/pubsub/push?token={VERIFY_TOKEN}
- 인증: Service Account (monitor-sa@{PROJECT_ID}.iam.gserviceaccount.com)

4.2 Pub/Sub 데이터 흐름

[Chat Server]
  ↓
ChatEvent 발행 (ChatEventPublisher)
  - sender: "user" | "bot" | "agent"
  - conversationId: sessionId
  - messageId: 고유 메시지 ID
  - text: 메시지 내용
  - status: "SUCCESS", "AGENT_RESPONSE" 등
  - intentList: 인텐트 리스트
  ↓
[Pub/Sub Topic: chat-events]
  ↓
[Pub/Sub Push Subscription: agent-monitor-sub]
  ↓
[Monitor Server: /pubsub/push]
  ↓
PubSubPushController.receiveMessage()
  ↓
1. Base64 디코딩
2. ChatEvent 파싱
3. Monitor Server DB에 저장 (saveEventToDatabase)
4. WebSocket 브로드캐스트 (AgentSessionManager.broadcast)

4.3 ChatEvent 모델

ChatEvent {
    conversationId: String      // 세션 ID
    messageId: String          // 고유 메시지 ID
    sender: String             // "user" | "bot" | "agent"
    text: String               // 메시지 내용
    timestamp: String          // ISO 8601 형식
    channel: String            // "web"
    intentDisplayName: String  // 인텐트 이름
    intentConfidence: Double   // 인텐트 신뢰도
    intentMatchType: String    // 인텐트 매칭 타입
    intentList: List<String>   // session_summary에서 추출한 모든 인텐트
    status: String             // "SUCCESS", "ERROR", "AGENT_RESPONSE"
}

4.4 Pub/Sub 발행 시점

Chat Server에서 발행하는 이벤트:

  1. 고객 메시지 이벤트 (sender="user")
    • ChatService.publishUserMessageEvent() 호출
    • 고객이 메시지를 보낼 때
  1. AI 응답 이벤트 (sender="bot")
    • ChatService.publishBotResponseEvent() 호출
    • Dialogflow CX 응답을 받았을 때
    • 상담사 메시지에 대한 AI 응답도 포함
  1. 상담사 메시지 이벤트 (sender="agent")
    • ChatService.publishAgentMessageEvent() 또는 saveAndPublishAgentMessage() 호출
    • 상담사가 메시지를 입력했을 때

5. 데이터 흐름 상세

5.1 고객 메시지 처리 흐름

[고객 입력]
  ↓
[고객 챗 서버 UI: script.js]
  POST /api/send-message
  ↓
[Chat Server: ChatController.sendMessage()]
  ↓
[ChatService.processMessage()]
  ↓
1. 세션 저장/조회 (saveSessionToDatabase)
2. Dialogflow CX API 호출 (getDialogflowResponse)
3. Chat Server DB에 저장 (saveChatResponse)
   - userMessage: 고객 메시지
   - reply: AI 응답
   - status: "SUCCESS"
4. Pub/Sub 발행
   - 고객 메시지 이벤트 (sender="user")
   - AI 응답 이벤트 (sender="bot")
  ↓
[Pub/Sub Topic: chat-events]
  ↓
[Monitor Server: PubSubPushController]
  ↓
1. Monitor Server DB에 저장
   - 고객 발화: userMessage 저장, reply=null
   - AI 응답: userMessage=null, reply 저장
2. WebSocket 브로드캐스트
  ↓
[Monitor Server UI: agent-monitor.js] 
  - WebSocket 또는 폴링으로 메시지 수신
  - 실시간 UI 업데이트

5.2 상담사 메시지 처리 흐름

[상담사 입력]
  ↓
[Monitor Server UI: agent-monitor.js]
  POST /api/agent/sessions/{sessionId}/respond
  ↓
[Monitor Server: AgentMonitorService.saveAgentResponse()]
  ↓
1. Monitor Server DB에 저장하지 않음 (Pub/Sub을 통해 받아서 저장)
2. Chat Server로 HTTP 요청
   POST {CHAT_SERVER_URL}/api/agent-message
  ↓
[Chat Server: ChatController.processAgentMessage()]
  ↓
1. 상담사 메시지 저장 및 Pub/Sub 발행
   - saveAndPublishAgentMessage()
   - Chat Server DB에 저장 (status="AGENT_RESPONSE")
   - Pub/Sub 발행 (sender="agent")
2. Dialogflow CX로 상담사 메시지 전달
   - processMessage() 호출 (agentMessage=true)
   - Dialogflow CX API 호출
3. AI 응답 저장 및 Pub/Sub 발행
   - saveChatResponse() 호출 (userMessage=null, 상담사 메시지에 대한 응답)
   - Pub/Sub 발행 (sender="bot")
  ↓
[Pub/Sub Topic: chat-events]
  ↓
[Monitor Server: PubSubPushController]
  ↓
1. 상담사 메시지 저장 (sender="agent")
   - 중복 체크 (responseId 기반)
   - Monitor Server DB에 저장
2. AI 응답 저장 (sender="bot")
   - Monitor Server DB에 저장
3. WebSocket 브로드캐스트
  ↓
[Monitor Server UI]
  - 실시간 UI 업데이트
  ↓
[고객 챗 서버 UI: script.js]
  - 폴링으로 메시지 조회 (GET /api/sessions/{sessionId}/messages)
  - 상담사 메시지 및 AI 응답 표시

5.3 메시지 저장 전략

중복 저장 방지:
- Monitor Server에서 상담사 메시지를 직접 저장하지 않음
- Chat Server에서 저장하고 Pub/Sub으로 발행
- Monitor Server는 Pub/Sub을 통해서만 받아서 저장
- responseId 기반 중복 체크 (existsByResponseId())

메시지 순서 보장:
- 모든 메시지가 Pub/Sub을 통해 Monitor Server에 도착
- createdAt과 id를 조합하여 정확한 순서 정렬
- 프론트엔드에서 타입 순서 추가 정렬 (고객 → AI → 상담사 → AI)

6. WebSocket 설정

6.1 Monitor Server WebSocket

엔드포인트: /ws/agent

설정 파일: WebSocketConfig.java (monitor profile 전용)

핸들러: AgentWebSocketHandler
- 연결 관리: AgentSessionManager
- 브로드캐스트: Pub/Sub 이벤트를 모든 연결된 상담사 세션에 전달

사용 목적:
- 실시간 UI 업데이트 (폴링 대체)
- Pub/Sub 이벤트를 WebSocket으로 브로드캐스트

6.2 WebSocket 데이터 흐름

[Pub/Sub 이벤트 수신]
  ↓
[PubSubPushController.receiveMessage()]
  ↓
[AgentSessionManager.broadcast(json)]
  ↓
[모든 연결된 WebSocket 세션에 전송]
  ↓
[Monitor Server UI: agent-monitor.js]
  - WebSocket onmessage 이벤트
  - 메시지 목록 새로고침
  - 세션 목록 새로고침

7. 배포 구조

Chat Server:
- 서비스명: customer-chat-server
- 리전: asia-northeast3
- 프로파일: chat
- 이미지: asia-northeast3-docker.pkg.dev/{PROJECT_ID}/chat-monitor-backend/chat-monitor-backend:latest

Monitor Server:

- 리전: asia-northeast3

Chat Server 환경 변수:

DIALOGFLOW_PROJECT_ID={PROJECT_ID}
DIALOGFLOW_LOCATION=asia-northeast1
DIALOGFLOW_AGENT_ID={AGENT_ID}
PUBSUB_TOPIC=chat-events

PUBSUB_VERIFY_TOKEN=secret-token

Chat Server:
- chat-sa@{PROJECT_ID}.iam.gserviceaccount.com
- Dialogflow CX API 호출 권한
- Pub/Sub 발행 권한

Monitor Server:
- monitor-sa@{PROJECT_ID}.iam.gserviceaccount.com
- Pub/Sub Push Subscription 인증
- Chat Server 호출 권한

8. 주요 서비스 및 컴포넌트

8.1 Chat Server (chat profile)

서비스:
- ChatService: 메시지 처리, Dialogflow CX 호출, DB 저장
- DialogflowCxApiService: Dialogflow CX API 통신
- ChatEventPublisher: Pub/Sub 이벤트 발행
- ExternalApiService: 외부 API 전달 (선택적)

컨트롤러:
- ChatController: 고객 메시지 및 상담사 메시지 처리
- HealthController: 헬스 체크
- WebhookController: Dialogflow CX Webhook (선택적)

8.2 Monitor Server (monitor profile)

서비스:
- AgentMonitorService: 세션 조회, 메시지 조회, 상담사 응답 처리
- AgentSessionManager: WebSocket 세션 관리 및 브로드캐스트
- AgentWebSocketHandler: WebSocket 연결 처리

컨트롤러:
- AgentMonitorController: 상담사 모니터링 API
- PubSubPushController: Pub/Sub Push Subscription 엔드포인트
- HealthController: 헬스 체크

9. 데이터 동기화 전략

9.1 Chat Server → Monitor Server

방식: Pub/Sub Push Subscription

흐름:
1. Chat Server에서 이벤트 발행
2. Pub/Sub이 Monitor Server의 /pubsub/push로 전달
3. Monitor Server가 DB에 저장

장점:
- 실시간 동기화
- 순서 보장 (Pub/Sub의 순서 보장)
- 중복 방지 (responseId 기반)

9.2 Monitor Server → Chat Server

방식: HTTP REST API

흐름:
1. Monitor Server에서 상담사 메시지 입력
2. Chat Server로 HTTP 요청 (/api/agent-message)
3. Chat Server가 처리 및 저장

이유:
- 상담사 메시지를 Dialogflow CX로 전달해야 함
- Chat Server가 단일 소스로 관리

10. 메시지 순서 보장

10.1 정렬 기준

  1. createdAt (시간순)
  1. 타입 순서 (같은 시간일 경우)
    • 고객 발화 (1)
    • AI 응답 (2)
    • 상담사 메시지 (3)
    • AI 응답 (2)
  1. id (최종 정렬)

10.2 예상 순서

일반 대화:

고객 → AI → 고객 → AI

상담사 개입:

고객 → AI → 고객 → 상담사 → AI

13. 현재 아키텍처의 특징

13.1 장점

  1. 서버 분리: Chat Server와 Monitor Server가 독립적으로 운영
  1. 실시간 동기화: Pub/Sub을 통한 비동기 메시지 전달
  1. 확장성: 각 서버를 독립적으로 스케일링 가능
  1. 단일 소스: Chat Server가 메시지의 단일 소스

13.2 제한사항

  1. 인메모리 DB: H2 In-Memory는 재시작 시 데이터 손실
  1. 데이터베이스 분리: Chat Server와 Monitor Server DB가 분리되어 있어 동기화 필요
  1. Pub/Sub 의존성: Monitor Server는 Pub/Sub을 통해서만 메시지를 받음

14. 향후 개선 방향

  1. 영구 저장소: Cloud SQL 또는 Firestore로 마이그레이션
  1. WebSocket 개선: Chat Server에도 WebSocket 추가 (고객 UI 실시간 업데이트)
  1. 에러 처리: Pub/Sub 실패 시 재시도 로직
  1. 모니터링: Cloud Monitoring 통합
