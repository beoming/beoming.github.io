전체 아키텍처 문서

1. 시스템 개요

이 애플리케이션은 Spring Boot 기반의 대화형 에이전트 시스템으로, 고객과 상담사 간의 실시간 채팅을 지원하며 Dialogflow CX를 통한 AI 챗봇 기능을 제공합니다.

주요 구성 요소

  • Chat Server (chat profile): 고객용 채팅 서버
  • Monitor Server (monitor profile): 상담사용 모니터링 서버
  • Dialogflow CX: Google의 대화형 AI 플랫폼
  • Google Cloud Pub/Sub: 서버 간 양방향 비동기 메시지 동기화
  • H2 In-Memory Database: 각 서버별 독립적인 인메모리 데이터베이스

핵심 아키텍처 원칙

  1. 메시지 처리의 일관성: 고객 메시지와 상담사 메시지 모두 동일한 흐름으로 처리
    • 메시지 입력 → DB 저장 → Dialogflow CX 호출 → AI 응답 저장 → Pub/Sub 발행
  1. 양방향 동기화: Pub/Sub을 통한 Chat Server ↔︎ Monitor Server 양방향 메시지 동기화
  1. 독립적인 데이터베이스: 각 서버는 독립적인 H2 In-Memory DB를 사용하며, Pub/Sub으로 동기화

2. 서버 구조

2.1 Chat Server (chat profile)

역할: 고객이 채팅을 하는 메인 서버

주요 기능:
- 고객 메시지 수신 및 처리
- Dialogflow CX API 호출 (고객 메시지에 대한 응답)
- 고객 챗 UI 제공 (/static/index.html)
- Pub/Sub Push Subscription을 통한 Monitor Server 메시지 수신
- 폴링 기반 실시간 메시지 업데이트

배포 정보:
- 서비스명: customer-chat-server
- 프로파일: chat
- 포트: 8080
- 데이터베이스: H2 In-Memory (독립적)
- 리전: asia-northeast3

주요 엔드포인트:
- POST /api/send-message: 고객 메시지 처리
- GET /api/sessions/{sessionId}/messages: 세션 메시지 조회 (고객 UI 폴링용)
- POST /pubsub/push: Pub/Sub Push Subscription 엔드포인트 (Monitor Server 메시지 수신)

주요 서비스:
- ChatService: 고객 메시지 처리, Dialogflow CX 호출, DB 저장, Pub/Sub 발행
- DialogflowCxApiService: Dialogflow CX API 통신
- ChatEventPublisher: Pub/Sub 이벤트 발행

2.2 Monitor Server (monitor profile)

역할: 상담사가 고객 대화를 모니터링하고 개입하는 서버

주요 기능:
- 상담사 모니터링 대시보드 제공 (/static/agent-monitor.html)
- 상담사 메시지 입력 및 처리 (Dialogflow CX 직접 호출)
- Dialogflow CX API 호출 (상담사 메시지에 대한 응답)
- Pub/Sub Push Subscription을 통한 Chat Server 메시지 수신
- WebSocket을 통한 실시간 UI 업데이트
- AI 분석 결과 표시 (Intent, Entity, 세션 파라미터)

배포 정보:
- 서비스명: agent-monitor-server
- 프로파일: monitor
- 포트: 8080
- 데이터베이스: H2 In-Memory (독립적, Chat Server와 분리)
- 리전: asia-northeast3

주요 엔드포인트:
- GET /api/agent/sessions: 모든 세션 목록 조회
- GET /api/agent/sessions/{sessionId}: 특정 세션 정보 조회
- GET /api/agent/sessions/{sessionId}/messages: 세션 메시지 조회
- POST /api/agent/sessions/{sessionId}/respond: 상담사 응답 전송
- POST /pubsub/push: Pub/Sub Push Subscription 엔드포인트 (Chat Server 메시지 수신)
- WebSocket /ws/agent: 실시간 이벤트 브로드캐스트

주요 서비스:
- AgentMonitorService: 세션 조회, 메시지 조회, 상담사 응답 처리 (Dialogflow CX 직접 호출)
- DialogflowCxApiService: Dialogflow CX API 통신
- ChatEventPublisher: Pub/Sub 이벤트 발행
- AgentSessionManager: WebSocket 세션 관리 및 브로드캐스트

3. 데이터베이스 구조

3.1 데이터베이스 분리

Chat Server DB와 Monitor Server DB는 완전히 독립적인 H2 In-Memory 데이터베이스입니다.

  • Chat Server DB:
    • 고객 메시지 저장
    • 고객 메시지에 대한 AI 응답 저장
    • Monitor Server에서 발행한 상담사 메시지 및 AI 응답 동기화 (Pub/Sub을 통해)
  • Monitor Server DB:
    • Chat Server에서 발행한 고객 메시지 및 AI 응답 동기화 (Pub/Sub을 통해)
    • 상담사 메시지 저장
    • 상담사 메시지에 대한 AI 응답 저장

3.2 엔티티 구조

ChatSession

- sessionId (PK): 세션 고유 ID
- language: 언어 설정
- firstRequestAt: 첫 요청 시간
- createdAt: 생성 시간
- updatedAt: 업데이트 시간
- responses: ChatResponse 리스트 (OneToMany)

ChatResponse

- id (PK): 자동 증가 ID
- responseId: 응답 고유 ID (중복 방지용, Pub/Sub 동기화에 사용)
- session: ChatSession (ManyToOne)
- userMessage: 고객 발화 (상담사 메시지인 경우 null)
- reply: AI 응답 또는 상담사 메시지
- status: "SUCCESS", "ERROR", "AGENT_RESPONSE" 등
- intentDisplayName: 인텐트 이름
- intentConfidence: 인텐트 신뢰도
- intentMatchType: 인텐트 매칭 타입
- intentsJson: 인텐트 리스트 (JSON, session_summary에서 추출)
- entitiesJson: 엔티티 정보 (JSON)
- sessionParametersJson: 모든 세션 파라미터 (JSON, Dialogflow CX에서 전달)
- createdAt: 생성 시간
- images: ProductImage 리스트
- shownProducts: ShownProduct 리스트

4. Pub/Sub 아키텍처

4.1 Pub/Sub 구성

Topic: chat-events
- 프로젝트: gsneotek-ncc-demo (또는 환경 변수로 설정)
- 생성 위치: infra/cloudrun-deploy-chat.sh 또는 infra/cloudrun-deploy-monitor.sh

Subscriptions:
1. agent-monitor-sub (Monitor Server용)
- 타입: Push Subscription
- Push Endpoint: https://agent-monitor-server-{region}.run.app/pubsub/push?token={VERIFY_TOKEN}
- 인증: Service Account (monitor-sa@{PROJECT_ID}.iam.gserviceaccount.com)
- 수신 메시지: Chat Server에서 발행한 고객 메시지 및 AI 응답

  1. customer-chat-sub (Chat Server용)
    • 타입: Push Subscription
    • Push Endpoint: https://customer-chat-server-{region}.run.app/pubsub/push?token={VERIFY_TOKEN}
    • 인증: Service Account (chat-sa@{PROJECT_ID}.iam.gserviceaccount.com)
    • 수신 메시지: Monitor Server에서 발행한 상담사 메시지 및 AI 응답

4.2 Pub/Sub 데이터 흐름

Chat Server → Monitor Server

[Chat Server]
  ↓
ChatEvent 발행 (ChatEventPublisher)
  - sender: "user" (고객 메시지)
  - sender: "bot" (AI 응답)
  ↓
[Pub/Sub Topic: chat-events]
  ↓
[Pub/Sub Push Subscription: agent-monitor-sub]
  ↓
[Monitor Server: /pubsub/push]
  ↓
PubSubPushController.receiveMessage()
  ↓
1. Base64 디코딩
2. ChatEvent 파싱
3. Monitor Server DB에 저장 (saveEventToDatabase)
4. WebSocket 브로드캐스트 (AgentSessionManager.broadcast)

Monitor Server → Chat Server

[Monitor Server]
  ↓
ChatEvent 발행 (ChatEventPublisher)
  - sender: "agent" (상담사 메시지)
  - sender: "bot" (AI 응답 - 상담사 메시지에 대한 응답)
  ↓
[Pub/Sub Topic: chat-events]
  ↓
[Pub/Sub Push Subscription: customer-chat-sub]
  ↓
[Chat Server: /pubsub/push]
  ↓
ChatPubSubPushController.receiveMessage()
  ↓
1. Base64 디코딩
2. ChatEvent 파싱
3. Chat Server DB에 저장 (saveEventToDatabase)

4.3 ChatEvent 모델

ChatEvent {
    conversationId: String      // 세션 ID
    messageId: String          // 고유 메시지 ID (responseId와 동일)
    sender: String             // "user" | "bot" | "agent"
    text: String               // 메시지 내용
    timestamp: String          // ISO 8601 형식
    channel: String            // "web"
    intentDisplayName: String  // 인텐트 이름
    intentConfidence: Double   // 인텐트 신뢰도
    intentMatchType: String    // 인텐트 매칭 타입
    intentList: List<String>   // session_summary에서 추출한 모든 인텐트
    status: String             // "SUCCESS", "ERROR", "AGENT_RESPONSE"
}

4.4 Pub/Sub 발행 시점

Chat Server에서 발행하는 이벤트:

  1. 고객 메시지 이벤트 (sender="user")
    • ChatService.publishUserMessageEvent() 호출
    • 고객이 메시지를 보낼 때
    • Chat Server DB에 저장 후 즉시 발행
  1. AI 응답 이벤트 (sender="bot")
    • ChatService.publishBotResponseEvent() 호출
    • Dialogflow CX 응답을 받았을 때 (고객 메시지에 대한 응답)
    • Chat Server DB에 저장 후 즉시 발행

Monitor Server에서 발행하는 이벤트:

  1. 상담사 메시지 이벤트 (sender="agent")
    • AgentMonitorService.saveAgentResponse() 내부에서 발행
    • 상담사가 메시지를 입력했을 때
    • Monitor Server DB에 저장 후 즉시 발행
  1. AI 응답 이벤트 (sender="bot")
    • AgentMonitorService.saveAgentResponse() 내부에서 발행
    • Dialogflow CX 응답을 받았을 때 (상담사 메시지에 대한 응답)
    • Monitor Server DB에 저장 후 즉시 발행

5. 데이터 흐름 상세

5.1 고객 메시지 처리 흐름

[고객 입력]
  ↓
[고객 챗 서버 UI: script.js]
  POST /api/send-message
  ↓
[Chat Server: ChatController.sendMessage()]
  ↓
[ChatService.processMessage()]
  ↓
1. 세션 저장/조회 (saveSessionToDatabase)
2. Dialogflow CX API 호출 (getDialogflowResponse)
   - DialogflowCxApiService.detectIntent()
   - 모든 세션 파라미터 추출 및 저장
3. Chat Server DB에 저장 (saveChatResponse)
   - userMessage: 고객 메시지
   - reply: AI 응답
   - status: "SUCCESS"
   - sessionParametersJson: 모든 세션 파라미터 (JSON)
4. Pub/Sub 발행
   - 고객 메시지 이벤트 (sender="user")
   - AI 응답 이벤트 (sender="bot")
  ↓
[Pub/Sub Topic: chat-events]
  ↓
[Monitor Server: PubSubPushController]
  ↓
1. Monitor Server DB에 저장
   - 고객 발화: userMessage 저장, reply=null
   - AI 응답: userMessage=null, reply 저장
2. WebSocket 브로드캐스트
  ↓
[Monitor Server UI: agent-monitor.js]
  - WebSocket 또는 폴링으로 메시지 수신
  - 실시간 UI 업데이트
  - AI 분석 결과 표시 (Intent, Entity, 세션 파라미터)

5.2 상담사 메시지 처리 흐름

[상담사 입력]
  ↓
[Monitor Server UI: agent-monitor.js]
  POST /api/agent/sessions/{sessionId}/respond
  ↓
[Monitor Server: AgentMonitorService.saveAgentResponse()]
  ↓
1. Monitor Server DB에 상담사 메시지 저장
   - userMessage: null
   - reply: 상담사 메시지
   - status: "AGENT_RESPONSE"
2. Pub/Sub 발행 (상담사 메시지 이벤트, sender="agent")
3. Dialogflow CX API 직접 호출
   - DialogflowCxApiService.detectIntent()
   - 상담사 메시지를 Dialogflow CX로 전달
   - 모든 세션 파라미터 추출 및 저장
4. Monitor Server DB에 AI 응답 저장
   - userMessage: null
   - reply: AI 응답
   - status: "SUCCESS"
   - sessionParametersJson: 모든 세션 파라미터 (JSON)
5. Pub/Sub 발행 (AI 응답 이벤트, sender="bot")
  ↓
[Pub/Sub Topic: chat-events]
  ↓
[Chat Server: ChatPubSubPushController]
  ↓
1. Chat Server DB에 저장
   - 상담사 메시지: userMessage=null, reply=상담사 메시지, status="AGENT_RESPONSE"
   - AI 응답: userMessage=null, reply=AI 응답, status="SUCCESS"
  ↓
[고객 챗 서버 UI: script.js]
  - 폴링으로 메시지 조회 (GET /api/sessions/{sessionId}/messages)
  - 상담사 메시지 및 AI 응답 표시

5.3 메시지 처리의 일관성

핵심 원칙: 고객 메시지와 상담사 메시지 모두 동일한 흐름으로 처리

  1. 메시지 입력 → DB 저장
  1. Dialogflow CX 호출 → AI 응답 받기
  1. AI 응답 저장 → DB 저장
  1. Pub/Sub 발행 → 다른 서버로 동기화

차이점:
- 고객 메시지: Chat Server에서 처리
- 상담사 메시지: Monitor Server에서 처리 (Chat Server로 HTTP 요청하지 않음)

5.4 메시지 저장 전략

중복 저장 방지:
- responseId 기반 중복 체크 (existsByResponseId())
- Pub/Sub Push Controller에서 중복 체크 후 저장
- 동일한 responseId가 이미 있으면 건너뜀

메시지 순서 보장:
- 모든 메시지가 createdAt 타임스탬프를 가짐
- 프론트엔드에서 createdAt과 메시지 타입을 조합하여 정확한 순서 정렬
- 타입 순서: 고객(1) → AI(2) → 상담사(3) → AI(2)

9. 데이터 동기화 전략

9.1 양방향 동기화

Chat Server → Monitor Server:
- 방식: Pub/Sub Push Subscription (agent-monitor-sub)
- 흐름: Chat Server에서 이벤트 발행 → Monitor Server의 /pubsub/push로 전달 → Monitor Server DB에 저장
- 이벤트 타입: 고객 메시지 (sender="user"), AI 응답 (sender="bot")

Monitor Server → Chat Server:
- 방식: Pub/Sub Push Subscription (customer-chat-sub)
- 흐름: Monitor Server에서 이벤트 발행 → Chat Server의 /pubsub/push로 전달 → Chat Server DB에 저장
- 이벤트 타입: 상담사 메시지 (sender="agent"), AI 응답 (sender="bot")

9.2 동기화 장점

  • 실시간 동기화: Pub/Sub Push를 통한 즉시 전달
  • 순서 보장: Pub/Sub의 메시지 순서 보장
  • 중복 방지: responseId 기반 중복 체크
  • 독립성 유지: 각 서버는 독립적인 DB를 유지하면서 동기화

10. 메시지 순서 보장

10.1 정렬 기준

  1. createdAt (시간순, 우선순위)
  1. 타입 순서 (같은 시간일 경우)
    • 고객 발화 (1)
    • AI 응답 (2)
    • 상담사 메시지 (3)
    • AI 응답 (2)
  1. id (최종 정렬)

10.2 예상 순서

일반 대화:

고객 → AI → 고객 → AI

상담사 개입:

고객 → AI → 고객 → 상담사 → AI

11. AI 분석 결과

11.1 Monitor Server UI 표시 항목

왼쪽 하단 “AI 분석 결과” 섹션:
1. Intent 분석: session_summary에서 추출한 모든 인텐트 리스트
2. Entity 추출: Dialogflow CX에서 추출한 엔티티 정보
3. 세션 파라미터: Dialogflow CX가 전달하는 모든 세션 파라미터 (JSON 형식)

11.2 세션 파라미터 저장

  • 저장 위치: ChatResponse.sessionParametersJson (JSON 문자열)
  • 추출 시점: Dialogflow CX API 응답에서 queryResult.parameters 추출
  • 표시 형식: 파라미터명과 값의 테이블 형식 (객체/배열은 JSON 문자열로 표시)

12. 보안 및 인증

12.1 Pub/Sub Push Subscription 인증

  • 토큰 검증: PUBSUB_VERIFY_TOKEN 환경 변수
  • Service Account: 각 서버의 Service Account로 인증
  • 엔드포인트: /pubsub/push?token={VERIFY_TOKEN}

12.2 Monitor Server 접근 제어

  • Basic Auth: 상담사 모니터링 대시보드 접근 제어 (선택적)
  • 환경 변수: AGENT_BASIC_USER, AGENT_BASIC_PASS

14. 현재 아키텍처의 특징

14.1 장점

  1. 서버 분리: Chat Server와 Monitor Server가 독립적으로 운영
  1. 양방향 실시간 동기화: Pub/Sub을 통한 양방향 메시지 전달
  1. 확장성: 각 서버를 독립적으로 스케일링 가능
  1. 일관된 처리 흐름: 고객 메시지와 상담사 메시지 모두 동일한 흐름으로 처리
  1. 독립적인 데이터베이스: 각 서버가 독립적인 DB를 유지하면서 Pub/Sub으로 동기화

14.2 제한사항

  1. 인메모리 DB: H2 In-Memory는 재시작 시 데이터 손실
  1. 데이터베이스 분리: Chat Server와 Monitor Server DB가 분리되어 있어 Pub/Sub 동기화 필요
  1. Pub/Sub 의존성: 양방향 동기화가 Pub/Sub에 의존

15. 향후 개선 방향

  1. 영구 저장소: Cloud SQL 또는 Firestore로 마이그레이션
  1. WebSocket 개선: Chat Server에도 WebSocket 추가 (고객 UI 실시간 업데이트)
  1. 에러 처리: Pub/Sub 실패 시 재시도 로직
  1. 모니터링: Cloud Monitoring 통합
  1. 세션 파라미터 필터링: 시스템 파라미터 자동 필터링 옵션

16. 주요 변경 이력

최근 주요 변경사항

  1. 상담사 메시지 처리 흐름 변경 (2025-01):

    • Chat Server로 HTTP 요청하지 않음

  1. 양방향 Pub/Sub 동기화 (2025-01):

    • Monitor Server UI에 “세션 파라미터” 섹션 추가
