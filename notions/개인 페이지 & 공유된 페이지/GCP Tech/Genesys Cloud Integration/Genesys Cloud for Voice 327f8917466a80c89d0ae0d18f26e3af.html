Genesys Cloud for Voice

1. Voice Connector가 하는 일

역할

Genesys Cloud Voice Connector는 Genesys Cloud의 실시간 통화 오디오를 받아서 Agent Assist 백엔드로 전달하는 브리지입니다.

핵심 동작:

  • Genesys Cloud의 AudioHook API 사용
  • 지속적인 WebSocket 연결로 오디오 스트림 수신
  • 내부적으로 gRPC 등을 통해 Agent Assist 연동 백엔드로 전달
  • Dialogflow / Agent Assist가 실시간 분석 수행
  • 결과를 상담사 데스크톱에 실시간 표시

즉 한 줄로 말하면:

통화 음성을 Agent Assist가 이해할 수 있는 실시간 분석 파이프라인으로 연결해주는 컴포넌트입니다.

2. 전체 흐름

흐름은 이렇게 보면 됩니다.

  1. 고객과 상담사가 통화 시작
  1. Genesys Cloud의 AudioHook Monitor가 오디오 스트림 캡처
  1. 캡처된 오디오를 Voice Connector 서버로 전달
  1. Voice Connector가 이 스트림을 Agent Assist / Dialogflow 쪽으로 전달
  1. Google 쪽에서 실시간으로
    • 전사(Transcription)
    • 감정 분석(Sentiment)
    • 의도 분석(Intent Detection)

      수행

  1. 분석 결과를 Agent Assist Integration Backend를 통해
  1. 상담사 Agent Desktop으로 전달

즉,

음성 → 스트리밍 → AI 분석 → 상담사 화면 피드백

3. 사전 조건

문서상 중요한 조건은:

먼저 이전 영상의 Genesys Cloud integration steps 1~6을 끝내야 한다는 점입니다.

즉 Voice Connector는 독립 배포가 아니라,

기존의 Agent Assist Integration Backend / UI Connector / Redis / Genesys Chat/Widget 연동 기반 위에 추가되는 구성입니다.

쉽게 말해:

  • 먼저 기본 Agent Assist 연동이 되어 있어야 하고
  • 그 다음 음성용 connector를 얹는 구조입니다.

4. 배포 절차 요약

1) 저장소 클론

새 디렉터리에 Git 저장소를 clone 합니다.

그 다음:

  • audiohook branch로 전환
  • audiohook 디렉터리로 이동

즉 이 음성 연동은 별도 브랜치/모듈로 관리되는 구조입니다.

2) Agent Assist Integration Backend 먼저 배포

Voice Connector보다 먼저 Agent Assist integration backend가 배포되어 있어야 합니다.

이유:

  • Voice Connector가 backend 설정값을 참조해야 함
  • Redis, UI Connector, VPC Connector 같은 공통 설정을 재사용하기 때문

3) 환경 변수 추출

Agent Assist integration backend 배포 후,

명령어를 실행해서 필요한 값을 .env.extracted 파일로 추출합니다.

이 파일에는 보통:

  • 이미 배포된 백엔드의 Cloud Run URL
  • Redis 관련 값
  • VPC Connector 관련 값
  • 기타 공통 설정

이 들어갑니다.

즉, Voice Connector는 완전히 새로 세팅하는 게 아니라

기존 백엔드 설정을 이어받는 방식입니다.

4) Genesys AppFoundry에서 AudioHook Monitor 설치

Genesys 쪽에서는 AudioHook Monitor를 설치해야 합니다.

이게 실제로 통화 오디오를 캡처해서

Voice Connector에 전달하는 역할입니다.

즉 Voice Connector만 배포해도 끝나는 게 아니고,

Genesys 쪽에 AudioHook Monitor integration도 붙어 있어야 합니다.

5) .env 파일 생성

.env.sample 파일을 .env로 복사합니다.

그 다음 필요한 환경 변수를 채웁니다.

5. 주요 환경 변수 정리

CONVERSATION_PROFILE_NAME

  • 애플리케이션 서버 배포 때 쓴 값과 동일하게 사용
  • 어떤 conversation profile을 사용할지 지정

SERVICE_REGION

  • Cloud Run 서비스가 배포된 리전
  • 예: us-central1 같은 값

GCP_PROJECT_ID

  • Google Cloud 프로젝트 ID
  • 전역 유일한 프로젝트 식별자

REDIS_INSTANCE_ID

  • Agent Assist Integration Backend에서 사용한 Redis 인스턴스 ID와 동일
  • 실시간 이벤트/상태 연동용

UI_CONNECTOR

  • Agent Assist Integration Backend 쪽 Cloud Run UI Connector 서비스의 도메인 이름
  • 음성 분석 결과를 최종적으로 상담사 UI와 연결하는 데 필요

SERVICE_ACCOUNT

  • AudioHook Connector 서비스가 사용할 서비스 계정
  • 없으면 deploy.sh가 새로 생성 가능

VPC_CONNECTOR_NAME

  • Agent Assist Integration Backend에서 사용한 값과 동일
  • Cloud Run에서 Redis 등 사설 자원 접근 시 필요할 가능성이 큼

API_KEY

  • Genesys AudioHook Monitor integration에서 생성한 API key
  • AudioHook Monitor가 Voice Connector에 안전하게 연결하기 위해 사용

VOICE_INTERCEPTOR_SERVICE

  • 음성 인터셉터용 Cloud Run 서비스 이름
  • 사용자가 지정

6. .env.extracted 값 병합

필수 변수들을 .env에 직접 입력한 뒤,

.env.extracted에 있는 추출값을 .env에 추가합니다.

즉 최종 .env는:

  • 수동 입력한 값
  • 기존 backend에서 가져온 값

이 합쳐진 형태가 됩니다.

7. 배포 실행

설정이 끝나면 audiohook 디렉터리에서

deploy.sh 스크립트 실행

이 스크립트가 보통 다음을 처리합니다.

  • Cloud Run 서비스 배포
  • 서비스 계정 생성/권한 처리
  • 환경 변수 반영
  • 네트워크 연결 구성

즉 실질적인 Voice Connector 배포 단계입니다.

8. 마지막 설정: Genesys Cloud AudioHook Monitor 구성

배포 후에는 Genesys Cloud에서 AudioHook Monitor integration 설정을 해야 합니다.

여기서 설정하는 핵심 값:

  • audio channel
  • connection URI

즉,

AudioHook Monitor가

“어느 오디오 채널을 어떤 URI로 보낼지”

지정해야 실제 음성 스트리밍이 Voice Connector로 들어옵니다.

이 단계까지 끝나야 배포 완료입니다.

9. 최종 구조 이해

완성된 구조는 이렇게 볼 수 있습니다.

음성 통화 발생

↓

Genesys Cloud AudioHook Monitor가 오디오 캡처

↓

Voice Connector로 스트리밍 전송

↓

Agent Assist / Dialogflow가 실시간 분석

  • STT
  • 감정 분석
  • 의도 분석

    ↓

Agent Assist Integration Backend로 전달

↓

Agent Desktop UI에 실시간 노출

10. 채팅 연동과의 차이

채팅 연동

  • 텍스트 기반
  • Interaction Widget / Messenger 중심
  • OAuth, UI 모듈, 위젯 설정이 핵심

음성 연동

  • 오디오 스트림 기반
  • AudioHook Monitor + Voice Connector 필요
  • WebSocket/gRPC/실시간 스트리밍 처리 필요

즉 Voice는 Chat보다 한 단계 더 복잡합니다.
