Integration Agent Assist with Chatbot Systems

Backend Integration - Pub/Sub

Agent Desktop에 Agent Assist를 실시간으로 붙이기 위한 백엔드 연동 구조.

conversational agent에서 발생한 이벤트를 Pub/Sub으로 받고 → Redis를 거쳐 → UI Connector가 WebSocket으로 상담사 화면에 실시간 전달하는 구조이다.

1. 전체 목적

상담사가 사용하는 Agent Desktop 안에 iFrame 형태의 Agent Assist UI 가 들어있고, 대화 중에 발생하는 추천 답변, 가이드, 이벤트 같은 정보를 실시간으로 화면에 띄우는 것이 목표임.

다만, Conversational Agent의 이벤트는 바로 브라우저로 가지 않고, 중간에 여러 백엔드 컴포넌트를 거친다.

2. 각 컴포넌트 역할

2-1. Conversational Agents

이벤트를 발생시키는 원천입니다.

예를 들면:

  • 대화 상태 변경
  • Agent Assist 추천 생성
  • 상담사 피드백 필요
  • 특정 conversation 업데이트

이런 이벤트를 Cloud Pub/Sub로 publish 합니다.

2-2. Cloud Pub/Sub

이벤트를 받아두는 메시지 큐 / 이벤트 버스 역할입니다.

왜 필요한가:

  • 이벤트 생산자와 소비자를 느슨하게 분리
  • 순간적인 이벤트 폭증을 흡수
  • 비동기 처리 가능

즉, Conversational Agents는 “이벤트를 Pub/Sub에 던지기만” 하면 되고,

그 다음 처리는 뒤의 서비스들이 맡습니다.

2-3. Cloud Pub/Sub Interceptor (Cloud Run)

이 서비스는 Pub/Sub 이벤트를 받아 Redis로 옮기는 중계기입니다.

역할:

  • Pub/Sub가 HTTP POST로 넘긴 이벤트 수신
  • 이벤트 payload 해석
  • 어떤 conversation에 대한 이벤트인지 파악
  • 해당 이벤트를 Redis Pub/Sub 채널에 publish

쉽게 말하면:

“Pub/Sub 메시지를 UI Connector가 읽기 좋은 형태로 Redis에 다시 전달하는 서비스”

입니다.

또 중요한 포인트는,

이 서비스가 이벤트를 그냥 공통 채널에 뿌리는 게 아니라

어느 UI Connector 인스턴스가 어떤 conversation을 담당 중인지를 기준으로 맞는 채널로 보내야 한다는 점입니다.

2-4. Memory Store Redis

여기서는 Redis가 두 가지 역할을 합니다.

1) 매핑 저장소

어떤 conversation이 현재 어떤 UI Connector 인스턴스에 붙어 있는지 저장합니다.

예:

  • conversation A → connector-1
  • conversation B → connector-3

이 매핑이 있어야 Interceptor가 이벤트를 정확한 대상에게 보낼 수 있습니다.

2) Pub/Sub 브로커

이벤트를 Redis 채널로 publish 하고, UI Connector가 subscribe 합니다.

즉 Redis는 여기서:

  • 단순 캐시
  • 상태 매핑 저장소
  • 실시간 publish/subscribe 허브

역할을 동시에 합니다.

2-5. UI Connector (Cloud Run)

이 구조에서 가장 핵심입니다.

이 서비스는 3가지 역할을 합니다.

1) 인증 게이트웨이

Agent Desktop이 처음 접속할 때 커스텀 토큰(예: Salesforce Bearer Token)을 받아 검증하고,

그 다음 내부적으로 사용할 임시 JWT를 발급합니다.

즉:

  • 외부 시스템 토큰 → 검증
  • 내부 통신용 JWT 발급

이후 프론트엔드는 원래 토큰 대신 이 JWT로 UI Connector와 통신합니다.

2) Conversational Agents API 프록시

프론트엔드가 Conversational Agents API를 직접 부르지 않고,

UI Connector를 통해 요청을 보냅니다.

예:

  • 피드백 전송
  • 특정 assist 요청
  • conversation 관련 API 호출

이렇게 하면:

  • 인증 제어가 쉬워지고
  • 백엔드에서 공통 로직 처리 가능
  • 프론트엔드에 민감한 자격정보를 노출하지 않게 됩니다

3) WebSocket 서버

프론트엔드와 stateful WebSocket 연결을 유지합니다.

이게 중요한 이유는:

  • 실시간 이벤트를 즉시 push 가능
  • Agent Desktop이 conversation 진행 중 계속 같은 컨테이너와 연결됨
  • UI Connector가 Redis 채널 구독 후 받은 이벤트를 브라우저로 바로 전달 가능

즉 UI Connector는

Redis subscriber + WebSocket push server + API proxy + auth server

를 모두 맡고 있습니다.

2-6. Agent Assist UI

상담사 화면에 들어가는 프론트엔드입니다.

보통 iFrame 형태로 Agent Desktop 안에 삽입됩니다.

역할:

  • UI Connector에 register 요청
  • JWT 발급받기
  • WebSocket 연결
  • 실시간 이벤트 수신
  • 상담사에게 추천, 힌트, 상태 변화 표시
  • 필요 시 UI Connector를 통해 Conversational Agents에 API 요청

2-7. Secret Manager

보안 정보 저장소입니다.

예:

  • JWT signing key
  • 서비스 인증용 secret
  • 기타 민감 설정값

즉, 인증 관련 비밀값을 코드나 환경에 직접 하드코딩하지 않고 여기서 관리합니다.

2-8. Authentication Backend

외부 인증 시스템입니다.

예를 들어:

  • Salesforce
  • 사내 SSO
  • 별도 인증 API

UI Connector가 register 시점에 이 백엔드와 연동해서

“이 상담사 토큰이 유효한가?”를 검증합니다.
