Telephony Integration for Agent Assist

1. 전체 핵심 요약

  1. Agent Assist는 텍스트 기반으로 잘 동작하지만, 전화 상담에서도 쓰려면 실시간 음성 연동이 필요하다.
  1. 그 연동 방식의 대표가 SIPREC 와 gRPC 이다.
  1. 음성은 결국 실시간 전사(STT) 를 거쳐 텍스트가 되고, 그 텍스트를 기반으로 상담사 추천, 인사이트 분석, UI 표시가 이뤄진다.

즉 구조를 한 줄로 줄이면:

고객 통화 음성 → 실시간 전달 → 전사(STT) → Agent Assist 분석 → 상담사 화면에 추천/자막 제공

2. 왜 Telephony Integration이 중요한가

Agent Assist의 실시간 추천은 라이브 통화의 오디오 스트림을 받아야 가능하다는 점

채팅은 원래 텍스트가 있으니까 바로 분석할 수 있지만, 전화 상담은 음성이기 때문에 먼저 시스템이 그 음성을 받아야 합니다.

즉 전화 상담에서 Agent Assist를 쓰기 위해선 두 단계가 필요합니다.

  • 실시간 음성 수집
  • 음성을 텍스트로 변환 후 분석

그래서 telephony integration이 필수입니다.

3. 두 가지 주요 연동 방식

3-1. SIPREC 방식

SIPREC는 Session Initiation Recording Protocol입니다.

쉽게 말하면 통화 녹취/복제용 표준 프로토콜입니다.

이 방식에서는 고객사의 전화 시스템이 통화 오디오를 복제해서 Google 쪽으로 보내고, Google 쪽이 그걸 받아 Agent Assist에 연결합니다.

이걸 좀 더 쉽게 말하면:

  • 고객 전화 시스템이 원래 고객과 상담사를 연결함
  • 동시에 오디오 한 벌을 복사(forking) 해서 Google에 보냄
  • Google이 그 음성을 전사하고 분석함
  • 결과를 상담사 화면에 추천으로 보여줌

즉 기존 전화 시스템을 유지하면서 녹취 스트림만 Google에 넘기는 방식으로 이해하면 됩니다.

3-2. gRPC 방식

gRPC 방식은 SIPREC처럼 별도의 SIP 녹취 프로토콜을 쓰는 대신,

고객이 직접 Conversation AI API endpoint로 오디오를 스트리밍하는 방식입니다.

쉽게 말하면:

  • 고객 시스템이 음성을 직접 API로 보냄
  • Google Conversation AI가 그 스트림을 받아서 처리함
  • 실시간 전사/추천 결과를 다시 전달함

즉 SIPREC보다 더 API 중심적이고, 통신 제어를 더 직접적으로 하는 구조라고 보면 됩니다.

4. 네 가지 Integration Flow 설명

강의에서는 SIPREC/gRPC를 기반으로 실제 배포 형태를 4가지로 설명합니다.

4-1. Direct SIPREC

이건 고객이 SIPREC 연동과 UI 연동을 모두 직접 관리하는 형태입니다.

특징:

  • 고객이 SBC(Session Border Controller)를 직접 제어해야 함
  • 통화 복제, Google 전달, 상담사 UI 연동까지 고객이 많이 담당함

즉 고객 주도형, 커스터마이징 자유도가 높은 방식입니다.

4-2. Through CCaaS Platform

여기서 CCaaS는 Contact Center as a Service입니다. 예: Twilio, Five9

특징:

  • 고객이 직접 복잡한 통합을 안 해도 됨
  • CCaaS 제공업체가 Agent Assist와의 연동을 지원함
  • 구현 부담이 줄어듦

즉 전화 플랫폼 사업자가 중간에서 연동을 쉽게 해주는 구조입니다.

4-3. Through Legacy Platform

기존 레거시 전화 시스템을 쓰는 고객의 경우입니다.

특징:

  • 기존 플랫폼이 최신 구조가 아닐 수 있음
  • 그래서 Conversation AI connector나 OEM connector가 필요할 수 있음
  • 제품/벤더별 맞춤 로직이 들어갈 가능성이 큼

즉 구형 시스템과의 호환을 위한 브리지 구조입니다.

4-4. Through Native OEM Partner

Google과 네이티브하게 연동되는 OEM 파트너를 통해 gRPC 통합이 더 자연스럽게 이뤄지는 방식입니다.

특징:

  • 성능 최적화
  • 연동 편의성 향상
  • 사용자 경험 개선

즉 공식/긴밀한 파트너 기반의 최적화된 연동 방식입니다.

5. SIPREC를 정확히 이해하기

5-1. SIPREC란?

  • Session Recording Protocol
  • IETF 표준
  • SIP(Session Initiation Protocol)의 확장
  • 목적: 통화 녹취/기록

즉 SIP 자체가 통화 연결용이라면, SIPREC는 그 통화를 녹취하거나 기록 세션으로 전달하는 용도입니다.

5-2. SIPREC의 두 핵심 구성요소

SRC

Session Recording Client

녹취를 시작하는 쪽입니다.

보통 고객사 장비(PBX, SBC 등)가 이 역할을 합니다.

강의에서는:

  • 고객 SBC 또는 telephony system = SRC

SRS

Session Recording Server

녹취 데이터를 받아서 처리하는 서버입니다.

강의에서는:

  • Google Telephony Platform(GTP) = SRS

즉 관계는 다음과 같습니다.

  • 고객 전화 시스템이 녹취 시작 → SRC
  • Google Telephony Gateway가 받아 처리 → SRS

5-3. SIPREC에서 무엇이 전달되나

SRC와 SRS는 SIPREC 메시지로 다음을 주고받습니다.

  • recording session 설정
  • call metadata 전달
  • 실제 audio stream 전달

즉 단순히 음성만 보내는 게 아니라,

누가 통화 중인지, 어떤 세션인지 같은 메타데이터와 오디오를 함께 전달합니다.

6. GTP(Google Telephony Platform)의 역할

강의 표현대로 하면 GTP는 번역기(translator) 역할을 합니다.

왜냐하면 고객 전화 시스템은 SIPREC를 쓰지만,

Conversational AI는 내부적으로 gRPC 기반 포맷을 더 잘 이해하기 때문입니다.

그래서 GTP가 하는 일은:

  • SIPREC 메시지 수신
  • 이를 Conversational AI가 이해할 수 있는 형태로 변환
  • 세션/컨텍스트 연결
  • 오디오 전달

즉 구조는 대략 이렇습니다.

고객 SBC(SIPREC) → GTP → Conversational AI(gRPC 형식)

7. Signaling과 Audio Separation

강의에서 중간에 “signaling과 media는 분리된다”는 얘기가 나옵니다.

이건 전화 시스템 이해에서 꽤 중요합니다.

signaling

통화 제어 정보입니다.

예:

  • 전화 연결 시작
  • 끊김
  • 세션 생성
  • INVITE 처리

media

실제 음성 데이터입니다.

즉 전화에서는

  • 통화 상태를 관리하는 신호
  • 실제 음성 흐름

    이 분리되어 있습니다.

강의에서는 GTP 내부에서:

  • TPAPI가 call event, signaling 관리
  • Iridessa service가 audio stream 처리

라고 설명합니다.

이걸 쉽게 말하면:

  • 누가 전화를 걸었는지, 세션이 뭔지 = TPAPI
  • 실제 음성이 흐르는 부분 = Iridessa

8. SIP INVITE와 Session ID, CallContext

GTP가 SIP INVITE를 받는다고 합니다.

흐름은 대략 이렇습니다.

  1. 고객이 지정된 번호로 전화
  1. GTP가 SIP INVITE 수신
  1. 그 안에서 session ID 확인
  1. 그 session ID를 바탕으로 CallContext 구성
  1. Conversational AI에 incoming call 알림

즉 요점은:

전화 한 통을 Conversation AI의 특정 세션/대화와 정확히 연결해야 한다는 것입니다.

그 연결의 핵심 식별자가 session ID이고,

그걸 기반으로 CallContext를 만든다고 이해하면 됩니다.

9. GTP와 Conversational AI의 매핑 방식 3가지

9-1. Dedicated Phone Number

고객이 전용 Conversational AI 전화번호로 직접 건다.

  • 번호 자체가 특정 conversation profile과 연결됨
  • 비교적 단순한 매핑

즉 번호 기반 매핑입니다.

9-2. Proxy Phone Number Assignment

Conversational AI가 각 conversation마다 임시 번호를 부여합니다.

  • 그 임시 번호가 conversation 식별자 역할을 함

즉 임시 번호 기반 추적 방식입니다.

9-3. SIP Integration + CallMatcher API

고객이 SIP header 정보를 가져와 내부 데이터와 연결합니다.

  • CallMatcher API 사용
  • SIP 헤더 기반으로 고객 시스템이 추가 매핑

즉 헤더 정보 기반 커스텀 매핑입니다.

10. SIPREC 기반 end-to-end 흐름

고객이 회사 고객센터로 전화했을 때 어떤 일이 일어나는지 순서대로 보면 됩니다.

1단계. Audio Forking

고객사 SBC가 통화 오디오를 복제해서 Google SIPREC endpoint로 보냅니다.

핵심:

  • 원래 통화는 계속 진행됨
  • Google은 복사본만 받음

즉 Agent Assist는 통화에 끼어드는 게 아니라 듣기만 하는 passive listener에 가깝습니다.

2단계. GTP가 Conversational AI로 전달

Google Telephony Platform이 이 오디오를 받아

Conversational Agents / Agent Assist 쪽으로 넘깁니다.

3단계. Agent Assist가 Listener로 참여

Agent Assist는 고객과 상담사의 대화를 실시간 청취합니다.

여기서 중요한 건:

  • 직접 통화를 제어하는 주체라기보다
  • 분석용 청취자 역할이라는 점입니다.

4단계. STT 전사

Speech-to-Text가 음성을 텍스트로 바꿉니다.

이건 필수 단계입니다.

왜냐하면 Agent Assist는 추천을 할 때 기본적으로 텍스트를 기반으로 분석하기 때문입니다.

5단계. 분석

전사된 텍스트를 바탕으로:

  • 상담 맥락 이해
  • 고객 의도 파악
  • 필요한 정보/답변/추천 생성

6단계. Agent Suggestions 전달

추천 결과는 두 가지 방식으로 상담사 화면에 갈 수 있습니다.

REST API

상담사 데스크톱 애플리케이션이 API로 받아서 표시

Pub/Sub

메시지 브로드캐스트 방식으로 데스크톱 클라이언트에 전달

즉 최종 목적지는 상담사 UI입니다.

11. 아키텍처 6단계 정리

1. Audio Forking

SBC가 음성을 복제해 Google로 전송

2. SIPREC to Conversational AI

GTP가 오디오를 Conversational AI로 전달

3. Agent Assist as Listener

Agent Assist가 수동 청취자로 참여

4. Transcription

실시간 STT 수행

5. Optional Redaction

필요하면 DLP로 민감정보 마스킹

6. Agent Desktop UI Integration

최종 텍스트/추천을 상담사 UI에 연결

이 6단계는 거의 그대로 외워도 좋습니다.

12. Optional Redaction이 중요한 이유

강의에서 DLP(Data Loss Prevention)를 통한 redaction을 옵션으로 언급합니다.

이건 실무적으로 매우 중요합니다.

예를 들어 전사 텍스트 안에 아래 정보가 들어갈 수 있습니다.

  • 카드번호
  • 주민/신분 정보
  • 전화번호
  • 이메일
  • 주소
  • 기타 개인정보

이런 정보는 그대로 상담사 앱, 저장소, 분석 시스템으로 보내면 위험할 수 있으므로

DLP를 통해 마스킹/치환 후 전달할 수 있습니다.

즉 목적은:

  • 개인정보 보호
  • 규제 대응
  • 로그/저장소 안전성 확보

13. Transcript 이후에 가능한 것들

13-1. 원본 오디오 저장

SIPREC 오디오는 Google Cloud Storage에 저장 가능

13-2. Transcript 저장

전사 텍스트도 함께 저장 가능

13-3. Conversational Insights 분석

대화 내용을 자동 분석하여

  • 상담 품질
  • 반복 이슈
  • 고객 불만 패턴
  • 상담 효율성

    같은 인사이트를 뽑아낼 수 있습니다.

즉 Agent Assist는 단순 실시간 추천 도구가 아니라,

사후 분석/품질 관리로도 확장되는 구조입니다.

14. gRPC 방식 흐름 정리

이제 SIPREC가 아닌 gRPC 방식입니다.

이 방식은 좀 더 API 중심이라서

개발자가 리소스 생성과 스트리밍 호출을 명시적으로 해줘야 합니다.

강의에서는 다음 순서로 설명합니다.

15-1. Conversation Profile 생성

API의 create method로 ConversationProfile 생성

이건 대화 처리 방식, 연결 설정 등을 담는 일종의 프로필입니다.

15-2. Conversation 생성

API로 대화를 시작합니다.

여기서 참가자와 역할 등의 기본 세팅이 들어갑니다.

15-3. Participant 생성

participant resource의 create method로 참가자 생성

역할은 다음 중 선택:

  • HUMAN_AGENT
  • AUTOMATED_AGENT
  • END_USER

즉 상담사, 자동화 에이전트, 고객을 시스템 안에서 개체로 정의하는 것입니다.

15-4. Audio Transmission and Transcription

StreamingAnalyzeContent 메서드로 해당 participant의 오디오를 스트리밍 전송

이 과정에서 실시간 전사가 함께 이뤄집니다.

즉 gRPC 방식의 핵심 API는:

StreamingAnalyzeContent

입니다.

SIPREC와 gRPC 차이 쉽게 비교

SIPREC

  • 전화 시스템 친화적
  • 통화 오디오를 복제해서 Google에 전달
  • 고객 SBC/telephony 인프라와 잘 맞음
  • 기존 전화 인프라 연계에 강함

gRPC

  • API 중심
  • 고객이 직접 오디오 스트리밍
  • 세션/참가자/대화 생성도 API 기반
  • 더 유연하지만 구현 부담이 더 직접적일 수 있음

아주 쉽게 말하면:

  • SIPREC = 전화 인프라 중심 통합
  • gRPC = 애플리케이션/API 중심 통합

헷갈리기 쉬운 부분 정리

1) Agent Assist가 통화를 제어하나?

아닙니다.

이 강의에서는 Agent Assist가 passive listener처럼 동작한다고 설명합니다.

즉 통화를 듣고 분석해서 추천을 제공하는 역할입니다.

2) SIPREC와 gRPC는 완전히 별개인가?

접근 방식은 다르지만 목표는 같습니다.

둘 다 실시간 음성을 Agent Assist에 연결하기 위한 방식입니다.

3) GTP는 왜 필요한가?

고객 전화 시스템의 SIPREC 메시지를 Google Conversational AI가 처리하기 쉬운 형태로 이어주는 중간 변환/중개 계층이기 때문입니다.

4) 왜 DLP가 중간에 들어가나?

전사 텍스트에 개인정보가 포함될 수 있어서입니다.

아주 간단한 압축 요약

  • 전화 상담에서 Agent Assist를 쓰려면 실시간 오디오 연동이 필요하다.
  • 주요 방식은 SIPREC 와 gRPC 두 가지다.
  • SIPREC에서는 고객 SBC가 SRC, Google GTP가 SRS 역할을 한다.
  • GTP는 SIPREC를 Conversational AI가 이해할 수 있게 중개한다.
  • 오디오는 STT를 통해 텍스트로 바뀌고, Agent Assist가 이를 분석해 상담사 추천을 제공한다.
  • 필요하면 DLP로 개인정보를 마스킹할 수 있다.
  • gRPC 방식은 Conversation/Participant를 API로 만들고 StreamingAnalyzeContent로 오디오를 전송한다.

Agent Assist Integration 전체 비교표

8. SIPREC 아키텍처 흐름

단계설명
1고객이 회사 지원 번호로 전화
2고객사 SBC가 오디오 스트림 복사본 생성
3복사된 오디오를 Google Telephony Platform(GTP)로 전송
4GTP가 Conversational AI / CES로 전달
5STT로 실시간 전사 수행
6Agent Assist가 추천 생성
7transcript / suggestion을 agent desktop으로 전달

핵심 포인트

  • Agent Assist는 passive listener처럼 동작
  • 오디오 원본은 통화용, 복사본만 분석에 사용
  • 실시간 추천은 전사된 텍스트 기반으로 생성

14. SIPREC가 preferred인 이유

이유설명
구현 effort 감소conversation lifecycle을 직접 다 만들 필요가 적음
자동화 수준 높음conversation 생성, participant 할당, 종료 처리 자동
telephony 친화적기존 SBC 환경과 잘 맞음
운영 단순화실시간 음성 통합이 더 자연스러움

결론

SIPREC를 사용할 수 있다면 일반적으로 SIPREC가 선호됨
