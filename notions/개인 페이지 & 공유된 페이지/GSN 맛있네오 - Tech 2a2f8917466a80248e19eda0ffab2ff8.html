AWS - GCP 연동 관련

아래는 AWS EC2 → GCP Dialogflow CX 를 안전하게 연동하기 위해 구성한

Workload Identity Federation(WIF) 전체 기술 내용

Workload Identity Federation(WIF) 기술 구성 문서

(AWS EC2 → GCP Dialogflow CX 인증 연동)

1. 개요 (Overview)

기존 방식은 GCP Service Account 키(JSON 파일) 을 EC2 서버에 배포하여

Dialogflow CX API 호출 시 인증에 사용했다.

그러나 이 방식은 다음과 같은 문제를 가진다:

  • 키 파일이 유출되면 영구 권한 탈취 가능
  • 키 만료 정책 적용 시 수동 재배포 필요
  • GitHub 등에 노출될 위험
  • 키 재발급 후 서버 재배포 필요

이를 해결하기 위해 GCP가 제공하는 Workload Identity Federation(WIF) 을 사용해

AWS EC2 인스턴스가 키 없이(Google Keyless Authentication)

GCP 리소스(Dialogflow CX)에 접근하도록 구성했다.

결과:

  • 서비스 계정 키 파일 필요 없음
  • 자동 토큰 교환(EC2 메타데이터 → AWS STS → GCP Identity Pool → SA Impersonate)
  • 보안성 10배 강화
  • 키 만료 문제 제거

2. 전체 아키텍처 흐름 (Token Exchange Flow)

아키텍처는 다음 순서로 작동한다:

EC2 Instance
   ⬇ (IMDSv2)
AWS Temporary Credentials 발급
   ⬇ (STS AssumeRole)
AWS Role: ec2-gcp-wif-role
   ⬇
Google Workload Identity Pool (aws-pool)
   ⬇ (Subject Token)
Google Workload Identity Provider (aws-provider)
   ⬇ (Federation)
GCP Service Account: [YOUR_SA_NAME]
   ⬇ (Impersonation)
Access Token 발급 (1시간 유효)
   ⬇
Dialogflow CX API 호출 성공

핵심:

AWS IAM 역할의 임시 자격 증명을 기반으로

GCP 서비스 계정 토큰을 자동으로 발급받는 구조이다.

3. AWS 측 구성

3.1 EC2 IAM Role 생성

ec2-gcp-wif-role

정책:

  • AWS STS 호출 및 메타데이터 서비스 접근 허용 (기본)
  • GCP가 요구하는 STS 정보를 제공할 수 있도록 IAM 접근 설정

Trust policy (요약):

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}

3.2 IAM Role을 EC2 인스턴스에 연결

AWS Console → EC2 → Instance Settings → IAM Role

→ ec2-gcp-wif-role 선택하여 연결.

3.3 IMDSv2 활성화 (필수)

GCP WIF는 IMDSv2 를 요구한다.

EC2 → Instance Metadata →

IMDSv2 = Required

4. GCP 측 구성

4.1 Workload Identity Pool 생성

GCP Console → IAM & Admin → Workload Identity Federation

Pool 이름: aws-pool

Provider 수: 1

4.2 AWS Provider 생성

Provider ID: aws-provider

Provider 유형: AWS

AWS 계정 및 역할 정보 입력:

AWS_ACCOUNT_ID = 970123029402
AWS_ROLE = arn:aws:sts::970123029402:assumed-role/ec2-gcp-wif-role

이 설정으로 GCP는

“이 AWS 역할을 가진 엔티티만 접근 가능”

이라고 판단한다.

4.3 WIF 사용자에게 Service Account 권한 부여

가장 핵심 단계.

Service account:

service-account@[YOUR_GCP_PROJECT].iam.gserviceaccount.com

Principal(주체):

principalSet://iam.googleapis.com/projects/369083780695/locations/global/workloadIdentityPools/aws-pool/attribute.aws_role/arn:aws:sts::970123029402:assumed-role/ec2-gcp-wif-role

부여한 Role:

  • Workload Identity User
  • Service Account Token Creator

Token Creator가 없으면

SA에 대한 access token을 만들 수 없기 때문에

403 PERMISSION_DENIED 가 발생한다.

5. 애플리케이션 변경 사항 (Spring Boot)

기존:

GOOGLE_APPLICATION_CREDENTIALS=/path/to/json

→ 제거

WIF는 파일 인증이 아님.

자동으로 Google Application Default Credential(ADC) 경로에서

AWS 기반 WIF를 감지하여 동작함.

적용된 코드:

GoogleCredentials cred = GoogleCredentials.getApplicationDefault();

결과적으로 Spring Boot는

더 이상 JSON 파일을 읽지 않고

AWS 임시 자격 증명을 기반으로 GCP 토큰을 자동 생성한다.

6. Docker / EC2 환경 설정

6.1 docker-compose에서 JSON 키 제거 후, WI록

environment:
      GOOGLE_APPLICATION_CREDENTIALS: /usr/local/gsncafeteria/keystore/gsn-wif-aws-cred.json
      SPRING_PROFILES_ACTIVE: default
volumes:
      - /usr/local/gsncafeteria/keystore:/usr/local/gsncafeteria/keystore:ro
      - /usr/local/gsncafeteria/imageFile:/app/imageFile:ro   # 이미지 파일 폴더를 읽기 전용으로 마운트

6.2 EC2 내부에서 확인

curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600"

그리고 실제로 AWS 임시 자격 증명이 나오는지 확인:

curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/ec2-gcp-wif-role

정상 출력됨 → 역할 연결 OK

7. 실제 동작 과정 (실시간 인증 흐름)

7.1 Spring Boot가 Dialogflow CX API 요청 →

7.2 Google Auth Library가 AWS 임시 토큰 요청 →

AWS STS: AccessKeyId / SecretAccessKey / Token 발급

7.3 이를 GCP WIF Provider에 제출 →

7.4 GCP가 Provider 설정을 검증 →

  • AWS 계정 ID 일치?
  • 역할 ARN 일치?
  • Signature OK?

7.5 Service Account Impersonation 수행 →

  • Service Account Token Creator 권한 필요

7.6 1시간짜리 GCP Access Token 발급

7.7 Dialogflow CX API 호출 → 성공

8. 장애 원인 및 해결 요약

초기 오류들

  1. Failed computing credential metadata

    → IMDSv2 비활성화 문제

  1. 401 Unauthorized (AWS metadata)

    → IMDSv2 Required 설정 없음

  1. 403 iam.serviceAccounts.getAccessToken denied

    → Service Account Token Creator 권한 누락 (가장 핵심)

  1. JSON 키가 남아 있었음

    → ADC가 WIF 대신 JSON 키를 로딩

모두 해결함.
