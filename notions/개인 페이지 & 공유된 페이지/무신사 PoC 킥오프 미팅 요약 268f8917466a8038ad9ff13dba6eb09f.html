무신사 PoC 킥오프 미팅 요약

1. POC 및 프로덕션 도입 전략 정리함.

  • 챗봇 (Text-based Chatbot):
    • 우선순위: 1순위임. POC 이후 바로 프로덕션 적용을 목표로 함.
    • 전략: 챗봇 기능은 가장 먼저 안정화함. 사용자 피드백을 빠르게 반영하여 개선해 나갈 예정임.
  • 음성봇 (Voice-based Bot):
    • 우선순위: 2순위임. 챗봇의 안정화 이후 프로덕션 도입 가능성 봄. 다만 테스트는 이루어져야함.
    • 필요 사항: 음성 인식(STT), 음성 합성(TTS) 품질, 지연 시간, 외부 IVR/전화 시스템과의 연동 등 음성 특화 테스트 및 검증이 별도로 필요함. POC에서 음성봇의 가능성만 확인하고, 실제 도입은 챗봇 안정화 이후 결정함.
  • Context 도달 과정 (Contextual Understanding):
    • 목표: 고객의 의도(Intent)를 선제적으로 파악하고, 전체 대화 맥락(Context)을 이해하여 응대하는 것이 궁극적인 목표임.
    • 어떻게 도달하는가?: Conversational Agent의 플레이북, 인텐트, 엔티티, 세션 파라미터, 그리고 외부 시스템(CRM 등) 연동을 통해 구현함. 특히 LLM 기반의 플레이북은 사용자의 다양한 발화를 이해하고 복잡한 시나리오에서도 맥락을 유지하는 데 강점을 가짐.
    • 확률 (정확도): POC 기간 동안 6개 카테고리에 대한 인텐트 감지 및 맥락 유지의 정확도를 측정하여 현실적인 성공률을 예측해야 함. 목표 정확도를 설정하고 지속적으로 개선함.
    • 비용: LLM 호출 횟수, 툴 사용 횟수, STT/TTS 사용량, 백엔드 서비스(Webhook) 사용량 등을 기준으로 비용이 발생함. POC에서 실제 사용량 기반의 비용 예측 모델을 수립해야 함. - 추후 논의

2. 메트릭 및 성공 지표 정의 (Conversational Agent에 국한)함.

기존 상담 시스템과는 별개로 Conversational Agent (챗봇) 자체의 독립적인 성능 평가를 위한 메트릭 정의가 필요함. - GS네오텍이 어떤 성공 지표로 수행할 예정인지 제안하길 원함.

  • Scope: Conversational Agent (챗봇)에 한정하여 POC를 진행하고 평가함.
  • Metric 정의:
    1. 6개 카테고리에 대한 문의 해결 비율 정의함.
      • 정의: 챗봇이 특정 6개 핵심 카테고리(예: 주문/결제, 배송/반품, 상품 문의, 멤버십, 이벤트, 기술 지원 등 무신사 특화 카테고리)에 대한 고객 문의를 상담원 개입 없이 성공적으로 완료한 비율임.
      • 측정 방법: 각 카테고리별 인텐트 감지 성공률, 관련 툴 호출 성공률, 최종 응답 만족도(추후 고객 피드백) 등을 복합적으로 고려하여 측정함.
    1. 고객의 상담원 컨택 없이 해결된 비율 (Self-Service Rate) 정의함.
      • 정의: 전체 챗봇 문의 중, 상담원 에스컬레이션 없이 챗봇 단독으로 고객의 문제를 해결한 비율임. 이는 "해결률" 또는 "자동화율"로 표현됨.
      • 측정 방법: 챗봇 대화 세션 종료 시점에 상담원_연결 인텐트가 호출되지 않은 세션의 비율로 측정함. (또는 챗봇이 문제 해결 완료 인텐트를 스스로 호출한 경우)
    1. 상담사 에스컬레이션 (Escalation Rate) 정의함.
      • 정의: 챗봇이 고객의 문제를 해결하지 못해 상담원에게 연결(에스컬레이션)된 비율임.
      • 측정 방법: 상담원_연결 인텐트 호출 횟수 / 전체 챗봇 대화 세션 수로 측정함. (낮을수록 좋음)
    1. CSAT (Customer Satisfaction Score) - 고객 만족도 정의함.
      • 정의: 챗봇과의 대화 경험에 대한 고객의 만족도임.
      • 측정 방법: 챗봇 대화 종료 시점에 만족도 설문(예: "챗봇과의 대화에 만족하셨나요? 예/아니오")을 제공하고 응답률을 측정함. (매우 중요함)
    1. (추가 제안) 정확한 파라미터 추출률 (Parameter Extraction Accuracy) 정의함.
      • 정의: LLM이 사용자 입력에서 필요한 핵심 정보(예: 상품명, 날짜, 수량 등)를 정확하게 추출하는 비율임.
      • 측정 방법: 특정 인텐트 처리 시 추출된 파라미터와 실제 의도된 파라미터의 일치율을 수동 또는 자동으로 검증함. (툴 사용 성공률에 직접적인 영향을 미침)

3. 비용 문제 정리함. - 추후 논의

  • 측정 비용:
    • 추후 논의 필요
  • 어디서 챗봇에 고객이 인입할 것인가 정의함.
    • 초기 인입 채널 정의: 무신사 웹사이트/앱의 고객센터 페이지, 특정 이벤트 페이지 등을 고려.

4. 필요한 소스 요청 및 구성도 정의. - 추후 논의

  • 시행사 측 요청 사항:
    • API 형태: 고객 정보, 주문/결제, 배송/반품, 상품 정보 등 챗봇이 조회/등록/업데이트해야 하는 핵심 CRM 및 백엔드 시스템의 API 문서 및 연동 정보를 요청함. (예: OpenAPI Spec, API Key, 인증 방식)
    • 대략적인 구축 구성도 (사용자 인증 관련 포함): 무신사의 기존 고객 인증 시스템과 챗봇/백엔드 서비스가 어떻게 연동될지(예: SSO 연동, 고객 ID 전달 방식)에 대한 대략적인 아키텍처 구상을 요청함.
    • Full Function에 대한 정의: 챗봇이 최종적으로 수행해야 할 모든 기능 목록 및 각 기능별 상세 요구사항을 요청함. (초기 6개 카테고리 외 추가 기능 포함)
  • 우리 회사/Google 측 역할:
    • Full Function에 대한 정의: Google Conversational Agent의 제공 가능한 모든 기능을 기반으로, 무신사의 요구사항을 충족시키기 위한 구체적인 기능 구현 방안 및 로드맵을 제안함. (POC 범위 외 향후 확장성 포함)

5. 궁극적 목표 정의함.

  • 고객 의도 선제적 대응: 고객이 문의하기 전에(또는 문의 초기에) 그 의도를 예측하고 필요한 정보를 먼저 제공하거나, 적절한 액션을 유도하여 고객 경험을 최적화함.
    • 예시: 로그인 시 최근 주문 정보를 미리 띄워주거나, 특정 페이지에서 고객이 오래 머물 시 "무엇을 찾으시나요?"라고 먼저 말을 거는 등임. 또한 고객에게 “교환 문의를 하시나요?” 등등
  • 챗봇의 전체 Context 이해: 챗봇이 단발적인 질문-답변이 아닌, 대화 전체의 흐름과 고객의 과거 상호작용 기록(CRM 연동), 현재 상황(웹페이지 등)을 통합적으로 이해하여 맞춤형 대화를 제공함.

다음 단계:

  1. POC 범위 재확정: 6개 카테고리를 포함한 초기 POC 범위와 목표 수치(예: 셀프 서비스율 70%, CSAT 80% 이상)를 구체화함.
  1. 무신사 API 정보 취합: 챗봇이 연동할 CRM/백엔드 API 정보를 상세하게 요청함.
  1. 아키텍처 설계: 무신사 환경에 맞는 연동 아키텍처를 구체화하고, 인증 및 데이터 흐름을 정의함.
  1. 비용 모델 수립: POC 기간 동안 예상되는 비용을 추산하고, 프로덕션 도입 시의 대략적인 비용 모델을 제시함.
