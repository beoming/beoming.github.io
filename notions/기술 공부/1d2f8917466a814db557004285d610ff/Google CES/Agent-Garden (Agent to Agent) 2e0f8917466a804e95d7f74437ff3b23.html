Agent-Garden (Agent to Agent)

현재 구현/구상 중인 Agent Garden 기능 정리

1. 목표

  • Agent Garden(Planning Agent + Tool 구조)를 기반으로
    • 항공편 검색
    • 호텔 검색/객실 선택
    • 일정(Itinerary) 생성
  • 선택된 결과는 memorize로 저장하고, 필요 시 booking_agent로 전달하는 구조

2. 현재 Agent Garden 구조

2.1 Planning Agent (Orchestrator)

  • 역할: 사용자의 “여정”을 파악하고 적절한 Tool을 호출하여 결과를 종합
  • 지원하는 사용자 여정
    • 항공만 찾기
    • 호텔만 찾기
    • 항공+호텔 선택 (일정 제외)
    • 항공+호텔+전체 일정 생성
    • (옵션) 자율적으로 추천/선택까지 진행

2.2 Tool 목록 (Agent가 호출 가능한 도구)

  • flight_search_agent: 항공권 옵션 검색
  • flight_seat_selection_agent: 선택 항공편 좌석 옵션
  • hotel_search_agent: 호텔 옵션 검색
  • hotel_room_selection_agent: 객실 옵션
  • itinerary_agent: 확정된 선택을 기반으로 일정 JSON 생성
  • memorize: 사용자가 선택한 값/메타데이터 저장
중요: Planning Agent는 “검색” 자체를 하지 않고, Tool 호출을 통해 결과를 받아 조합함.

3. “실제 백엔드 연결” 구현 방향

3.1 현재 상태 (샘플/데모)

  • Tool이 실제 API를 호출하지 않고
  • LLM이 프롬프트 기반으로 “그럴듯한 JSON”을 생성하는 형태

3.2 목표 상태 (실서비스)

  • Tool을 HTTP API Wrapper로 교체
  • 외부 백엔드(항공/호텔/좌석/객실)를 호출해 실제 결과를 JSON으로 반환
  • Planning Agent 프롬프트/흐름은 가급적 유지하고, Tool 구현부만 교체

4. 외부 백엔드 붙이는 표준 아키텍처

4.1 권장 아키텍처 (Tool = HTTP Adapter)

[User/Web UI]
   ↓
[Agent Garden Planning Agent]
   ↓ (tool call)
[Tool Adapter Layer]
   ↓ (HTTP)
[Flight Search Backend] / [Hotel Search Backend] / ...
   ↓
[JSON Response] → Agent → User

4.2 백엔드 API 예시 (스키마 고정이 중요)

Flight Search API (예)

  • POST /flights/search
  • Request:
    • origin, destination, start_date, end_date, preferences
  • Response:
    • 현재 프롬프트의 FLIGHT_SEARCH_INSTR에서 요구하는 형태와 동일한 JSON 스키마로 반환

Hotel Search API (예)

  • POST /hotels/search
  • Request:
    • destination, check_in_date, check_out_date, preferences
  • Response:
    • 현재 프롬프트의 HOTEL_SEARCH_INSTR에서 요구하는 형태와 동일하게 반환
핵심 원칙:
- Planning Agent 변경 최소화
- Tool 출력 JSON 형식을 프롬프트 스키마와 1:1로 맞춤
- 실제 공급자(Amadeus/Skyscanner/Booking 등)는 백엔드 뒤에서 교체 가능

5. 세션/상태 관리

  • memorize tool을 통해 다음을 저장 (예)
    • origin, destination, start_date, end_date
    • outbound/return flight + seat
    • hotel + room
    • itinerary JSON
  • 운영 단계에서는 memorize의 저장소를
    • Firestore/Redis/DB 등으로 매핑 가능

6. 보안/운영 체크리스트 (필수)

  • Tool이 호출하는 백엔드 API는 외부 공개 금지
    • API Key / IAM / mTLS 등 적용
  • 타임아웃(예: 3~5초), 재시도 정책, 오류 메시지 표준화
  • Rate limit / Abuse 방지
  • (선택) 로그/추적: 요청 ID, tool 호출 이력, 응답 크기, 오류율 등

7. Sendbird 포함 vs 미포함 비교

7.1 “Sendbird 미포함” 구성 (Private App으로 내부 채팅 UI/로직 개발)

구조

[웹/앱 내부 채팅 UI]
  - 직접 구현 (WebSocket/SSE/HTTP)
   ↓
[자체 Chat Backend]
   ↓
[Agent Garden]
   ↓
[Zendesk Connector]
   ↓
[Zendesk]

장점

  • 벤더 락인 낮음 (GCP + Zendesk 중심)
  • 비용 구조 단순 (MAU 기반 외부 요금 부담 ↓)
  • UI/UX 및 데이터 저장 방식 완전 자유
  • “AI + 업무처리”가 중심이면 과하지 않음

단점 (직접 책임짐)

  • 실시간 채팅 인프라를 직접 운영
    • WS 재연결/순서 보장/오프라인 처리/멀티 디바이스 동기화
    • 첨부파일/읽음 처리/스케일링/관측성 등
  • 구현 난이도 및 운영 부담 증가

적합한 경우

  • 챗봇이 “실시간 메신저 수준” UX가 필수가 아닐 때
  • 상담사 Copilot, FAQ/문의 생성 중심
  • 사용자 트래픽이 상대적으로 제한적이거나 단계적으로 확장할 때

7.2 “Sendbird 포함” 구성 (Sendbird SDK + 플랫폼)

구조

[웹/앱]
  - Sendbird SDK
   ↓
[Sendbird Platform]
   ↓ (Webhook/Event)
[Agent Garden]
   ↓
[Zendesk Connector]
   ↓
[Zendesk]

장점

  • 채팅 인프라를 거의 즉시 확보 (시간 단축)
  • 대규모 실시간 메시징 운영에 강함
    • 재연결, 멀티 디바이스, 메시지 순서/전송 보장, 첨부파일 등
  • 고객 채팅 UX가 제품 경쟁력인 경우 유리

단점

  • 비용이 MAU/사용량 기반으로 증가할 수 있음
  • 메시지 데이터/이벤트 흐름이 Sendbird에 종속 (락인)
  • 커스터마이징에는 한계가 있을 수 있음 (플랫폼 제약)

적합한 경우

  • “채팅 경험 자체”가 핵심인 앱/서비스
  • 모바일 중심, 높은 동시접속/실시간성 요구
  • 빠른 출시가 최우선

7.3 가장 큰 차이점

  • Sendbird 포함: “채팅 인프라(실시간 메시징)”의 책임을 외부 플랫폼이 가져감
  • Sendbird 미포함: “AI/업무 처리”에 맞춘 채팅을 직접 통제할 수 있지만, 메시징 운영 책임이 커짐

8. 선택해야 될 사항

질문 1)

“우리 서비스에서 채팅 자체가 핵심 가치인가?”
- YES → Sendbird 포함 고려
- NO → Sendbird 없이 내부 구현 + Agent Garden 권장

질문 2)

“대규모 실시간 메시징(멀티디바이스, 오프라인, 순서 보장)이 초기부터 필수인가?”
- YES → Sendbird 포함 유리
- NO → 미포함으로 시작 후 필요 시 단계적 도입 가능

GCP Agent Garden + Zendesk 연동 아키텍처/구성 정리 (초안)

1) 목표

  • 고객 챗봇(Customer Bot): 고객 문의 응답 + 필요 시 Zendesk 티켓 생성/조회
  • 상담사 코파일럿(Agent Copilot): 티켓 요약/답변 초안/내부 노트 작성 지원
  • 자동화(Automation): 분류/라우팅/상태 변경 등 운영 자동화
  • 공통 원칙: 권한 최소화(Least Privilege), 역할별 에이전트 분리, 감사/추적(Audit) 가능

2) 전체 아키텍처

3) Agent 역할 분리 (필수 설계)

3.1 Customer Agent (고객 챗봇)

  • 책임: 고객 질문 응답, 본인 티켓 상태 안내, 필요 시 티켓 생성
  • Zendesk 권한(예시)
    • Tickets: GET, CREATE (가능하면 “본인 티켓만” 정책으로 제한)
    • Comments: CREATE (public) (필요 시)
    • Users: GET (self) (필요 시)
  • 금지/주의
    • Tickets: UPDATE는 가급적 금지 (자동 상태변경 사고 방지)
    • Internal note 작성 금지(고객에게 노출될 위험)

3.2 Copilot Agent (상담사 코파일럿)

  • 책임: 티켓 요약, 답변 초안, 내부 노트, 고객 맥락 정리
  • Zendesk 권한(예시)
    • Tickets: GET, LIST
    • Comments: CREATE (internal note) (옵션)
    • Users/Orgs: GET
  • 특징
    • “실행(Write)”은 상담사 승인 후에만 수행하는 Human-in-the-loop 권장

3.3 (Optional) Automation Agent (운영 자동화)

  • 책임: 분류/라우팅/우선순위/SLA 기반 에스컬레이션 등
  • Zendesk 권한(예시)
    • Tickets: UPDATE (status, group_id, custom fields)
  • 특징
    • 이벤트 트리거 기반(Eventarc/Cloud Tasks/PubSub)
    • 변경 이력/로그 필수
핵심: 한 에이전트에 고객/상담사/자동화를 섞지 말 것

(보안/권한/감사/테스트 모두 어려워짐)

4) Zendesk Connector 연결(Connections) 설계

4.1 Connection 분리 전략 (권장)

  • zendesk-conn-customer : 고객 챗봇 전용 (권한 최소)
  • zendesk-conn-copilot : 상담사 코파일럿 전용 (조회 + 내부노트 옵션)
  • zendesk-conn-automation : 자동화 전용 (Update 포함)

장점
- 권한 사고 시 영향 범위 축소
- 감사(Audit)에서 “누가 무엇을 했는지” 추적 쉬움
- 운영/배포 분리 가능

5) ADK Toolset 구성 방식(개념)

  • ADK에서 Zendesk를 ApplicationIntegrationToolset으로 등록
  • entity_operations로 에이전트별 허용 액션을 고정

예시(개념)
- Customer Agent Toolset:
- Tickets: GET, CREATE
- Comments: CREATE(public)
- Copilot Agent Toolset:
- Tickets: GET, LIST
- Comments: CREATE(internal)
- Users/Orgs: GET
- Automation Agent Toolset:
- Tickets: UPDATE

운영에서는 “Read-only agent”부터 시작하고, Write 권한은 단계적으로 확장 권장

6) 주요 유즈케이스별 플로우

6.1 고객: 문의 → 답변/티켓 생성

User: "배송이 안 와요"
 -> Customer Agent
    - (RAG) 정책/FAQ 조회
    - 필요 시 Tickets.CREATE
    - 티켓 번호 안내 + 다음 액션 안내

6.2 고객: 내 티켓 상태 확인

User: "내 문의 진행상황 알려줘"
 -> Customer Agent
    - Tickets.GET (식별자/이메일/세션 매칭)
    - status/최근 코멘트 요약 제공

6.3 상담사: 티켓 요약/답변 초안

Agent opens ticket in Zendesk
 -> Copilot UI가 ticket_id/context 전달
 -> Copilot Agent
    - Tickets.GET + Comments LIST
    - 요약/다음 질문/답변 초안 생성
    - (선택) internal note draft 생성(승인 후 게시)

6.4 자동화: 분류/라우팅

New ticket created (Trigger/Webhook)
 -> Automation Agent
    - 티켓 내용 분류
    - Tickets.UPDATE (group_id, tags, priority)

7) 인증/보안/가드레일 (실무 체크리스트)

7.1 외부 사용자(고객) 인증

  • 고객 챗봇은 “사용자 식별”이 핵심
    • 로그인 기반: JWT/Session
    • 비로그인: 이메일 OTP/간편 인증 등
  • “본인 티켓만 조회” 정책 필요(가장 중요)

7.2 Tool 권한 최소화

  • Customer: GET/CREATE 중심
  • Copilot: GET/LIST 중심 (Write는 승인 후)
  • Automation: UPDATE는 별도 런타임/계정으로 격리

7.3 PII/민감정보

  • 로그에 ticket description/raw payload를 그대로 남기지 않기(마스킹)
  • RAG 인덱싱 시 개인정보 필터링/보관정책 필요

7.4 Rate limit / Abuse 방지

  • 고객 챗봇 endpoint: IP/계정 단위 rate limit
  • Tool 호출 timeout/retry 정책 고정

8) 운영/관측성

  • Cloud Logging: agent request_id / tool_call_id / ticket_id 매핑
  • Cloud Trace: tool latency 분해 (Zendesk call, RAG call 등)
  • 대시보드 KPI
    • Tool call 성공률/오류율
    • 평균 응답시간(p50/p95)
    • 티켓 생성 전환율
    • Copilot 사용률/수정률
