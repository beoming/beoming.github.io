1. Google ADK (Agent Development Kit) 개요

1.1 ADK란?

Agent Development Kit (ADK)는 Google이 제공하는 AI 에이전트 개발 프레임워크로, 복잡한 멀티 에이전트 시스템을 구축할 수 있게 해주는 Python 라이브러리입니다.

1.2 핵심 특징

1.2.1 멀티 에이전트 지원

  • 계층적 에이전트 구조: Root Agent와 Sub-Agent로 구성
  • 에이전트 간 전환 (Agent Transfer): transfer_to_agent() 메서드를 통해 동적으로 에이전트 전환
  • 병렬 처리: 여러 에이전트가 동시에 작업 수행 가능
from google.adk.agents import Agent

root_agent = Agent(
    model="gemini-2.0-flash-001",
    name="root_agent",
    sub_agents=[
        inspiration_agent,
        planning_agent,
        booking_agent,
        # ...
    ]
)

1.2.2 Tool 통합

  • FunctionTool: Python 함수를 직접 Tool로 사용
  • AgentTool: 다른 Agent를 Tool로 사용 (Agent를 Tool처럼 호출)
  • MCP Tool: Model Context Protocol을 통한 외부 서비스 연동
from google.adk.tools.agent_tool import AgentTool

planning_agent = Agent(
    tools=[
        flight_search_tool,  # FunctionTool
        AgentTool(agent=flight_seat_selection_agent),  # AgentTool
        hotel_search_tool,  # FunctionTool
    ]
)

1.2.3 세션 관리

  • Session State: 대화 컨텍스트와 상태 정보 저장
  • Memory Tool: memorize를 통한 정보 영구 저장
  • Session Callbacks: before_agent_callback, after_agent_callback 등
from travel_concierge.tools.memory import _load_precreated_itinerary

root_agent = Agent(
    before_agent_callback=_load_precreated_itinerary,  # 에이전트 실행 전 콜백
)

1.2.4 구조화된 출력 (Structured Output)

  • Pydantic Schema: 타입 안전한 출력 스키마 정의
  • JSON Response Config: JSON 형식으로 강제 출력
  • Output Key: 특정 키로 응답 저장
itinerary_agent = Agent(
    output_schema=types.Itinerary,  # Pydantic 모델
    output_key="itinerary",  # 세션 상태에 저장될 키
    generate_content_config=types.json_response_config,  # JSON 강제
)

1.3 ADK의 장점

  1. 선언적 프로그래밍: Agent, Tool, Sub-Agent를 선언적으로 정의
  1. 타입 안전성: Pydantic을 통한 스키마 검증
  1. 확장성: 새로운 Tool과 Agent를 쉽게 추가
  1. 프로덕션 준비: Vertex AI Agent Engine으로 직접 배포 가능

2. 멀티 에이전트 아키텍처

2.1 Travel Concierge 에이전트 구조

root_agent (루트 에이전트)
├── inspiration_agent (여행지 영감 제공)
├── planning_agent (여행 계획 수립)
│   ├── flight_search_tool (항공편 검색 - FunctionTool)
│   ├── hotel_search_tool (호텔 검색 - FunctionTool)
│   ├── flight_seat_selection_agent (좌석 선택 - AgentTool)
│   ├── hotel_room_selection_agent (객실 선택 - AgentTool)
│   └── itinerary_agent (일정 생성 - AgentTool)
├── booking_agent (예약 처리)
├── pre_trip_agent (출발 전 준비)
├── in_trip_agent (여행 중 지원)
└── post_trip_agent (여행 후 피드백)

2.2 에이전트 전환 메커니즘

2.2.1 Transfer to Agent

에이전트는 프롬프트 지시에 따라 다른 에이전트로 제어권을 넘길 수 있습니다:

# root_agent의 프롬프트 예시
ROOT_AGENT_INSTR = """
- If the user asks about general knowledge, vacation inspiration or things to do,
  transfer to the agent `inspiration_agent`
- If the user asks about finding flight deals, making seat selection, or lodging,
  transfer to the agent `planning_agent`
- If the user is ready to make the flight booking or process payments,
  transfer to the agent `booking_agent`
"""

2.2.2 Agent 제약 조건

  • disallow_transfer_to_parent=True: 부모 에이전트로 전환 금지
  • disallow_transfer_to_peers=True: 동급 에이전트로 전환 금지
itinerary_agent = Agent(
    disallow_transfer_to_parent=True,
    disallow_transfer_to_peers=True,
    # 오직 root_agent에서만 호출 가능
)

2.3 Planning Agent의 복합 구조

Planning Agent는 FunctionTool과 AgentTool을 혼합 사용합니다:

planning_agent = Agent(
    model="gemini-2.5-flash",
    tools=[
        # FunctionTool: 백엔드 API 직접 호출
        flight_search_tool,
        hotel_search_tool,

        # AgentTool: 다른 Agent를 Tool로 사용
        AgentTool(agent=flight_seat_selection_agent),
        AgentTool(agent=hotel_room_selection_agent),
        AgentTool(agent=itinerary_agent),

        # Memory Tool
        memorize,
    ]
)

설계 이유:
- FunctionTool: 빠른 API 호출이 필요한 경우 (항공편/호텔 검색)
- AgentTool: 복잡한 의사결정이 필요한 경우 (좌석 선택, 일정 생성)

3. MCP (Model Context Protocol)

3.1 MCP란?

Model Context Protocol (MCP)은 AI 에이전트가 외부 데이터 소스와 도구에 접근할 수 있도록 하는 표준 프로토콜입니다. ADK는 MCP를 지원하여 다양한 외부 서비스와 통합할 수 있습니다.

3.2 MCP의 특징

  1. 표준화된 인터페이스: 다양한 서비스와 일관된 방식으로 통신
  1. 비동기 지원: stdio 기반 비동기 통신
  1. 도구 자동 발견: MCP 서버가 제공하는 도구를 자동으로 감지

3.3 Travel Concierge에서의 MCP 사용

3.3.1 Airbnb MCP 예시

# tests/mcp_abnb.py에서 MCP 도구를 planning_agent에 추가
from google.adk.tools.mcp import MCPTool

# Airbnb MCP 서버 연결
mcp_tools = get_mcp_tools()  # Airbnb search, listing 도구

# planning_agent에 MCP 도구 추가
planning_agent.tools.extend(mcp_tools)

3.3.2 MCP 도구 사용 흐름

  1. MCP 서버 시작: Node.js 기반 MCP 서버 실행
  1. 도구 발견: ADK가 MCP 서버에서 제공하는 도구 자동 감지
  1. 도구 호출: Agent가 MCP 도구를 일반 Tool처럼 호출
[user]: "Find me an airbnb in San Diego, April 9th to April 13th"

[planning_agent]: airbnb_search({
    "checkin": "2025-04-09",
    "location": "San Diego",
    "checkout": "2025-04-13"
})

[planning_agent]: airbnb_search responds -> {
    "searchUrl": "https://www.airbnb.com/s/San%20Diego/...",
    "searchResults": [...]
}

3.4 MCP vs FunctionTool

특징MCPFunctionTool
통신 방식stdio 프로토콜Python 함수 호출
언어 독립성Node.js, Python 등Python만
설정 복잡도높음 (서버 필요)낮음 (함수만)
사용 사례외부 서비스 통합내부 로직, API 호출

4. Tool 시스템

4.1 Tool 유형

4.1.1 FunctionTool

Python 함수를 직접 Tool로 사용:

def flight_search_tool(
    origin: str,
    destination: str,
    departure_date: str,
    return_date: Optional[str] = None,
    tool_context: ToolContext = None,
) -> dict:
    """백엔드 항공편 검색 API를 호출하는 tool."""
    flight_api_url = get_flight_api_url()
    url = f"{flight_api_url}/search"
    response = requests.post(url, json=payload, timeout=10)
    return flights_selection.model_dump()

특징:
- Pydantic 모델로 파라미터 타입 검증
- ToolContext를 통해 세션 정보 접근
- 반환값은 Pydantic 모델로 변환

4.1.2 AgentTool

다른 Agent를 Tool로 사용:

AgentTool(agent=flight_seat_selection_agent)

특징:
- Agent의 출력이 Tool의 반환값이 됨
- Agent의 output_schema가 Tool의 반환 스키마
- Agent의 output_key로 세션 상태에 저장

4.1.3 Memory Tool

세션 상태에 정보 저장:

def memorize(key: str, value: str, tool_context: ToolContext):
    """세션 상태에 정보 저장"""
    tool_context.session_state[key] = value
    return {"status": f'Stored "{key}": "{value}"'}

4.2 Tool 호출 흐름

User Query
    ↓
Agent (LLM)
    ↓
Tool Selection (Function Calling)
    ↓
Tool Execution
    ├── FunctionTool → Python 함수 실행
    ├── AgentTool → Sub-Agent 실행
    └── MCP Tool → MCP 서버 통신
    ↓
Tool Response
    ↓
Agent (LLM) → 최종 응답 생성

4.3 백엔드 API 연동 Tool

4.3.1 환경 변수 기반 URL 관리

def get_flight_api_url():

    return os.getenv(
        "FLIGHT_SEARCH_API_URL",
        "https://flight-search-api-545259847156.us-central1.run.app"
    )

def flight_search_tool(...):
    # 함수 호출 시점에 환경 변수 읽기
    flight_api_url = get_flight_api_url()
    url = f"{flight_api_url}/search"
    # ...

중요: 모듈 로드 시점이 아닌 런타임에 환경 변수를 읽어야 Agent Engine에서 환경 변수가 제대로 적용됩니다.

4.3.2 에러 처리

try:
    response = requests.post(url, json=payload, timeout=10)
    response.raise_for_status()
    return flights_selection.model_dump()
except requests.exceptions.RequestException as e:
    return {
        "error": f"항공편 검색 API 호출 실패:{str(e)}",

5. Agent Engine 배포

5.1 Agent Engine이란?

Vertex AI Agent Engine은 ADK로 개발한 에이전트를 프로덕션 환경에 배포할 수 있는 관리형 서비스입니다.

5.2 배포 프로세스

5.2.1 AdkApp 생성

from vertexai.preview.reasoning_engines import AdkApp

app = AdkApp(
    agent=root_agent,
    enable_tracing=True,  # OpenTelemetry 추적 활성화
    env_vars={  # 환경 변수 전달
        "FLIGHT_SEARCH_API_URL": "https://...",
        "HOTEL_SEARCH_API_URL": "https://...",
        "GOOGLE_PLACES_API_KEY": "...",
    }
)

5.2.2 Agent Engine 생성

from vertexai import agent_engines

remote_agent = agent_engines.create(
    app,
    display_name="Travel-Concierge-ADK",
    requirements=[
        "google-adk (>=1.16.0)",
        "google-cloud-aiplatform[agent_engines] (>=1.93.1)",
        "google-genai (>=1.16.1)",
        "requests (>=2.32.3,<3.0.0)",
        # ...
    ],
    extra_packages=[
        "./travel_concierge",  # 로컬 패키지 포함
    ],
)

5.2.3 배포 단계

  1. 의존성 분석: requirements에 명시된 패키지 설치
  1. 패키지 패키징: extra_packages의 로컬 코드를 tar.gz로 압축
  1. GCS 업로드: Agent 코드와 의존성을 GCS 버킷에 업로드
  1. 컨테이너 빌드: Agent Engine 런타임 컨테이너 생성
  1. 서비스 배포: Vertex AI에 서비스로 배포

5.3 배포 스크립트 구조

def create(env_vars: dict[str, str]) -> None:
    """Agent Engine 생성"""
    # 1. AdkApp 생성 (환경 변수 포함)
    app = AdkApp(agent=root_agent, env_vars=env_vars)

    # 2. Agent Engine 생성
    remote_agent = agent_engines.create(
        app,
        requirements=[...],
        extra_packages=["./travel_concierge"],
    )

    # 3. Resource ID 반환
    print(f"Created:{remote_agent.resource_name}")

5.4 환경 변수 전달 메커니즘

# deploy.py에서 환경 변수 수집
env_vars = {}
env_vars["FLIGHT_SEARCH_API_URL"] = os.getenv("FLIGHT_SEARCH_API_URL")
env_vars["HOTEL_SEARCH_API_URL"] = os.getenv("HOTEL_SEARCH_API_URL")

# AdkApp에 전달
app = AdkApp(agent=root_agent, env_vars=env_vars)

중요: Agent Engine 내부에서는 os.getenv()로 환경 변수를 읽을 수 있지만, 모듈 로드 시점이 아닌 런타임에 읽어야 합니다.

8. 세션 및 메모리 관리

8.1 Session State

ADK는 세션별로 상태를 관리합니다:

# 세션 상태에 저장
tool_context.session_state["origin"] = "San Diego"
tool_context.session_state["destination"] = "Seattle"
tool_context.session_state["itinerary"] = {...}

8.2 Memory Tool

def memorize(key: str, value: str, tool_context: ToolContext):
    """세션 상태에 정보 저장"""
    tool_context.session_state[key] = value
    return {"status": f'Stored "{key}": "{value}"'}

8.3 Itinerary 저장

# itinerary_agent의 출력이 자동으로 세션 상태에 저장됨
itinerary_agent = Agent(
    output_key="itinerary",  # session_state["itinerary"]에 저장
    output_schema=types.Itinerary,
)

8.4 초기 상태 로드

def _load_precreated_itinerary(tool_context: ToolContext):
    """에이전트 실행 전 초기 상태 로드"""
    scenario_path = os.getenv("TRAVEL_CONCIERGE_SCENARIO")
    if scenario_path:
        with open(scenario_path) as f:
            initial_state = json.load(f)
            tool_context.session_state.update(initial_state)

root_agent = Agent(
    before_agent_callback=_load_precreated_itinerary,
)

9. 주요 기술적 결정 사항

9.1 FunctionTool vs AgentTool 선택 기준

상황선택이유
단순 API 호출FunctionTool빠르고 직접적
복잡한 의사결정AgentToolLLM 기반 추론 필요
구조화된 출력 필요AgentTooloutput_schema 활용
실시간 데이터FunctionTool낮은 지연시간

9.2 환경 변수 관리 전략

  1. 로컬 개발: .env 파일 사용
  1. 배포: deploy.py에서 환경 변수 수집 → AdkApp에 전달
  1. 런타임: 함수 내부에서 os.getenv() 호출
