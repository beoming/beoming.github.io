📘 Technical Breakdown: Core Configuration Concepts

✅ 1. SIP Profile 설정 (Codec: G.711 / G.729)

📘 SIP란?

  • SIP (Session Initiation Protocol)는 인터넷 기반 통신에서 음성/영상 세션을 설정, 유지, 종료하는 데 사용되는 애플리케이션 계층 신호 프로토콜입니다. SIP는 통화의 "전화 벨을 울리게 하고, 통화를 연결하며, 끊는" 역할을 합니다.
  • 실시간 미디어(RTP)는 SIP로 설정된 세션을 통해 별도로 전송됩니다.
  • 메시지 예: INVITE, ACK, BYE, 180 RINGING, 200 OK

📘 Codec이란?

  • Codec (Coder-Decoder)는 음성 데이터를 디지털 신호로 변환하고 압축하거나 해제하는 기술입니다.
CodecBitrate특징
G.71164kbps무압축 PCM, 고음질, 대역폭 많이 사용
G.7298kbps압축된 음성, 저대역폭 환경에 적합, 낮은 품질

실무 설정 예시:

Preferred Codec: G.711
Fallback Codec: G.729
DTMF Method: RFC 2833

✅ 2. DTMF 방식 확인 (RFC 2833 / SIP INFO)

📘 DTMF란?

  • *DTMF (Dual-Tone Multi-Frequency)**는 고객이 전화기에서 누르는 키패드 입력(예: '1', '2')을 시스템에 전달하는 방식입니다.

📘 주요 방식 비교:

방식전송 방식특징
RFC 2833RTP 패킷 내 DTMF 신호 포함✅ 대부분 SIP 시스템에서 표준, 실시간 처리에 적합
SIP INFOSIP 메시지 내 텍스트 형태 전송🔶 구식 방식, 지연 문제 발생 가능

실무 팁:

  • Genesys Cloud는 RFC 2833을 기본적으로 지원합니다.
  • Gateway나 PBX 장비가 SIP INFO만 지원할 경우, 일치하지 않으면 메뉴 입력이 인식되지 않을 수 있음

✅ 3. NAT Traversal 여부 확인

📘 NAT란?

  • *NAT (Network Address Translation)**는 내부 사설 IP 주소를 외부 공인 IP로 변환하는 네트워크 기능입니다. 문제는 SIP나 RTP처럼 연결 지향적이지 않은 프로토콜에서 NAT를 통과할 때 문제가 발생할 수 있다는 것입니다.

📘 Traversal 기술:

기술목적
STUNNAT 환경에서 자신의 공인 IP를 알 수 있게 함
TURNNAT를 통과하지 못할 경우 중계 서버로 우회
ICESTUN/TURN 조합, WebRTC 및 SIP 통신에 최적화

실무 팁:

  • Gateway ↔ Edge 사이에 NAT가 있으면 SIP ALG 기능 비활성화 권장 (장비에서 SIP 패킷이 손상될 수 있음)
  • Edge 및 WebRTC 연결 시 STUN 서버와의 통신 허용 필요

✅ 4. TLS 인증서 적용

📘 TLS란?

  • *TLS (Transport Layer Security)**는 SIP, HTTPS 등의 통신에서 데이터를 암호화하여 보안을 제공하는 프로토콜입니다.

📘 SIP TLS 구성:

  • SIP 신호 (INVITE 등)를 암호화하여 도청/변조 방지
  • Gateway ↔ Edge 또는 Edge ↔ Genesys Cloud 간 SIP over TLS 필요할 수 있음

실무 팁:

  • 인증서는 공인 CA에서 발급받는 것이 가장 호환성이 좋음
  • 인증서 만료 시 SIP REGISTER 실패나 TLS 연결 오류 발생
  • Edge와 Gateway 양쪽에 동일하게 적용되어야 정상 동작

✅ 5. Genesys Cloud에 Edge 등록 완료

📘 Genesys Edge란?

Genesys Edge는 SIP 및 RTP 트래픽을 Genesys Cloud와 연결해주는 온프레미스 장비 또는 소프트웨어입니다. Edge는 로컬망에 설치되어 PSTN 또는 SIP Gateway에서 온 음성 세션을 클라우드로 전달합니다.

📘 등록 절차:

  1. Genesys Admin > Telephony > Edges에서 새 Edge 생성
  1. Activation Key 생성 후 Edge 장비에 입력
  1. 설치 스크립트를 통해 자동 구성

실무 팁:

  • Edge 장비는 반드시 인터넷이 가능한 환경에 위치해야 함
  • DHCP보다 고정 IP(Static IP) 설정 권장
  • 등록 후에는 "Connected" 상태를 반드시 확인할 것

✅ 6. 공인 IP 또는 NAT 매핑 확인

📘 개념 설명:

공인 IP는 인터넷 상에서 고유하게 식별 가능한 주소이며, **NAT(Network Address Translation)**은 사설망 내부 장비를 외부와 통신 가능하게 만드는 기술입니다.

📘 실무 적용:

  • Edge 장비가 사설망에 있을 경우 NAT 라우터에서 포트 포워딩 필요
  • SIP 트래픽 및 RTP 포트 범위 (UDP 16384~32768) 허용해야 함

도구:

  • tcpdump, sngrep → SIP 메시지 흐름 확인
  • NAT 문제 발생 시 SIP INVITE 응답이 내부 IP로 설정돼 세션 실패 가능

✅ 7. TLS 연결 상태 테스트

📘 개념 설명:

Edge ↔ Genesys Cloud 간 통신은 TLS (TCP 443) 을 통해 암호화됩니다. 연결 실패 시 Edge는 클라우드와 세션을 맺지 못함.

📘 확인 방법:

  • 리눅스 명령어: openssl s_client -connect voice.genesyscloud.com:443
  • Edge 로그 확인: /var/log/edge/edge.log
    • "TLS connection established" 메시지 확인

실무 팁:

  • 방화벽에서 TCP 443 포트 열려 있어야 함
  • 인증서 에러가 있으면 로그에 ssl_error로 표시됨
  • 시스템 시간이 맞지 않으면 TLS handshake 실패 가능 (NTP 동기화 필수)

✅ 8. 로그 수집 위치 확인 (/var/log/edge*)

📘 개념 설명:

Edge의 모든 동작 상태는 로컬 로그 파일에 기록됩니다. 트러블슈팅 및 감사 추적에 필수 자료입니다.

주요 로그 파일:

파일명내용
edge.log주요 이벤트 및 시스템 상태 로그
sipstack.logSIP 메시지 송수신 내역
mediapath.logRTP 미디어 경로 상태
event.log각종 이벤트 트리거 기록

실무 팁:

  • 로그 분석 시 grep “ERROR” 또는 tail -f 명령어 활용
  • 로그 보존 정책 설정(30일, 순환 삭제 등) 확인 필요
  • 용량 초과로 인해 로그 손실되지 않도록 모니터링 설정 권장

✅ 9. 브라우저 버전 호환성 (Chrome / Edge 최신)

📘 개념 설명:

WebRTC 기반 상담은 브라우저에서 직접 음성 통화를 처리하므로 최신 브라우저 호환성이 매우 중요합니다.

실무 적용:

  • 지원 브라우저: Google Chrome, Microsoft Edge (Chromium 기반)
  • 지원되지 않는 브라우저: IE, Safari 일부 구버전 등

실무 팁:

  • 브라우저 자동 업데이트 유지 권장
  • 컨택센터 정책 상 브라우저 버전 고정 시 예외 처리 필요
  • 브라우저 보안 설정이 WebRTC 사용을 차단하지 않도록 확인

✅ 10. 미디어 권한 (마이크 접근 허용)

📘 개념 설명:

WebRTC 기반 상담을 위해 브라우저에서 마이크 접근 권한을 허용해야 합니다. 미허용 시 상담사는 고객과 음성 통신을 할 수 없습니다.

실무 적용:

  • 브라우저 주소창 좌측 자물쇠 아이콘 클릭 → ‘마이크 허용’ 선택
  • Windows: 설정 > 개인 정보 보호 > 마이크 사용 허용 확인

실무 팁:

  • 다중 마이크 환경에서는 기본 장치 설정 확인 필수
  • 상담 전 자체 테스트 절차(마이크 음성 반응 확인) 권장

✅ 11. NAT 환경에서 STUN / TURN 지원 여부 확인

📘 개념 설명:

WebRTC는 NAT 환경에서 미디어가 직접 연결되지 않는 경우 STUN, TURN 서버를 통해 음성 경로를 확보합니다.

실무 적용:

  • STUN은 공인 IP 확인용, TURN은 중계 서버로 음성 릴레이 역할
  • 방화벽에서 관련 포트 허용: STUN(UDP 3478), TURN(UDP 3478, TCP 443, 5349 등)

실무 팁:

  • https://test.webrtc.org 를 통해 사전 환경 테스트 가능
  • 사내망 내 Proxy나 DPI 장비가 WebRTC 흐름을 차단하는지 확인

✅ 12. 상담사 연결 품질 모니터링 가능 여부 (QoS Metrics)

📘 개념 설명:

WebRTC 상담 품질은 연결 지연, 손실률, 지터, MOS 등으로 측정됩니다. 이는 실시간/사후 분석에 중요합니다.

실무 적용:

  • Genesys Cloud > Performance > Interactions > Media Details 탭 확인

주요 지표:

지표설명
Jitter음성 패킷 간 시간 차이, 높을수록 품질 저하
Packet Loss패킷 손실률, 1% 이상이면 품질 저하 발생
MOSMean Opinion Score, 5점 만점 기준 4 이상이 양호

실무 팁:

  • 주기적 품질 리포트 수집 → 스케줄링된 네트워크 테스트 수행
  • 상담사 네트워크 품질 이슈 발생 시 즉시 QoS 기반 분석 시행

✅ 13. 장애 대응 전략 (Incident Response Strategy)

📘 개념 설명:

Genesys Cloud 기반 컨택센터에서는 다양한 장애 유형(네트워크, SIP, Edge, API, WebRTC 등)에 대한 사전 대응 체계가 필요합니다. 장애는 단순 통화 불가부터 API 실패, 상담사 연결 품질 저하까지 다양하게 나타날 수 있습니다.

🧱 장애 유형별 분류 및 대응

장애 유형주요 증상원인 예시대응 방안

통화 중 음질 저하목소리 끊김, 딜레이, 에코Jitter, Packet Loss, 네트워크 지연QoS 설정 확인, 상담사 네트워크 품질 진단, ISP 확인 요청

🔁 실시간 대응 플로우 (Recommended)

  1. Alert 감지
    • Genesys Cloud 알림 설정 또는 외부 모니터링 도구로 장애 발생 감지
  1. 로그 수집 및 분석
    • 관련 로그 (SIP, RTP, API, Agent 미디어 등) 수집
    • 예: sipstack.log, mediapath.log, cloud-api.log
  1. 원인 파악
    • 장애 범위: 단일 사용자 / 전체 사용자
    • 네트워크 레벨 or 애플리케이션 레벨 여부 확인
  1. 임시 복구 조치
    • 회선 전환, Edge 재등록, Routing 우회, 메시지 안내 등
  1. 근본 원인 분석 (RCA)
    • Post-Mortem 문서화 (장애 발생 → 원인 → 조치 → 재발 방지 계획)

🔒 재발 방지 전략

  • 정기 TLS 인증서 갱신 주기 설정
  • Edge/Cloud 연결 감시 → 자동 재시도 스크립트 구성
