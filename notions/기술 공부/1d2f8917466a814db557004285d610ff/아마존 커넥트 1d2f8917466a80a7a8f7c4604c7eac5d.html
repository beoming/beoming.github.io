AWS 리소스 접근 방식

1. Console

: 웹 브라우저에서 사용할 수 있는 GUI로 접근 및 관리할 수 있게 함.

사용이 쉽고, 실시간 상태를 시각적으로 확인 가능하지만 반복 작업이나 자동화에는 부적합 하며 대규모 인프라 관리에는 비효율적임.

2. CLI(Command Line Interface)

: 터미널이나 명령어 기반으로 AWS 리소스를 제어할 수 있게 함.

DevOps 자동화 및 배치 처리에 강력하지만 실시간 상태를 보기엔 다소 불편함

aws s3 cp ./myfile.txt s3://my-bucket/
aws ec2 describe-instances

3. AWS SDK

: 다양한 언어에서 코드로 AWS 서비스를 제어할 수 있는 라이브러리 모음

앱 내에서 AWS 연동 가능하며 커스터마이징된 로직 작성에 유리함. 다만 디버깅과 유지보수가 CLI나 콘솔보다 복잡할 수 있음.

import boto3

s3 = boto3.client('s3')
s3.upload_file('myfile.txt', 'my-bucket', 'myfile.txt')

AWS Lambda :

이벤트가 발생했을 때, 자동으로 코드가 실행되게 하는 서비스.

서버를 직접 만들거나 관리하지 않고 이벤트 실행을 트리거하여 리스폰스를 준다.

  • 서버 관리 불필요
  • 오토스케일링 : 많은 트래픽이 들어올 경우
  • pay as you go

Amazon Kinesis : 카프카와 비슷함.

  • 실시간 스트리밍 데이터를 수집, 처리 및 분석하여 적시에 인사이트를 확보하고 새로운 정보에 신속하게 대응할 수 있음.
  • 기계 학습, 분석 및 기타 애플리케이션을 위해 비디오, 오디오, 애플리케이션 로그, 웹사이트 클릭스트림 및 IoT 텔레메트리 데이터와 같은 실시간 데이터를 수집함.
  • 모든 데이터가 수집된 후에야 처리를 시작할 수 있는 것이 아니라 데이터가 수신되는 대로 처리 및 분석하고 즉시 대응할 수 있음.
  • 데이터 스트리밍을 S3, ES(엘라스틱 서치)와 같은 AWS 저장 서비스로 바로 전송

Amazon Lex :

Alexa와 동일한 기술을 기반으로 하는 Amazon Lex는 사용자가 자연어 음성이나 채팅을 사용하여 어떤 애플리케이션과도 상호 작용할 수 있는 AI 채팅 빌더이다.

애플리케이션에서 AI 챗봇과 음성 봇을 설계, 구축, 테스트, 배포하기 위해 고급 자연어 모델을 사용하는 완전 관리형 인공지능 서비스이다. 사용자의 필요에 따라 멀티턴 대화를 일시 중시하고 다시 시작할 수 있도록 지속적인 스트리밍 채팅 기능을 구축할 수 있다.

Amazon Polly

온디맨드 방식으로 음성을 생성하여, 어떤 텍스트라도 오디오 스트림으로 변환해주는 완전관리형 서비스임. 딥 러닝 기술을 사용하여 기사, 웹 페이지, pdf 문서 및 기타 TTS를 변환한다.

웹기반 앱이나 모바일 앱에서 API 혹은 SDK를 통해 자연어를 전송하여 Chatbot 기능 활용 가능

이 아키텍처는 AWS 기반의 실시간 음성 텍스트 변환(Realtime Speech-to-Text) 시스템입니다. 특히 Amazon Connect를 활용한 콜센터 시나리오에서 고객의 음성을 실시간으로 텍스트로 변환하고, 다양한 처리(번역, 저장 등)를 자동화한 구조입니다.

전체 흐름 설명

🔹 1. 고객 음성 수신 (Customer Audio)

  • 고객이 전화로 콜센터에 연결되면 Amazon Connect가 음성을 수신합니다.
  • Amazon Connect Contact Flow에서 Start Media Streaming을 통해 실시간 스트리밍을 시작합니다.

🔹 2. Lambda 트리거 (Step 1 & 2)

  • Contact Flow 안에서 AWS Lambda 함수가 호출됩니다.
  • 이 Lambda는 Amazon Kinesis Video Streams에 고객 음성 스트림을 전달합니다.

🔹 3. 실시간 음성 텍스트 변환 (Step 3 & 4)

  • Kinesis에 전달된 오디오 데이터는 Lambda 함수를 통해 Amazon Transcribe Realtime API에 전달됩니다.
  • Amazon Transcribe는 실시간으로 오디오를 받아 텍스트로 변환합니다.

🔹 4. 텍스트 처리 및 저장

  • 변환된 텍스트는 다시 Lambda 함수로 전달되어:
    • Amazon DynamoDB에 실시간 텍스트 저장 (예: 상담 기록)
    • Amazon S3에 오디오 파일 및 전체 텍스트 저장

🔹 5. AI 처리 및 번역 (선택)

  • 변환된 텍스트는 Amazon Translate, Amazon Comprehend를 통해:
    • 다른 언어로 번역하거나
    • 감정 분석, 키워드 추출 등의 AI 처리가 가능합니다.

🔹 6. 에이전트 UI 제공

  • 최종적으로 변환된 텍스트 결과는 WebSocket 연결을 통해 Agent UI에 실시간 표시됩니다.
  • 상담사는 실시간으로 고객 발화를 텍스트 형태로 확인할 수 있습니다.

Agent UI(상담원) ←—→ Amazon API Gateway

단순한 http 요청이 아니라, 실시간 통신을 위한 WebSocket 연결.

실시간으로 음성이 텍스트로 변환되어야 하기 때문에 WebSocket 통신이 필요함. 만약 HTTP만을 사용한다면 매번 Agent UI가 서버에 polling (요청) 해야됨. - 비효율적. 실시간성 떨어짐.

웹소켓은 한 번 연결되면 계속 열려있는 통로,

서버가 새 텍스트가 생기자마자 바로 push해서 Agent UI로 전달 가능.

Routing Configuration

스킬 → 숙련도

스킬 라우팅 → 상담사 개인역량

큐 → 최종적으로 콜 유입 지점 1, 2, 3

상담원 한명당 routing profile 하나는 가지고 있어야 한다.

이 다이어그램은 Amazon Connect에서 통화가 처리되는 전체 생명주기를 보여줍니다.

1. Arriving Calls (수신 전화)

DNIS (Dialed Number Identification Service) 번호들

  • +1 222 333 4444, +1 222 333 4445: 고객 서비스 일반 문의
  • +1 333 333 3333: VIP 고객 전용 라인
  • +1 444 333 4444: 기술 지원 전용 라인
  • +1 555 333 5555: 영업 문의 전용 라인

예시: 고객이 기술적 문제로 +1 444 333 4444로 전화를 걸면, 시스템은 이 번호를 식별하여 기술 지원팀으로 라우팅할 준비를 합니다.

2. Contact Flow (연락 플로우)

Flow1: 일반 고객 서비스 플로우

  • 인사말: "안녕하세요, ABC 고객센터입니다"
  • IVR 옵션: "청구서는 1번, 제품 문의는 2번..."

Flow2: 기술 지원 플로우

  • 인사말: "기술 지원팀입니다"
  • 문제 유형 선택: "하드웨어는 1번, 소프트웨어는 2번..."

Flow3: 영업 플로우

  • 인사말: "영업팀입니다. 어떤 제품에 관심이 있으신가요?"
  • 제품 카테고리 선택 메뉴 제공

3. Call Queues (통화 대기열)

: 스킬이다. 스킬별로 부서들이 있을 수 있다. 예를 들어 예금 상담 부서, 대출 상담 부서 등등

Queue1: 일반 고객 서비스 대기열

  • 예상 대기 시간: 3분
  • 대기 음악: 회사 홍보 메시지

Queue2: 기술 지원 대기열

  • 우선순위: 긴급 이슈는 앞으로 배치
  • 스킬 기반 라우팅: 하드웨어/소프트웨어 전문가 구분

Queue3: 영업 대기열

  • 지역별 라우팅: 고객 위치에 따른 담당자 배정
  • VIP 고객 우선 처리

4. Routing Profile (라우팅 프로필)

상담원 한명은 적어도 하나의 라우팅 프로필은 가져야 한다

정의 : 특정 상담원이 어떤 큐에 있는 고객들을 상담할지 결정하는 설정이다. 상담원 한명이 스킬별로 어떤 순서나 어떠한 큐를 응대를 할 지 결정하는 것.

라우팅 프로필 안에서는 우선순위가 크게 작용하고, Delay가 n이라면 n초 뒤에 해당 상담원과 연결할 수 있다는 것을 의미한다.

Profile1: 일반 상담원

  • Queue1만 처리
  • 근무 시간: 09:00-18:00
  • 동시 통화: 1건

Profile2: 다기능 상담원

  • Queue1과 Queue2 모두 처리 가능
  • 기술 지원 자격증 보유
  • 동시 통화: 1건

Profile3: 시니어 상담원

  • 모든 Queue(1,2,3) 처리 가능
  • 복잡한 이슈 에스컬레이션 담당
  • VIP 고객 우선 처리

실제 시나리오 예시

시나리오 1: 일반 고객 문의

  1. 고객 A가 청구서 문의로 +1 222 333 4444 전화
  1. Flow1 진입 → "청구서 문의는 1번 선택"
  1. Queue1 대기 → 2분 대기
  1. Profile1 상담원 연결 → 문제 해결

시나리오 2: 복잡한 기술 문제

  1. 고객 B가 서버 다운으로 +1 444 333 4444 전화
  1. Flow2 진입 → "긴급 기술 지원은 1번"
  1. Queue2 대기 → 우선순위 높음으로 설정
  1. Profile2 상담원이 처리 불가 → Profile3로 에스컬레이션
  1. Profile3 시니어 상담원이 원격 지원으로 해결

시나리오 3: VIP 고객 영업 문의

  1. VIP 고객 C가 +1 333 333 3333 전화
  1. Flow1 진입하지만 VIP 식별 → Flow3로 자동 전환
  1. Queue3 대기 → VIP 우선 처리로 즉시 연결
  1. Profile3 시니어 상담원이 맞춤형 상담 제공

고객 전화 → Contact Flow → [QUEUE 선택] → [Routing Profile 기반 상담사 탐색]

  • Incoming contacts are routed to Queues
  • Queues look for the Logest Available Agent (가장 오래기다린 상담원을 우선적으로 배치해줌)
  • If Outbound Calls are allowed, mush choose one Queue per Routing Profile for reporting purposes


요소설명
Queue고객이 기다리는 줄 (예: 영어 상담, 기술지원, VIP 등)
Skill Set (개념적)특정 Queue를 다룰 수 있는 상담 능력 (예: 영어 가능, 대출 전문 등)
Routing Profile어떤 Queue를 받을 수 있는 상담사인지 설정한 구성
상담사Routing Profile에 따라 연결 대상이 됨

Routing Profile 안에서 Priority와 Delay를 설정할 수 있다.

Priority 순위가 높을수록 우선 라우팅이 될 확률이 높고, delay는 정해진 시간이 꼭 지나야만 해당 큐에 입장할 수 있다는 뜻이다.

즉, ‘Savings’에 대한 문의를 했을 때, Savings를 갖는 큐의 우선순위를 먼저 보고, 그 후 delay를 보고 최종 큐를 선택하여 라우팅 해준다.

Amazon Connect → Amazon Kinesis Video Streams

  • 에이전트가 부재중일 경우, 고객의 음성 메시지가 Kinesis Video Streams로 전송됩니다
  • 실시간 오디오 스트리밍을 처리합니다

Amazon Kinesis Data Streams

  • CTR 데이터가 Kinesis Data Streams로 전송됩니다
  • 통화 메타데이터(발신자 정보, 시간, 내선번호 등)를 포함합니다

3. 음성메일 처리 단계

Lambda Function (process voicemail)

  • Kinesis Video Streams에서 오디오를 읽고 인코딩합니다
  • 처리된 음성메일 파일을 S3 버킷에 저장합니다
  • DynamoDB 테이블(voicemails)에 음성메일 메타데이터를 저장합니다

Amazon Transcribe

  • S3에 저장된 음성메일을 텍스트로 변환합니다
  • 음성-텍스트 변환이 완료되면 Lambda function을 트리거합니다

Lambda Function (process transcript)

  • Transcribe 결과를 처리하여 텍스트 형태의 메시지를 생성합니다

4. 알림 전송 단계

Lambda Function (deliver voicemail)

  • AWS Secrets Manager에서 AWS 액세스 키를 가져옵니다
  • DynamoDB에서 음성메일 정보를 조회합니다
  • 알림을 전송할 준비를 합니다

Amazon SNS & Amazon SES

  • SNS: 에이전트에게 푸시 알림이나 SMS를 발송합니다
  • SES: 음성메일 내용과 오디오 파일 링크가 포함된 이메일을 발송합니다

주요 특징

  1. 서버리스 아키텍처: Lambda 함수를 활용하여 확장성과 비용 효율성을 확보
  1. 실시간 처리: Kinesis를 통한 실시간 스트리밍 처리
  1. 다중 알림 채널: 이메일과 SMS/푸시 알림을 동시에 지원
  1. 음성-텍스트 변환: Transcribe를 통해 음성메일 내용을 텍스트로도 제공
  1. 보안: Secrets Manager를 통한 안전한 자격 증명 관리

이 아키텍처는 고객이 에이전트와 연결되지 못했을 때 음성메일을 남기고, 에이전트가 다양한 채널을 통해 해당 메시지를 받을 수 있도록 하는 완전한 솔루션입니다.

Contact Flow

Customer Queue Flow [e.g Queue Music]

: 고객이 큐에 들어 왔을때 실시되는 플로우 시나리오.

This contact flow is executed when a customer is placed in a queue.

Customer Hold Flow [e.g Hold Music]

: This contact flow starts when the customer is put on hold. It plays the audio that the customer hears while on hold.

Customer Whisper Flow [e.g Agent/Q Greeting]

: 고객과 상담원이 연결되기 직전에 안내멘트, 등과 같은 속삭임 멘트

This contact flow starts immediately before the call is connected. Default uses a “beep” sound to notify a customer thata their call has been connected to an agent.

Customer Outbound Whisper Flow

: 연결될때, 고객에게 알려주는 안내 멘트.

Customer will hear whats is in the Outbound whisper flows after an agent made a direct dial or accept callback. Configured in “Edit Queue”

Agent Hold Flow [e.g Hold Music For Agent]

: This contact flow starts when the customer is put on hold. It plays the audio that the Agent hears while customer is on hold.

Agent Whiper Flow [e.g Zip Tone For Agent]

: Executed before call is connected to agent. Like play a prompt for the agent for the name of the Queue.

Transfer to Agent Flow

: This default transfer flow is the customer’s experience when the customer is transferred to another agent. A Play prompt plays the message “Transferring now.” Then the Transfer to agent block is used to transfer the contact to the agent Specified in Quick Connect setup.

라우팅 전략

이 다이어그램은 Amazon Connect의 Queue Overflow 라우팅 전략을 보여줌. 대기 시간이 길어지거나 대기 중인 고객이 많을 때 자동으로 다른 상담원 그룹으로 통화를 전환하는 전략.

Change Routing Priority / age 를 조정할 수 있다. 우선순위를 조정하여 상담원 연결을 앞당기거나, 대기시간(age)을 늘려서 오래 기다린 고객으로 지정하여 상담원 연결을 앞당길 수 있다.

핵심 구성 요소

1. 오버플로우 조건 (Overflow Conditions)

  • Longest Time in Queue > 20 sec: 대기 시간이 20초를 초과할 때
  • Calls in Queue > 5: 대기 중인 통화가 5건을 초과할 때

이 두 조건 중 하나라도 충족되면 오버플로우가 발생

2. 큐 상태 체크 (Queue Status Check)

Contact Flow에서 실시간으로 큐 상태를 모니터링합니다:

  • Time in Queue > 20: 현재 가장 오래 대기 중인 고객의 대기 시간
  • Queue capacity > 5: 현재 대기 중인 통화 수

3. 라우팅 프로필 계층 구조

Primary Queue: Savings (저축 상담)

  • Pri 1 (우선순위 1): Savings 큐
  • Pri 1: Savings Overflow 큐 (대체 큐)
  • 0 sec: 즉시 라우팅

Overflow Target: Senior Agents

  • 오버플로우 발생 시 시니어 상담원 그룹으로 전환
  • 더 숙련된 상담원이 밀린 통화를 처리

    실제 작동 예시

    시나리오 1: 정상 상황

    1. 고객이 저축 상담을 위해 전화
    1. Savings 큐로 라우팅
    1. 대기 시간 15초, 대기 인원 3명
    1. 정상적으로 Savings Agents가 처리

    시나리오 2: 오버플로우 발생

    1. 고객이 저축 상담을 위해 전화
    1. 현재 상황:
      • 대기 시간: 25초 (20초 초과 ✓)
      • 대기 인원: 7명 (5명 초과 ✓)
    1. 오버플로우 조건 충족
    1. 자동으로 Senior Agents 그룹으로 라우팅
    1. 시니어 상담원이 저축 상담 통화도 처리

이 전략의 장점

  1. 고객 경험 개선
    • 긴 대기 시간 방지
    • 포기 통화(Abandoned Calls) 감소
  1. 리소스 효율성
    • 유휴 상담원 활용
    • 피크 시간대 유연한 대응
  1. 서비스 레벨 유지
    • SLA(Service Level Agreement) 목표 달성
    • 평균 대기 시간 감소
  1. 자동화된 관리
    • 실시간 모니터링과 자동 전환
    • 수동 개입 불필요

구성 시 고려사항

  • 스킬 매칭: Senior Agents가 Savings 상담도 처리할 수 있어야 함
  • 우선순위 설정: 오버플로우 통화의 우선순위 조정 필요
  • 임계값 조정: 비즈니스 요구에 따라 20초, 5건 등의 값 조정
  • 모니터링: 오버플로우 빈도 추적으로 인력 계획 수립

이 전략은 특히 예측하기 어려운 통화량 급증이나 계절적 피크 시즌에 매우 효과적입니다.

이 다이어그램은 Amazon Connect의 동적 라우팅 우선순위 조정 전략을 보여줌. AWS Lambda와 비즈니스 인텔리전스를 활용하여 실시간으로 큐의 우선순위를 조정하는 고급 라우팅 전략

핵심 구성 요소

1. Service Level Goal (서비스 레벨 목표)

  • 목표: 30초 이내 응답
  • 모든 고객 통화를 30초 이내에 상담원과 연결하는 것이 목표

2. AWS Lambda 통합

Lambda 함수가 두 가지 데이터 소스와 연동:

  • Customer Databases: 고객 정보, 등급, 이력
  • Business Intelligence: 실시간 큐 성능, 대기 시간 분석

3. 동적 우선순위 조정

초기 상태:

  • Savings Agents: Savings Queue만 처리 (Pri 1, 0초)
  • Senior Agents: Checking Queue (Pri 1, 15초), Savings Queue (Pri 1, 15초)

조정 후 ("15 seconds later..."):

  • Lambda가 실시간 분석을 통해 우선순위를 동적으로 재조정
  • 대기 시간이 긴 큐의 우선순위를 높이거나 라우팅 지연 시간을 조정

작동 메커니즘

1. 실시간 모니터링

Lambda 함수가 매 15초마다:
- 각 큐의 평균 대기 시간 체크
- Service Level 달성률 계산
- 상담원 가용성 확인

2. 지능형 의사결정

IF Savings Queue 평균 대기 > 25초 THEN
  - Senior Agents의 Savings Queue 지연 시간을 15초 → 5초로 단축
  - 더 많은 Senior Agents가 Savings 통화 처리

3. 비즈니스 규칙 적용

  • VIP 고객 식별 시 즉시 Senior Agents로 라우팅
  • 고객 이력 기반 적절한 스킬 그룹 매칭
  • 피크 시간대 자동 조정

    실제 시나리오 예시

    시나리오 1: 정상 운영

    1. 오전 10시, 평균 대기 시간 15초
    1. Lambda 분석: "Service Level 목표 달성 중"
    1. 기존 우선순위 유지
    1. Senior Agents는 주로 Checking 업무 집중

    시나리오 2: Savings Queue 폭주

    1. 오후 2시, Savings Queue 대기 시간 35초 초과
    1. Lambda 분석: "Service Level 위험"
    1. 동적 조정 실행:
      • Senior Agents의 Savings Queue 지연: 15초 → 0초
      • Checking Queue 지연: 15초 → 30초
    1. 더 많은 Senior Agents가 Savings 통화 즉시 처리

    시나리오 3: VIP 고객 우선 처리

    1. VIP 고객 인입
    1. Lambda가 Customer Database에서 VIP 상태 확인
    1. 즉시 Senior Agents 그룹으로 라우팅
    1. 일반 큐 대기 없이 우선 연결

이 전략의 장점

1. 지능형 자동화

CTR(Contact Trace Records)

CTR은 Amazon Connect에서 모든 고객 상호작용(통화, 채팅, 작업)에 대한 상세한 기록을 담은 데이터 레코드. 콜센터의 ‘블랙박스’라고 생각하면 됨.

CTR의 주요 구성 요소

1. 기본 정보

  • Contact ID: 각 상호작용의 고유 식별자
  • 시작/종료 시간: 통화 시작과 종료 시각
  • 채널: 음성, 채팅, 작업(task) 구분
  • 초기 연락 ID: 전환된 통화의 경우 원래 통화 추적

2. 고객 정보

  • 전화번호: 발신자 번호
  • 고객 엔드포인트: 고객이 사용한 디바이스/채널 정보
  • 위치 정보: 가능한 경우 지역 정보

3. 라우팅 정보

  • 큐(Queue): 어느 대기열을 거쳤는지
  • 상담원: 담당한 상담원 ID와 이름
  • 라우팅 프로필: 사용된 라우팅 규칙
  • 전환 이력: 다른 상담원/큐로 전환된 기록

4. 시간 측정 데이터

  • Queue Duration: 대기열에서 대기한 시간
    - Agent Interaction Duration: 상담원과 통화한 시간
    - Hold Time: 보류 시간
    - After Contact Work Time: 후처리 시간
    - Total Duration: 전체 통화 시간

5. 통화 품질 메트릭

  • 녹음 여부: 통화 녹음 링크
  • 전환 횟수: 상담원/부서 간 전환 횟수
  • 연결 해제 사유: 고객/상담원/시스템 중 누가 끊었는지
