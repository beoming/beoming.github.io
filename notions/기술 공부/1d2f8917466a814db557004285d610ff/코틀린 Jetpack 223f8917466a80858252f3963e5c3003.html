ViewBinding vs findViewById

뷰 결합에는 findViewById를 사용하는 것에 비해 다음과 같은 장점이 있다.

  • null 안전 : 뷰 결합은 뷰의 직접적인 참조를 생성하므로 유효하지 않은 뷰 ID로 인해 null 포인터 예외가 발생할 위험이 없다. 또한 레이아웃의 일부 구성에만 뷰가 있는 경우 결합 클래스에서 참조를 포함하는 필드가 @Nullable로 표시된다.
  • 유형 안전 : 각 바인딩 클래스에 있는 필드의 유형이 XML 파일에서 참조하는 뷰와 일치한다. 즉 클래스 변환 예외가 발생할 위험이 없다.
  • findViewById는 런타임에 뷰를 찾기 때문에 해당 ID의 뷰가 없거나 잘못된 타입으로 캐스팅하면 앱 충돌로 이어질 수 있다. ViewBinding과 Data Binding은 생성 시점에 안전한 뷰 참조를 제공한다.
  • 어댑터의 ViewHolder에서 findViewById 대신 ViewBinding이나 Data Binding을 사용하면, 각 아이템 레이아웃에 대한 바인딩 객체를 통해 뷰에 직접 접근할 수 있어 코드가 간결해지고 런타임 오류 위험이 줄어든다.

DataBinding vs ViewBinding

  • 데이터 바인딩은 이름처럼 데이터를 연결해주는 역할을 할 수 있다. 데이터와 함께 결합해서 사용할 수 있으며, DataBinding과 LiveData를 함께 활용할 수 있다.
  • Data Binding은 XML 레이아웃 내에서 데이터 표현식 (@{})을 사용하여 변수 값을 표시하거나, 조건에 따라 텍스트를 변경하거나, 클릭 이벤트를 함수에 연결하는 등 간단한 로직을 포함할 수 있다.
  • ViewBinding은 주로 findViewById를 대체하여 뷰 접근을 안전하게 만드는 데 초점을 맞춥니다. 반면, Data Binding은 뷰 접근 외에도 데이터를 UI 요소에 바로 연결하여 동적으로 업데이트하는 기능을 제공해요.

구글이 Android 뷰 접근에 Kotlin Extensions 사용을 권장하지 않는 주된 이유는 강의에서 언급했듯이, Kotlin Extensions 방식이 특히 복잡한 프로젝트나 여러 화면에서 성능 저하와 널 객체 참조 같은 문제를 유발할 수 있기 때문입니다. 따라서 ViewBinding이나 Data Binding 사용이 권장됩니다.

ViewModel

  1. 안드로이드 생명 주기를 관리하기 쉽다.
  1. onSaveInstanceState()를 사용해서 데이터를 관리할 수 있지만 적절치 않다.
  1. 상태(Life Cycle)가 변경될 때마다 데이터를 관리해줘야 하는데 이런 관리가 편하다.
  1. UI 컨트롤러(Activity, Fragment)에서 모든 것을 다 하려고 하면 복잡해진다.
  1. ViewModel을 사용하면 테스트나 관리가 용이하다.

뷰모델은 화면 회전 시 데이터 유실을 막고 UI 컨트롤러와 분리하여 생명주기에 안전하게 데이터를 관리한다.

뷰모델은 연관된 액티비티나 프래그먼트가 설정 변경으로 파괴/재생성될 때에도 스스로는 파괴되지 않고 데이터를 유지한다.

공유 뷰모델은 액티비티와 프래그먼트 사이에서 데이터를 주고받거나 상태를 일관되게 유지하는 데 유용하며 생명주기 문제도 해결한다. (by activityViewModels())

뷰모델 팩토리는 뷰모델을 생성할 때 생성자에 필요한 매개변수(예: 레포지토리)를 전달할 수 있게 해주는 표준적인 방법이다.

뷰모델은 UI 컨트롤러에서 데이터 관리 로직을 분리하여 코드를 깔끔하게 만들고 단위 테스트를 더 쉽게 할 수 있도록 도와준다.

Life Cycle

onCreate :

프로젝트를 새로 생성하면 MainActivity 클래스 안에 onCreate가 자동으로 생성된다. Activity가 시작될 때 호출되는 메서드로써 생명 주기 동안 딱 한번 실행되는 메서드이다.

액티비티 최초 실행 시에만 수행 해야된 작업들을 여기에 작성하면 된다. ex) 화면 레이아웃 정의, 뷰 생성

onStart :

Activity가 보이기 전에 호출되는 메서드.

화면에 진입할 때마다 실행되어야 하는 코드를 이곳에 작성해야한다.

onResume :

Activity의 일부가 보일 때 호출되며 다른 Activity가 Activity를 덮어버리거나 앱 사용중 전화, 문자 등으로 잠시 앱을 떠나는 등 액티비티가 일시 정지되었다가 돌아오는 경우 onResume 메서드가 호출된다.

Activity가 재개되었을 때 실행해야 하는 코드가 있다면 이곳에 작성하면 된다.

다만, 일부가 보일때부터 전체가 보일 때까지의 시간만큼만 수행되기 때문에 가벼운 작업만 수행해야 한다.

onPause :

방해되는 이벤트가 발생하고 Activity의 일부는 보이지만 다른 Activity가 foreground로 나올 때 이 메소드가 호출된다.

onPause는 아주 잠깐 실행되는 메소드이므로 메소드가 완료되기 전에 작업이 끝나지 않을 수 있기 때문에 데이터를 저장하거나, 네트워크를 호출하는 등 무거운 작업을 하면 안된다고 경고하고 있다.

그렇기 때문에 무거운 작업을 수행할때는 바로 다음 설명할 onStop에서 수행해야한다.

onStop :

Activity가 사용자에게 더 이상 보이지 않을때 이 메서드가 호출되며 앞서 말한것 처럼 무거운 작업은 이곳에서 처리해야 한다.

onRestart :

사용중인 앱에서 홈으로 나갔다가 다시 돌아오거나 다른 Activity로 갔다가 뒤로 가기 버튼을 통해서 돌아오는 경우 이 메소드가 호출된다.

onDestroy :

활동이 종료되는 경우나 구셩변경(ex:기기 회천)으로 인해 시스템이 일시적으로 활동을 소멸시키는 경우에 호출되는 메소드이다.

코드 분석

// LiveData Transformations

class MainActivity : AppCompatActivity() {

    private lateinit var binding : ActivityMainBinding
    private lateinit var viewModel: MainViewModel

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = DataBindingUtil.setContentView(this, R.layout.activity_main)
        viewModel = ViewModelProvider(this).get(MainViewModel::class.java)

        viewModel.liveCount.observe(this, Observer {

            binding.resultArea1.text = (it + it).toString()
            binding.resultArea2.text = (it * it).toString()
        })

        binding.btnArea.setOnClickListener {
            val count = binding.inputArea.text.toString().toInt()
            viewModel.setLiveDataValue(count)
        }


    }
}

MVVM 패턴

LiveData + DataBinding + ViewModel

MVVM 패턴은 Mode, View, ViewModel 세 가지 구성 요소를 가지는 디자인 패턴이다. 각 구성 요소는 특정한 역할을 담당하고 서로 독립적으로 동작할 수 있도록 설계되어있다.

뷰와 비즈니스 로직을 명확히 분리하기 위해 DataBinding과 LiveData를 사용한다. 이로 인해 뷰와 모델 간의 의존성은 없고, 뷰와 뷰모델 간의 의존성도 낮아서 모듈화, 재사용성, 유지보수성, 가독성이 향상된다. LiveData는 수명 주기를 인식하는 데이터 홀더 클래스로 데이터 변경 시 활성 상태의 관찰자에게만 알림을 보내 메모리 누수를 방지하고 데이터 일관성을 유지할 수 있다.

View 뷰 : 사용자 인터페이스를 담당. 사용자의 입력을 뷰 모델(ViewModel)에게 전달하고, 데이터 바인딩을 통해 뷰 모델(ViewModel) 데이터가 화면에 자동으로 반영된다.

Model 모델 : 데이터를 관리. 데이터 관련 비즈니스 로직을 처리한다.

ViewModel 뷰 모델 : 뷰와 모델을 연결하는 중간 관리자 역할. 사용자 입력을 처리하고 모델과 간접적인 상호 작용을 한다. 뷰와 데이터 바인딩을 통해 데이터가 변경될 때 뷰에게 알려준다.

Map & Switch Map

사용자가 값을 입력했을 경우, 해당 값은 ViewModel의 LiveData에 값이 저장됨. 이 값을 기반으로,

  • 단순 연산 결과 처리 → mapLiveData
  • 복잡한 처리 결과 → switchMapLiveData
class MainViewModel : ViewModel() {

    // Transformations -> map / switchMap

    // map -> 값을 return

    private var _mutableCount = MutableLiveData(0)
    val liveCount : LiveData<Int>
        get() = _mutableCount

    val mapLiveData = Transformations.map(liveCount) {
        it+it
    }

    val switchMapLiveData = Transformations.switchMap(liveCount) {
        changeValue(it)
    }

    fun setLiveDataValue(count : Int) {
        _mutableCount.value = count
    }

    fun changeValue(count : Int) : MutableLiveData<Int> {
        val testLiveData = MutableLiveData(count*count)
        return testLiveData
    }


}
private var _mutableCount = MutableLiveData(0)
val liveCount: LiveData<Int>
    get() = _mutableCount

이 코드에서 MutableLiveData는 변경 가능한 내부 변수임.

LiveData는 외부에 노출되는 읽기 전용 데이터

외부(Activity)에서는 직접 값을 바꿀 수 없고, setLiveDataValue()로만 바꿀 수 있음. → 캡슐화

val mapLiveData = Transformations.map(liveCount) {
    it + it
}

map()은 원본 LiveData의 값을 변형해서 새로운 LiveData를 만듦.

liveCount = 3이면 mapLiveData 는 6이됨. 이 값은 Activity에서 Observe 가능

val switchMapLiveData = Transformations.switchMap(liveCount) {
    changeValue(it)
}

fun changeValue(count: Int): MutableLiveData<Int> {
    val testLiveData = MutableLiveData(count * count)
    return testLiveData
}

switchMap()은 LiveData 값을 기반으로 다른 LiveData를 만들어서 연결해줌.

changeValue(3) → MutableLiveData(9) → switchMapLiveData가 9가됨.

↓
